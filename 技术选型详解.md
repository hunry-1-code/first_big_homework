# 技术选型详解——每个选择的区别与理由

> 逐项拆解每个技术决策的"为什么"，把备选方案的区别讲清楚。
> 一个原则：**对 5 人团队、7 天周期的课程项目来说，够用 > 完备，简单 > 炫技。**

---

## 一、后端框架：Flask vs FastAPI vs Django

先说结论：**选 Flask。** 下面是三者对比。

### 三种框架分别是什么？

| | Flask | FastAPI | Django |
|------|-------|---------|--------|
| **一句话描述** | Python 的"微型框架"，只给最基本的东西，其余你自由组合 | Python 的"现代异步框架"，自动生成 API 文档，性能高 | Python 的"全家桶框架"，什么都帮你做好了，但什么都得按它的规矩来 |
| **学习曲线** | 极低，半天能上手 | 中等，需要理解异步 | 陡峭，概念多 |
| **自带什么** | 路由、模板渲染、请求/响应处理 | 路由、自动API文档（Swagger）、数据校验（Pydantic）、异步支持 | ORM、后台管理、用户认证、表单、模板……全套 |
| **灵活性** | 你想怎么组织代码就怎么组织 | 推荐按模块组织，但自由度仍然高 | 必须按 Django 的方式组织（app 结构）|
| **性能** | 同步模式，并发能力一般 | 异步模式，并发能力强 | 同步模式，性能类似 Flask |
| **生态** | 最大，插件最丰富 | 快速增长，但插件不如 Flask 多 | 大而全，Django REST Framework 很成熟 |

### 为什么选 Flask 而不是 FastAPI？

```
FastAPI 的优势（异步高性能 + 自动 Swagger 文档）对课程项目意义不大：
- 自动 API 文档？团队内部协作用口头沟通+Apifox就够了，不需要 Swagger 自动生成
- 异步高性能？课程项目没有几百人同时并发访问，同步完全够用
- 异步编程的 async/await 是一套心智模型，如果团队不熟，反而增加 bug

Flask 的优势：
- 最简单，花半天就能写出可用的路由，剩下的时间留给业务逻辑
- BettaFish（41.6k 星）的主应用就是 Flask，证明它支撑得起舆情分析系统
- 团队 5 人里只要有 1 人会 Flask，就能帮其他人快速上手
```

### 为什么没选 Django？

Django 不适合的原因很直接：**"重"。**
- 你只需要 7 张表，Django 的 ORM 迁移系统显得杀鸡用牛刀
- Django 的模板系统、表单系统、admin 后台——能用的不多，但学起来要花时间
- 题目要求的功能（爬虫调度、情感分析、趋势预测）Django 不提供任何帮助，还是得自己写

**一句话**：Flask 是你自己搭积木，Django 是一套精装房——精装房改格局反而费劲。

## 二、前端方案：前后端分离（Vue） vs 不分离（Flask 模板）

先说结论：**优先 Flask 模板 + CDN 引入 ECharts。除非团队已经有会 Vue 的人，再升级为前后端分离。**

### 两种方案长什么样？

**方案 A：不分离（Flask 模板渲染）**
```
浏览器请求 /events → Flask 查数据库 → Jinja2 模板渲染成 HTML → 返回给浏览器
ECharts 通过 CDN <script> 标签直接引入，数据通过 API 或模板变量传入
```
- BettaFish 就这么做的，整个系统只有一个 `templates/index.html`
- 优点：一个服务跑所有，不需要跨域配置，不需要构建工具
- 缺点：交互复杂的页面（如智能问答的对话界面）JS 代码会写得很长

**方案 B：前后端分离（Vue 3 + Flask API）**
```
浏览器加载 Vue 单页面应用 → Vue 通过 Axios 调 Flask API → Flask 返回 JSON → Vue 渲染
```
- 优点：前端交互体验好，页面切换无刷新，适合看板/详情页/问答这种多页面系统
- 缺点：需要配置跨域、需要 Node.js 环境、需要学 Vue 的模板语法和组件化

### 为什么优先推荐方案 A？

不是因为方案 A 更好，而是**课程项目的瓶颈不在前端体验，在后端功能**。

```
时间分配计算：
- 方案 A：Flask 路由 + 模板 ≈ 1-2 天，剩下 5 天写爬虫/NLP/LLM
- 方案 B：Vue 搭建 + 组件开发 + 联调 ≈ 3-4 天，剩下 3 天写后端功能
```

如果团队有人已经会 Vue，方案 B 可以同时进行（前端人写 Vue，后端人写 API），那就选方案 B。

### ECharts 为什么是必选项？

题目明确要求 4 种图表：
- 趋势图（折线图，标注关键节点）→ ECharts 原生支持
- 情感分布图（饼图）→ ECharts 原生支持
- 平台分布（饼图/柱状图）→ ECharts 原生支持
- 高频关键词（词云）→ ECharts 有 wordCloud 扩展

ECharts 是百度开源的，中文文档好，示例全，CDN 引入即可。三大参考项目（BettaFish / MediaCrawler / stay-leave）的可视化部分一个用 Plotly（Python 端渲染）、一个用 Matplotlib、一个没做前端可视化——对前端的可视化，ECharts 是最直接的选择。

## 三、数据库：MySQL vs PostgreSQL vs SQLite vs MongoDB

先说结论：**MySQL + SQLAlchemy ORM。**

### 四种数据库的区别？

| | MySQL | PostgreSQL | SQLite | MongoDB |
|------|-------|------------|--------|---------|
| **类型** | 关系型 | 关系型 | 关系型（嵌入式）| 文档型（NoSQL）|
| **安装** | 需要装服务 | 需要装服务 | **零安装**，就是一个文件 | 需要装服务 |
| **性能** | 读快 | 写快、复杂查询强 | 单机够用，不能并发写 | 适合非结构化数据 |
| **JSON支持** | 有，但弱 | 很强（JSONB） | 弱 | 原生就是 JSON |
| **Python ORM** | SQLAlchemy 完美支持 | SQLAlchemy 完美支持 | SQLAlchemy 支持 | MongoEngine / Motor |
| **参考项目使用** | BettaFish、MindSpider、Astianjy | BettaFish（也支持） | — | — |

### 为什么选 MySQL 而不是 PostgreSQL？

```
全部参考项目的主选都是 MySQL，这是最简单的理由。

技术上的区别：
- MySQL 安装更简单，尤其是 Windows 环境
- 舆情数据的结构很固定（用户、事件、报道、分析结果），不需要 PostgreSQL 的 JSONB 高级特性
- 团队大概率在大学课程里学过 MySQL，切换成本为零

但如果你团队有人熟 PostgreSQL，选它也完全没问题——
BettaFish 的 config.py 同时支持 MySQL 和 PostgreSQL，SQLAlchemy 屏蔽了底层差异。
```

### 为什么不用 SQLite？

SQLite 最大的诱惑是**不需要安装**。但它有一个致命问题：
爬虫多线程写数据库时会锁文件，5 个人开发时不能共享同一个数据库文件。
所以**开发阶段可以用 SQLite 快速验证，但答辩演示必须切到 MySQL**。

### 为什么不用 MongoDB？

舆情数据（事件、报道、用户）天然是关系型的——一个事件有多篇报道，一篇报道属于一个事件。MongoDB 处理这种关联关系不如关系型数据库直觉。而且题目要求中没有非结构化数据存储的需求。

### 为什么用 SQLAlchemy ORM？

- **自动防 SQL 注入**——这是题目安全要求的直接解法
- 写 Python 类 = 建数据库表，不用手写 CREATE TABLE
- BettaFish 全程用 SQLAlchemy 做数据库操作，可以直接参考

## 四、爬虫框架：requests+BS4 vs Scrapy vs Playwright vs Selenium

先说结论：**先用 requests + BeautifulSoup，遇到 JS 渲染的问题再加 Playwright。**

### 四种方案的区别？

| | requests+BS4 | Scrapy | Playwright | Selenium |
|------|-------------|--------|------------|----------|
| **原理** | 直接发 HTTP 请求拿 HTML | 爬虫框架，调度/去重/管道化管理 | 启动真实浏览器，模拟点击和滚动 | 启动真实浏览器（老牌方案）|
| **能处理 JS 渲染吗** | ❌ 不能 | ❌ 不能（需加中间件）| ✅ 能 | ✅ 能 |
| **速度** | 极快 | 快（异步）| 慢（要启动浏览器）| 最慢 |
| **资源消耗** | 极低 | 低 | 高 | 最高 |
| **学习成本** | 极低（半天）| 高（框架概念多）| 中 | 中 |
| **反爬能力** | 弱 | 中 | 强（真浏览器行为）| 强 |
| **参考项目** | stay-leave、weiboSpider | Astianjy | BettaFish、MediaCrawler、MindSpider | — |

### 为什么先从 requests+BS4 开始？

```
简单站（微博热搜榜、知乎热榜、百度热搜）：
  直接 requests.get(URL) → BS4 解析 HTML → 提取标题/链接 → 搞定
  全程几行代码，调试简单，速度极快

复杂站（需要登录、内容由 JS 动态加载）：
  requests 拿到的 HTML 是空的 → 需要 Playwright 启动浏览器 → 等 JS 渲染 → 再抓数据
```

关键认知：**不是所有网站都需要 Playwright。** 先区分目标网站是服务端渲染（直接拿 HTML）还是客户端渲染（JS 动态生成），前者用 requests 足够，后者才上 Playwright。

参考 stay-leave 的项目：它的 weibo-crawler 全部用 requests 就完成了微博爬取。

### Playwright vs Selenium：为什么选 Playwright？

MediaCrawler 和 BettaFish 都用 Playwright，理由：
- Playwright 比 Selenium 新，API 设计更现代，一个 pip install 完事
- 自动管理浏览器驱动，不需要像 Selenium 那样单独下载 chromedriver
- 速度比 Selenium 快

## 六、情感分析：SnowNLP vs 基于词典 vs LLM API vs 训练模型

先说结论：**SnowNLP 做基线，LLM API 做主力，二者互补。**

### 四种方案的区别？

| | SnowNLP | 基于情感词典 | LLM API | 训练/微调 BERT |
|------|---------|-------------|---------|----------------|
| **原理** | 基于贝叶斯模型，自带训练好的模型 | 匹配正/负向词，计算得分 | 把文本发给大模型，让它判断 | 用标注数据训练分类器 |
| **准确率** | 一般（70%左右）| 低（取决于词典质量）| 高（90%+）| 高（取决于标注数据）|
| **成本** | 零（离线运行）| 零 | 每条几分钱 | 零推理成本，但训练需要GPU和时间 |
| **速度** | 快 | 快 | 慢（网络延迟）| 快（训练好后）|
| **数据要求** | 无 | 需要准备情感词典 | 无 | 需要几千条标注数据 |
| **代码量** | 3行 | 几十行 | 几行（调API）| 几百行 |
| **参考项目** | stay-leave、Astianjy | stay-leave（正/负向语料）| BettaFish | BettaFish（微调BERT/GPT-2/Qwen）|

### 为什么组合使用？

```
快速出结果阶段（Day 1-3）：
  爬完数据 → SnowNLP 直接跑 → 立刻看到情感分布 → 前端可以先联调界面

提高准确率阶段（Day 4-5）：
  智能问答模块已经接入了 LLM API → 情感分析也复用同一个 API → 
  写个 Prompt："判断以下微博内容的情感倾向，输出正面/负面/中性" →
  准确率从 70% → 90%+

为什么不训练模型？
  BettaFish 的 SentimentAnalysisModel 目录下有 5 套情感分析方案，
  其中 4 套是基于微调/训练的模型。但那是人家长期迭代的结果。
  7 天内最务实的选择是：LLM API 足够准，零训练成本。
```

## 七、LLM 大模型：DeepSeek vs 通义千问 vs Kimi vs GPT

先说结论：**DeepSeek API，OpenAI 兼容格式。**

### 几个主流选项对比？

| | DeepSeek | 通义千问（Qwen）| Kimi（Moonshot）| GPT-4o |
|------|---------|-----------------|-----------------|--------|
| **中文能力** | 极强 | 极强 | 强 | 强 |
| **价格** | 极便宜 | 便宜 | 中等 | 贵 |
| **API兼容性** | 完全兼容 OpenAI 格式 | 兼容 | 兼容 | 原生 |
| **参考项目使用** | BettaFish 的 QueryEngine | BettaFish 的 ForumEngine（硅基流动的Qwen）| BettaFish 的 InsightEngine | — |

### 为什么选 DeepSeek？

```
1. 中文舆情分析 —— DeepSeek 的中文理解和生成是第一梯队的
2. 极便宜 —— 100万token几块钱，课程项目爬几千条数据分析花不到几块钱
3. OpenAI 格式 —— 一行代码切换模型，如果 DeepSeek 挂了可以立刻切换到通义千问
4. BettaFish 和 MindSpider 都在用，可靠性经过验证
```

### 最关键的设计：OpenAI 兼容格式的抽象

这是从 BettaFish 学到的最有价值的设计模式：

```python
# 不这样写（绑定死一个厂商）：
import deepseek
client = deepseek.Client(api_key="...")

# 而是这样写（通用，换模型只改配置）：
from openai import OpenAI
client = OpenAI(
    api_key=config.LLM_API_KEY,      # 读 .env，不硬编码
    base_url=config.LLM_BASE_URL,    # https://api.deepseek.com
)
response = client.chat.completions.create(
    model=config.LLM_MODEL_NAME,     # deepseek-chat
    messages=[{"role": "user", "content": prompt}]
)
```

这样做的好处：DeepSeek 某天挂了/没钱了/想换模型 → **只改 .env 文件的一行 URL，代码一行不动。**

## 八、主题分类/热点发现：LDA vs LLM vs 传统聚类

先说结论：**先用 LLM API 做主题分类，如果需要无监督发现新主题再加 LDA。**

### 方案区别？

| | LDA | LLM API | K-Means + TF-IDF |
|------|-----|---------|------------------|
| **原理** | 概率模型，自动发现文档中的隐含主题 | 直接把文章发给大模型，让它归类 | 向量化 → 计算距离 → 聚成几类 |
| **优点** | 无监督，不需要预定义类别，不需标注数据 | 最准确，能理解上下文和隐喻 | 简单直接 |
| **缺点** | 主题可解释性差（输出是一堆概率词），需要调参 | 成本高（每篇文章调一次API）| 需要预定义K值，中文文本直接TF-IDF效果差 |
| **参考项目** | stay-leave | MindSpider、BettaFish | — |

### 为什么组合使用？

```
LLM API 方案（主要）：
  文章内容 → Prompt："这篇文章属于以下哪个类别：社会事件/公共卫生/自然灾害/政治事件/经济事件/其他" → 返回类别
  每篇文章调一次 API，成本几厘钱。对课程项目几百条数据来说，总成本不到一块钱。

LDA 方案（备选）：
  当不知道有哪些类别时，用 LDA 无监督跑一遍 → 自动发现 N 个主题 → 
  每个主题输出 Top 10 关键词 → 人工命名主题 → 作为 LLM 分类的类别列表

关键洞察：
  题目要求"利用机器学习方法对舆情主题进行分类"——
  这里既可以用 LLM（现代方法），也可以用 LDA（经典方法）。
  推荐两个都做：LDA 满足"机器学习方法"的要求，
  LLM 提供更高的准确率，互相补充。
```

stay-leave 的项目里 LDA 实现非常完整：LDA.py + 超参调优 + 分词处理 + 自定义词表 + 停用词表 + 近义词表，可以直接参考。

## 九、趋势预测：规则引擎 vs 时间序列模型 vs LLM 推理

先说结论：**先用规则引擎判断生命周期阶段，有精力再上 Prophet 做数值预测。**

### 方案区别？

| | 规则引擎 | Prophet/ARIMA | LLM 推理 |
|------|---------|---------------|----------|
| **原理** | 人工定义规则（如"报道量连续3天下降且低于峰值30% = 衰退期"）| 统计模型拟合时间序列，预测未来值 | 把趋势数据发给LLM，让它推理 |
| **准确率** | 中等（取决于规则质量）| 中高（需要足够数据点）| 不可控 |
| **实现难度** | 极低（几个 if/else）| 中（需理解时间序列概念）| 低（调API）|
| **数据要求** | 无 | 需要至少几十个时间点 | 无 |
| **可解释性** | 完全可控 | 模型可解释 | 黑盒 |
| **参考项目** | stay-leave（热度_1/2/3.py）| — | BettaFish |

### 具体怎么做？

```
题目要求判断四个生命周期：潜伏期 → 成长期 → 高潮期 → 衰退期

规则引擎（Day 1-4 完成）：
  每天计算 报道量 和 增长率
  - 潜伏期：报道量 < 阈值1 且 增长率 < 阈值2
  - 成长期：增长率 > 阈值3，持续上升
  - 高潮期：报道量 > 阈值4，增长率趋近于0
  - 衰退期：报道量下降，连续N天低于峰值的一定比例

  阈值怎么定？——先用 stay-leave 的三版热度公式思路跑一遍数据，人工观察设定。

Prophet 升级（Day 5-6，如果有时间）：
  输入：过去 N 天的每日报道量序列
  输出：未来 3 天的预测报道量
  结合规则引擎判断阶段

LLM（不推荐用在预测上）：
  LLM 对数值预测不可靠。更适合用来生成趋势解读文本——
  比如"本事件正处于高潮期，预计未来3天报道量将保持高位……"
```

## 十、数据可视化：ECharts vs Matplotlib vs Plotly

先说结论：**ECharts（前端渲染），不要在后端画图。**

### 方案区别？

| | ECharts（前端）| Matplotlib（后端）| Plotly（后端）|
|------|--------------|------------------|---------------|
| **运行位置** | 浏览器中渲染 | Python 代码生成 PNG 图片 | Python 代码生成 HTML/图片 |
| **交互** | 天生交互（hover/缩放/点击）| 静态图片 | 交互式 |
| **Web集成** | CDN引入，前端代码控制 | 后端生成图片 → 前端 `<img>` 显示 | 后端生成 HTML → 前端嵌入 |
| **与Flask的配合** | API 返回 JSON 数据 → ECharts 绑定渲染 | 复杂，需要管理图片文件 | 还行 |
| **参考项目** | — | stay-leave、yutao-arch、BettaFish（部分）| BettaFish（报告渲染）|

### 为什么选 ECharts？

核心原因：**题目要求的所有图表类型（趋势图、饼图、词云），ECharts 一个库全部覆盖，且全部支持交互。**

如果你在 Python 后端用 Matplotlib 生成图片：
```
后端：数据分析 → Matplotlib 画图 → 保存为 trend.png → 前端 <img src="/static/trend.png">
问题：数据更新了要重新生成图片；不能 hover 看具体数值；关键时间节点标注麻烦
```

如果你用 ECharts 前端渲染：
```
后端：提供 /api/events/1/trend → 返回 JSON [{date:"7月1日", count:120}, ...]
前端：ECharts 绑定 JSON → 折线图 → 自动支持 hover/缩放/标注
```

BettaFish 的最终报告是用 Plotly 生成 HTML 的（ReportEngine 渲染），但那是因为它要做 PDF 导出。如果不需要 PDF 导出，ECharts 前端渲染是最优解。

## 十一、部署方式：Docker vs 直接运行

**不需要 Docker。** 答辩时一台电脑本地启动 Flask + MySQL，浏览器访问 localhost:5000 即可。

BettaFish 提供 Docker 部署是因为它要给别人用。课程项目只有你们自己跑，Docker 只会增加一个学习成本。

## 附录：一个重要原则

```
技术选型的判断标准不是"这个技术好不好"，而是：

1. 团队里有人会吗？            → 会什么用什么
2. 7天内能调通吗？             → 够用就停，不要过度设计
3. 答辩时能稳定演示吗？        → 优先稳定性，别让 API 超时/爬虫被封影响答辩
4. 参考项目是怎么做的？        → 别人踩过的坑不要重踩
```

每当你纠结两个技术选项时，用这四个问题过一遍，选那个能让你最快看到结果、答辩时不会翻车的。
