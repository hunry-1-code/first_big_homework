# 舆情分析系统后端现状与接口说明

## 1. 文档说明

本文按当前代码实际情况整理，主要面向前端开发、后端维护和课程验收。文档不会把“接口存在”直接写成“功能完整”：已经跑通的功能标为“已实现”，能够使用但仍有明显限制的标为“部分实现”，只有路由或预留结构的标为“未完整实现”。

截至 2026 年 7 月 13 日，基线回归修复后的后端全量自动化测试结果为 `270 passed, 5 failed, 5 warnings`。剩余失败已经收敛到传播图误连边和生命周期判定两个独立模块，不能再沿用旧的 `263 passed, 0 failed` 结论。另外历史上执行过三轮真实关键词搜索流程和三轮真实热榜流程；新的密钥安全真实验证将在自动化测试归零后重新执行。

## 2. 技术栈与目录结构

后端使用 Flask 提供 REST API，SQLAlchemy 负责数据持久化，默认开发数据库为 SQLite。文本处理使用 jieba、TF-IDF、LDA、SnowNLP 等组件；BGE 向量模型可以通过配置启用；需要大模型的地方统一使用 OpenAI-compatible 接口，目前已经用 `.env` 中的 DeepSeek 配置完成真实调用。

主要目录如下：

```text
backend/
├─ app/
│  ├─ api/              HTTP 路由、权限检查和参数接收
│  ├─ services/         业务编排、数据库读写、跨模块流程
│  ├─ analysis/         分词、特征、聚类、热点、情感、趋势、虚假检测
│  ├─ preprocessing/    正文提取、标准化、清洗、质量评估、去重
│  ├─ crawler/          各平台爬虫适配器
│  ├─ propagation/      传播节点、边和关键节点评分
│  ├─ llm/              OpenAI-compatible LLM 客户端
│  ├─ models/           SQLAlchemy 数据模型
│  ├─ tasks/            后台任务执行、心跳和恢复
│  └─ core/             配置、认证、统一响应
├─ migrations/          数据库迁移脚本
├─ tests/               自动化测试
└─ .env                 数据库、爬虫和 LLM 配置
```

## 3. 整体架构

系统采用“API 层—服务层—算法层—持久化层”的结构。

```text
Vue 前端或其他客户端
        │ JWT + JSON
        ▼
Flask API /api/*
        │
        ├─认证和参数校验
        ├─同步查询
        └─创建后台 Task
                │
                ▼
业务服务 services
        │
        ├─爬虫与预处理
        ├─内容分析
        ├─主题和事件聚合
        ├─情感、热度、趋势和风险
        └─事件详情、问答和报告
                │
        ┌───────┴────────┐
        ▼                ▼
算法组件 analysis     SQLAlchemy models
        │                │
        └───────┬────────┘
                ▼
        SQLite / MySQL 等数据库
                │
                ├─外部新闻和平台 API
                └─DeepSeek/OpenAI-compatible LLM
```

API 层不直接实现复杂算法。以关键词搜索为例，路由只负责创建爬虫任务；任务依次调用采集、预处理、内容分析、聚合和情感服务。正式发布搜索聚类时，系统还会补齐事件级情感快照、热点热度和报告，避免各模块独立运行后数据断开。

## 4. 实际业务流程

### 4.1 关键词搜索流程

```text
POST /api/crawler/search
→ 多平台采集
→ 正文提取、标准化、清洗、去重和质量评分
→ 分词、关键词、TF-IDF/BGE 特征
→ 搜索作用域事件聚类
→ 文章及聚类情感分析
→ 管理员发布聚类
→ 创建正式 Event
→ 正式事件情感快照
→ 报告和下游事件接口
```

搜索结果默认属于共享搜索作用域，不会自动成为首页正式事件。管理员调用发布接口后，文章才会绑定正式 `event_id`。

### 4.2 热榜发现流程

```text
POST /api/crawler/trigger
→ 微博/百度/知乎热榜种子
→ 根据种子扩展多个搜索平台
→ 清洗和内容分析
→ LDA 主题发现与主题命名
→ 正式事件聚合
→ 事件热度计算与排名
→ 情感分析
→ 首页热点接口
```

真实测试中三轮热点流程的 `topic_status` 和 `heat_status` 均为 `success`。

### 4.3 后台任务机制

爬虫、分析、热点、聚合和情感计算使用统一 Task 模型。任务包含状态、进度、结果、租约、心跳和失败信息。服务启动时可以扫描长时间未更新的任务并恢复执行。

开发和测试可以设置 `TASKS_RUN_SYNC=true` 同步执行。同步 SQLite 模式不会再启动独立心跳写线程，以免心跳更新与主任务事务竞争数据库锁。生产环境建议使用 MySQL 或 PostgreSQL，并保持异步任务执行。

## 5. 功能模块完成情况

| 模块 | 状态 | 当前能力 | 主要限制 |
|---|---|---|---|
| 用户认证 | 已实现 | 用户名密码登录、PBKDF2-SHA256 密码验证、JWT、当前用户查询、禁用用户拦截 | 没有刷新令牌和找回密码流程 |
| 用户管理 | 已实现 | 管理员分页查询、创建、编辑、重置密码、删除、角色列表 | 角色目前只有 `admin`、`user` |
| 用户配置 | 已实现 | 个人资料、采集配置和数据源预设读写 | 配置项仍以当前前端需求为主 |
| 网络爬虫 | 已实现但依赖外部服务 | 百度、知乎、微博、B站、小红书、抖音、RSS、热榜采集及统一格式 | 抖音真实扩展请求出现 403；小红书部分关键词可能返回空数据；外部 SSL、限流和配额会造成降级 |
| 数据清洗与预处理 | 已实现 | 正文提取、字段标准化、时间和互动量解析、噪声清理、URL/标题/语义去重、质量评分 | 动态页面正文质量仍受上游内容影响 |
| 内容分析 | 已实现 | 自动分词、关键词、TF-IDF、n-gram、可选 BGE 表示、分析快照 | BGE 首次下载和推理成本较高，可关闭后回退 TF-IDF |
| 热点主题发现 | 已实现 | LDA 主题数选择、主题指标、LLM 命名降级、热度快照和排名 | 小样本会产生 `SMALL_CORPUS` 警告；热度质量依赖真实发布时间和跨平台数据量 |
| 舆情事件聚合 | 已实现 | 搜索/热点/手动/发布作用域，事件创建、文章归属、合并候选、发布和幂等处理 | 聚类阈值仍需要用真实标注事件持续校准 |
| 历史及相似事件检索 | 已实现 | 时间、标题、关键词、主题和向量相似性检索 | 事件库较小时结果可能为空，不代表接口异常 |
| 情感倾向分析 | 已实现 | DeepSeek LLM 结构化判断、正负中立、置信度、文章结果缓存、事件快照、平台和日期聚合、SnowNLP 降级 | 小平台样本不足时会标记警告；情感不是事实真假判断 |
| 趋势与生命周期 | 部分实现 | 按时间统计报道量和热度，输出潜伏期、成长期、高潮期、衰退期 | 当前属于统计和规则型预测，不是经过大规模历史数据训练的时序模型；单时间点事件只能给出有限判断 |
| 虚假/可疑文本检测 | 部分实现 | 来源可信度、多源验证、官方媒体、传播速度、情感、煽动标题、广告引流、外链、标题正文一致性和高风险 LLM 辅助 | 本质是风险评估，不是事实核查；没有接入权威事实库，也没有标注数据集上的准确率、召回率评测 |
| 事件溯源与传播路径 | 已实现但属于推断图 | 识别起源候选、大 V 放大、媒体介入、官方回应和峰值内容，输出节点、边、阶段及解释 | 缺少平台原生转发链时，边由时间、文本和互动特征推断，不能等同于真实转发关系 |
| 智能问答 | 已实现 | 使用事件标题、摘要、生命周期、情感和报道正文调用真实 LLM；失败时明确降级 | `/ask/stream` 目前不是逐 token 流式响应，只是复用普通问答并保留路由 |
| 报告生成 | 部分实现 | 事件概述、趋势、情感、平台、风险数据组合及 HTML 导出 | `format=pdf` 当前仍返回 HTML，不是真正 PDF；报告关键词可能为空，长篇 LLM 自动撰写尚未实现 |
| 数据导入 | 已实现 | JSON 文档校验、标准化和入库 | 当前只提供 JSON 接口，没有 CSV/Excel 导入接口 |
| 前端数据契约 | 已实现 | 统一 `{code,message,data}`，平台、情感、生命周期和风险字段标准化 | 个别旧代码中文字符串存在历史编码显示问题，建议后续统一源码编码 |

## 6. 关键数据模型

后端并不是把所有结果都塞入 Article 表，而是保留运行记录和快照，便于追踪某次分析使用了哪些数据和配置。

| 数据模型 | 用途 |
|---|---|
| `User` | 用户、角色、状态和登录信息 |
| `UserConfig` | 用户采集和展示配置 |
| `Task` | 后台任务状态、进度、租约、心跳和结果 |
| `Article` | 统一文章、平台字段、清洗结果、事件归属、情感和风险摘要 |
| `ArticleSnapshot` | 文章处理时的内容快照 |
| `DocumentFeatures` | 分词、关键词、实体和特征数据 |
| `ArticleEmbedding` | BGE 等向量表示 |
| `AnalysisRun` | 一次内容分析的数据集和配置记录 |
| `HotspotRun`、`TopicResult` | 热点主题运行、主题和文章分配 |
| `AggregationRun`、`AggregationCluster` | 聚合运行、事件簇和作用域 |
| `Event` | 正式事件当前摘要、热度、情感和生命周期 |
| `EventArticleMembership` | 文章与正式事件的可追踪归属关系 |
| `EventHeatSnapshot` | 某一时刻的事件热度和排名依据 |
| `SentimentRun` | 一次情感分析运行 |
| `ArticleSentimentResult` | 文章级情感、模型和缓存来源 |
| `EventSentimentSnapshot` | 事件级情感比例、趋势和平台分布 |
| `Report` | 事件报告的持久化数据 |

## 7. 前后端通用约定

### 7.1 基础地址与认证

开发环境默认前端地址为 `http://localhost:5173`，所有业务接口以 `/api` 开头。除登录外，大多数接口都要求：

```http
Authorization: Bearer <token>
Content-Type: application/json
```

管理员接口必须使用 `role=admin` 的令牌。

### 7.2 统一响应格式

成功和失败均使用同一外层结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

前端应同时检查 HTTP 状态码和响应中的 `code`。任务创建接口通常只立即返回 `task_id`，实际结果需要查询任务详情。

### 7.3 推荐的前端任务调用方式

```text
1. POST 创建爬虫/分析任务
2. 取得 task_id
3. GET /api/tasks/{task_id}
4. status=pending/running 时继续轮询
5. status=success 时读取 result
6. status=failed 时展示 message 和 result.error
```

同步开发模式下，POST 返回时任务可能已经完成，但前端仍应按相同方式处理，避免切换到异步部署后改接口逻辑。

## 9. `.env` 关键配置

不要把真实 API Key 写入文档、前端代码或 Git。后端会自动加载 `backend/.env`。

```dotenv
DATABASE_URL=sqlite:///opinion_analysis_dev.db
FRONTEND_ORIGIN=http://localhost:5173

JWT_SECRET_KEY=replace-me
DEMO_ADMIN_USERNAME=admin
DEMO_ADMIN_PASSWORD=replace-me

LLM_API_KEY=...
LLM_BASE_URL=https://api.deepseek.com
LLM_MODEL_NAME=deepseek-chat

TIKHUB_API_KEY=...
TIKHUB_WEIBO_API_KEY=...
TIKHUB_XIAOHONGSHU_API_KEY=...
TIKHUB_DOUYIN_API_KEY=...

QIANFAN_API_KEY=...
ZHIHU_ACCESS_SECRET=...

TASKS_RUN_SYNC=false
BGE_ENABLED=true
```

前端只需要知道后端地址和 API 数据格式，不应读取这些服务端密钥。

服务存活检查使用 `GET /api/health`。该接口不需要登录，适合前端启动检查、反向代理健康检查和部署平台探针；它只能证明 Flask 服务能够响应，不能代替爬虫、数据库和 LLM 的业务可用性检查。

## 10. 已验证结果与当前风险

最近一次全量测试：

```text
270 passed
5 failed

```

失败项为：不相关文章被传播图错误连接 1 项；生命周期“衰退期/消退期”旧命名 2 项；低报道量错误判为高潮期 1 项；稳定高位错误判为成长期 1 项。这些问题进入核心算法可靠性修复阶段，当前文档不将其描述为已完成。

真实关键词流程连续三轮均完成：四个平台各采集 3 条、每轮处理 12 条，正式事件情感和报告后处理成功，问答真实调用 `deepseek-v4-flash`，事件详情、趋势、情感、传播图、相似事件和报告接口均返回 200。

真实热点流程连续三轮完成，扩展报道数分别为 36、36、39，主题发现和热度计算均成功。测试同时说明系统仍需面对外部平台的不稳定性：抖音扩展接口出现 403，小红书对部分热榜词返回空数据，TikHub 曾出现一次 SSL EOF。当前实现会记录平台错误并继续使用其他来源，不会因为一个平台失效而终止整条分析链路。

后续优先级建议如下：

1. 为虚假文本检测引入公开中文事实核查数据集，给出准确率、召回率和 F1，而不只依赖规则测试。
2. 接入平台原生转发或引用关系，提高传播路径边的真实性。
3. 实现真正的 SSE 流式问答和 PDF 导出。
4. 补齐事件关键词接口，并让报告概述、关键词和风险结论形成稳定快照。
5. 使用历史事件序列训练或校准生命周期预测模型。
6. 生产部署改用 PostgreSQL/MySQL 和独立任务队列，避免 SQLite 的并发限制。

## 11. 总结

当前后端已经不是一组互不关联的演示接口。关键词搜索和热榜采集都能带动后续清洗、分析、聚合、情感和事件数据生成，正式事件又可以继续提供趋势、风险、传播、相似事件、问答和报告接口。核心业务链路已经跑通。

需要保持清醒的是，虚假检测、传播路径和生命周期预测目前仍带有明显的算法推断性质。它们可以作为舆情研判辅助，但不能包装成事实核查结论或真实平台转发链。前端展示这些数据时，建议使用“风险评分”“起源候选”“推断传播关系”“生命周期判断”等措辞。
