# 舆情系统八项改进设计

## 目标与验收边界

本轮工作落实《改进指导书》的八项要求，目标不是增加演示数据，而是保证采集、分析、展示和后台任务使用同一批真实数据。所有结论必须能回溯到文章、评论、聚合分配或规则证据；LLM 只做基于已采集证据的结构化解释，失败时保留可复现的规则结果。

验收分为四条工作流：

1. 数据真实性：主流新闻文章和评论进入清洗、关键词、聚合、情感、风险、传播、生命周期及前端平台分布；前端不再伪造缺失平台或趋势。
2. 解释增强：传播图先由显式引用、时间、语义、关键词和实体生成，再由 LLM 解释高频词关系、源头候选和关键路径；生命周期先由多信号规则判定，再由 LLM 给出趋势解释和下一阶段理由。
3. 采集与算法验证：验证 B 站公开评论接口；用真实管线输入测试虚假文本检测和 HDBSCAN，输出包含输入、过程和结果的 Markdown 报告。
4. 热点闭环与稳定性：热榜只负责采集、融合、去重和分类；详情采集改为用户逐条手动触发；统一调度器生命周期，检查前后端契约、死代码、异常任务和残留进程。

## 一、数据真实性与前后端契约

### 主流新闻

`mainstream_news` 是用户看到的单个平台卡片，内部采集人民网、36氪、澎湃、InfoQ、少数派。落库后保留实际 `news_*` 平台代码，后续分析与其他平台共用 `Article`、`Comment`、`AnalysisRunArticle`、`AggregationAssignment` 和 `Event` 数据模型。

评论采集采用站点能力状态：`success`、`empty`、`unsupported`、`failed`。公开评论可用时落入 `Comment`；站点没有公开接口时明确展示不支持，不能把“无接口”伪装成“0 条评论”。

事件详情的平台分布只来自后端 `Article.platform` 统计。前端可以从文章列表做等价回退，但不得补齐未出现的平台。没有数据时展示空态。

### API 契约

事件详情继续返回：

- `platform.platforms[]`: `platform`、`count`、`percentage`
- `articles.articles[]`: 实际平台、标题、互动数据、发布时间
- `public_opinion`: 评论数量、情感、关键词、诉求和覆盖提示

传播接口增加可选解释字段，不破坏现有图结构：

- `origin_analysis`: 源头候选、依据、限制、方法
- `keyword_relations`: 高频词、共现/关联理由、关联文章
- `llm_analysis`: 状态、模型、摘要、关键路径解释、证据引用

事件详情生命周期增加：

- `lifecycle_evidence.momentum`
- `lifecycle_evidence.next_stage_hint`
- `lifecycle_evidence.trend_explanation`
- `lifecycle_evidence.llm_status`

## 二、传播溯源与生命周期增强

### 传播路径

传播图仍由确定性规则生成，边的证据包括显式转发/引用、时间先后、文本相似、关键词/实体交集和跨平台扩散。新增关键词关系计算：统计事件高频词在文章中的共现，输出每对词的共同文章数、共同平台和示例标题。

LLM 输入严格限制为：事件标题、按时间排序的文章摘要、规则边、源头候选、关键词关系和已知限制。输出必须为 JSON，并校验候选文章 ID、路径文章 ID和证据引用确实存在。任何非法输出或请求失败都标记为 `unavailable`/`invalid`，不覆盖规则图。

DeepSeek API 不被假定有联网搜索能力。所谓“溯源”仅表示对本系统已采集材料的最早来源候选和传播链分析；前端必须显示这一限制。

### 生命周期和趋势

规则层统一按日构建报道量、评论量、情感极化和平台数序列，避免四个序列日期错位。监控模式使用四阶段算法；单次搜索使用“当前活跃度快照”，但仍返回动量和下一阶段提示，不冒充完整历史生命周期。

LLM 输入为最近 14 天的多信号序列、规则阶段、置信度、动量和样本限制。LLM 只生成趋势解释、支持/反对证据和下一阶段理由，不直接覆盖规则阶段。低样本时必须明确“不足以可靠预测”。

## 三、B 站、虚假文本和 HDBSCAN 验证

### B 站评论

公开评论流程为 `BV -> aid -> /x/v2/reply`。请求应复用初始化后的公开设备 Cookie，并保留必要 Referer。测试覆盖 BV 转换、顶级评论、内嵌回复、空评论、风控错误和非法 ID。真实网络验证只低频请求一条公开视频，不绕过登录、验证码或访问控制。

### 虚假文本检测

端到端报告使用已采集文章或固定落库样本，记录标题、正文节选、来源平台、交叉平台上下文、规则特征、规则分数、LLM 复核状态、最终判断和证据。报告明确这是“可疑风险检测”，不是事实核查结论。

### HDBSCAN

HDBSCAN 输入必须来自内容分析输出的实际 BGE/TF-IDF 特征。向量矩阵构建需处理维度不一致、全零向量、少量样本和依赖不可用；所有异常均有显式 warning 并回退到贪心聚合。报告记录文章输入、特征维度、簇分配、噪声点、回退原因和最终发布事件。

## 四、每日热点闭环与运行稳定性

### 热点流程

流程改为：

`热榜采集 -> 跨平台融合 -> 主题去重 -> 规则分类 -> 前端按类别筛选 -> 用户选择单条热点 -> 抓取详情并进入完整分析管线`

热榜刷新和定时任务不得自动创建详情补全任务。新增单条热点详情抓取接口，校验条目存在、不是已合并条目、没有正在运行的任务；重复点击返回现有任务。前端按钮按 `pending/running/completed/no_event/failed` 展示不同状态，并在完成后允许进入事件详情。

分类直接写入 `DailyHotItem`，不依赖详情事件已发布后才出现。规则分类返回 `category`、`topic_name`、`topic_keywords`、`confidence`；事件发布后可以用全文重新分类并同步条目。

### 调度和进程

系统只保留 `task_recovery_scheduler` 这一套 APScheduler；管理员开关通过添加、修改或移除同一个 scheduler 中的 `daily-hot-refresh` job 实现，不再创建第二个后台调度器。下次运行时间读取 APScheduler job，不手工用 `datetime.replace` 计算。

后台执行器保持有界队列；修复重复查询和可重复启动问题，并提供幂等 shutdown。最终检查监听端口、Python/Node 进程、任务状态、线程池和调度 job，记录哪些是本项目进程，避免误杀用户其他进程。

## 错误处理与可观察性

- 外部采集失败返回平台、错误码、是否可重试，不泄露 Cookie 或 API Key。
- LLM 输出经过 JSON、类型、ID 引用和长度校验；失败只影响解释字段。
- 手动热点抓取返回任务 ID，任务状态沿用现有任务接口。
- HDBSCAN 回退原因进入 `ClusterResult.warnings` 和验证报告。
- 前端空态区分“没有数据”“接口不支持”“任务仍在运行”和“请求失败”。

## 测试策略

每项生产行为先写失败测试，再做最小实现：

- 后端单元/服务/API：pytest。
- 前端纯展示逻辑：Vitest；再运行 TypeScript 检查和 Vite 构建。
- B 站、主流新闻、热点：低频真实网络冒烟，自动测试使用固定响应。
- 虚假检测与 HDBSCAN：创建可重复的端到端脚本/测试并生成 `docs/verification` 报告。
- 最终全量测试若被既有问题阻塞，先修复与本轮相关或测试已过期的问题；仍有无关问题时在最终报告中列出精确文件和错误。

## 明确不做

- 不声称 LLM 能联网检索或完成事实核查。
- 不抓取需要登录、破解签名、绕过验证码或违反访问控制的评论。
- 不为图表补造平台、趋势、传播边或互动量。
- 不重置或覆盖当前工作区已有改动，不输出 `.env` 中的密钥。
