## 3. 趋势预测设计

### 3.1 输入与输出

`predict_lifecycle_stage(daily_counts)` 接收按时间升序排列的每日有效报道量，输出以下四种中文状态之一：

- `潜伏期`
- `成长期`
- `高潮期`
- `衰退期`

空数据返回潜伏期。负数输入按零处理，避免异常采集值影响判断。

### 3.2 辅助计算

- `_smooth` 使用居中滑动窗口均值降低单日噪声，边界处使用可用的数据点。
- `_growth_rate` 计算相邻平滑值的增长率；前值为零且当前值为正时返回 `1.0`。
- `get_lifecycle_change_points` 对每个时间前缀计算阶段，并在阶段发生变化时返回日期、原阶段和新阶段。

### 3.3 阶段规则

规则以需求书建议阈值和现有测试预期为基准：

- 数据总量和单日量均较低时为潜伏期。
- 连续上升且近期增长明显时为成长期。
- 处于高位、近期波动较小且没有明显下跌时为高潮期。
- 相对历史峰值明显下降，或近期连续下降时为衰退期。

判断顺序应先识别低量潜伏期，再识别从峰值衰退，随后判断高位平台期，最后判断增长期。这样可避免高位下降数据被误判为高潮期。

## 4. 爬虫验证设计

### 4.1 分层验证

每个平台依次通过四层检查：

1. 注册检查：配置存在时适配器出现在 `CrawlerRegistry`。
2. 合同检查：适配器接受统一 `CrawlRequest` 并返回 `CrawlBatch`。
3. 真实请求检查：执行 1～3 次低量请求，确认鉴权、接口路径和真实响应结构。
4. 管线检查：至少一条有效 `RawDocument` 能通过标准化、正文提取、清洗、质量评估、去重和分词。

真实请求使用中性测试关键词“人工智能”，每个平台限制为满足结构验证所需的最小条数。热榜接口不传关键词。

### 4.2 成功标准

平台标记为“通过”需要同时满足：

- HTTP 请求成功或平台明确返回合法的空结果；
- 适配器没有未处理异常；
- 至少在非空结果场景中生成字段合法的 `RawDocument`；
- `platform`、`title`、`source_type`、`raw_json` 等关键字段符合统一合同；
- 有正文或摘要的数据能够进入预处理管线。

合法空结果只能证明接口可访问，不能证明内容映射完整，因此最终报告中会单独标记“接口可用但本次无数据”。

### 4.3 失败分类

- `CODE_ERROR`：请求构造、字段解析、类型转换或项目内部逻辑错误。本轮修复。
- `AUTH_ERROR`：Key 无效、过期或缺少权限。报告真实原因，不替换密钥。
- `QUOTA_ERROR`：套餐额度或接口产品未开通。报告限制，不增加调用次数。
- `REMOTE_CHANGED`：外部接口路径或响应结构变化。依据真实响应修复适配器并补回归测试。
- `PLATFORM_BLOCKED`：验证码、风控或公开接口限制。停止请求，不绕过限制。
- `NETWORK_ERROR`：DNS、TLS、超时等环境问题。有限重试后记录。

## 5. 修复原则

- 所有生产代码修复先有能够复现问题的失败测试。
- 外部 API 测试与单元测试分离，避免日常测试套件自动消耗额度。
- 外部测试不得打印 API Key、Authorization 请求头或完整敏感响应。
- 不绕过验证码、登录限制、签名校验或平台风控。
- 不为了让验收显示成功而使用样例数据替代真实平台响应。
- 只修改与趋势预测和目标平台采集直接相关的代码，不扩展到评论、弹幕或抖音采集。

## 6. 数据与副作用

真实接口验收默认先直接调用适配器并在内存中检查结果，避免污染正式数据库。确认适配器输出正确后，选择每个平台最多一条数据通过预处理管线；除非入库问题本身需要复现，否则不写入当前开发数据库。

付费 API 使用限制：每个平台最多 1～3 次请求，不进行 100 条完整采集，不进行自动翻页压力测试。

## 7. 测试与验证

验证顺序：

1. 运行趋势预测专项测试，保留修复前失败证据。
2. 实现趋势预测并使专项测试通过。
3. 运行不访问外网的完整后端测试，修复由趋势模块暴露的回归。
4. 运行各平台现有适配器测试。
5. 执行单平台真实冒烟测试，逐个平台记录结果。
6. 对发现的内部代码问题添加失败测试、最小修复并回归。
7. 再次运行完整后端测试和真实冒烟测试。

最终交付报告包含：平台、接入方式、注册状态、真实请求状态、返回条数、预处理状态、修复内容和外部限制。
