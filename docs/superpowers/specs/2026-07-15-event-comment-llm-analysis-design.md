# 事件评论 DeepSeek 优先分析设计

## 目标

所有已支持平台的评论保持现有基础入库处理，不复用文章的完整预处理和聚类流程。文章关联到正式事件后，其评论统一进入事件级公众意见分析，并以 DeepSeek 批量情感分析为主、SnowNLP 为失败降级。

## 数据边界

- 适用于 B站、微博、知乎、小红书、抖音及已支持公开评论的新闻网站。
- 评论继续保存正文、作者、点赞数、回复数、父子关系、平台评论 ID 和原始响应。
- 评论不进入文章内容分析、BGE/TF-IDF 文章向量、HDBSCAN 事件聚类，也不增加事件报道数量。
- 评论通过所属文章的 `event_id` 进入事件级分析；没有事件归属的评论保留在库中，待文章关联事件后再分析。

## 情感策略

1. 评论入库时允许 SnowNLP 立即生成临时结果，保证外部 LLM 不可用时仍有可展示数据。
2. 事件发布或文章关联事件后，对该事件未完成 LLM 分析的评论调用 DeepSeek 批量复核。
3. DeepSeek 成功时覆盖临时标签和分数，并将 `analysis_status` 记为 `llm`。
4. DeepSeek 超时、限流、额度不足、响应无法解析或未配置时，保留 SnowNLP 结果，并将状态明确记录为 `snownlp_fallback`，不得伪装为 LLM 结果。
5. 重复执行只处理未完成 LLM 分析的评论；已为 `llm` 的评论不重复消耗额度。

## 事件级分析

事件详情基于其全部关联文章的评论统一计算：

- 原始及加权情感分布；
- 负面率与情感分歧度；
- 评论高频词和公众词云；
- 公众诉求；
- DeepSeek 提取的公众关注主题；
- 媒体报道与公众评论之间的叙事差异；
- 生命周期中的每日评论量和评论热度信号。

B站与其他平台不设置特殊分析分支。只要 B站文章具有 `event_id`，其评论就进入相同事件级查询和分析。

## 触发时机

- 新事件发布完成后，触发该事件评论的 DeepSeek 批量复核，再生成公众主题和叙事差异缓存。
- 已发布事件后来新增评论时，评论保留临时 SnowNLP 结果；下一次事件分析刷新时补做 DeepSeek 复核。
- 为历史数据提供显式补跑入口或维护脚本，使现有评论也能按相同策略升级。

## 性能与失败处理

- 每批评论数量设置上限，避免单次请求过长；按互动量和文本质量优先选择样本。
- 批次失败不得回滚已入库评论，也不得阻止事件发布。
- LLM 失败时保存可诊断但不含密钥的错误摘要，并继续使用 SnowNLP 降级结果。
- 本设计不改变当前评论同步抓取行为；“文本预处理阶段较慢”的性能问题已定位为同步评论请求，后续可独立设计异步采集优化。

## 验收标准

- 任一平台文章关联事件后，其评论能出现在事件公众意见快照中。
- B站评论与新闻、微博等评论使用同一事件分析代码路径。
- DeepSeek 成功时评论状态为 `llm`，并使用其结果。
- DeepSeek 失败时状态为 `snownlp_fallback`，事件分析仍能完成。
- 评论不进入文章聚类，不改变事件报道数。
- 历史评论补跑后可在数据库和事件接口中核对输入、情感结果及分析来源。
