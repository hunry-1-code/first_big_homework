# 后端成熟化与今日 Top10 热点设计

## 1. 目标

本轮工作把项目从“功能链路可演示”提升到“后端行为稳定、算法结果可解释、API 契约可长期维护”的成熟状态。工作范围只包含后端、数据库、测试、接口契约和必要文档，不包含前端视觉设计。

具体目标：

1. 修复当前内容分析回归，恢复爬虫、内容分析、热点、聚合和情感全链路。
2. 根据 `D:\newbigwork\backend_algorithm_audit.md` 修复事件聚合、传播路径、风险评估、生命周期、关键词、情感、热度和 AI 元数据问题。
3. 保持现有前端可消费的核心字段和枚举兼容，只通过新增字段增强可解释性。
4. 新增可靠的“今日 Top10 热点”数据链路，首页不再依赖普通事件列表大水漫灌。
5. 每个修复均采用测试先行，完成局部测试、下游测试和最终全量回归。

## 2. 设计原则

### 2.1 证据优先

- 不把文本相似误称为真实传播关系。
- 不把可疑风险评分误称为权威事实核查。
- 数据不足时返回证据不足状态，不伪造确定结论。
- 所有关键算法结果携带版本、置信度或分项证据。

### 2.2 向后兼容

- `lifecycle_stage` 继续只返回 `潜伏期`、`成长期`、`高潮期`、`消退期`。
- `sentiment_positive`、`sentiment_negative`、`sentiment_neutral` 继续保持 0～1 比例。
- 现有 `GET /api/hotspots` 保留，新接口作为增强而不是破坏式替换。
- 现有事件详情主体结构保持稳定。

### 2.3 可靠降级

- BGE 不可用时允许 TF-IDF 降级，但必须返回警告。
- LLM 不可用时使用确定性规则或已有缓存，不把占位文本当真实结果。
- 单个平台热榜失败时继续使用其他来源，并在响应中列出失败来源。

### 2.4 不引入未经验证的重依赖

SBERT/BGE 适合当前语义聚类和检索；BERTopic 的“嵌入聚类 + c-TF-IDF”有成熟依据，但当前项目的小样本、现有模型依赖和回归范围不适合直接替换整条热点链路。本轮保留现有结构并增强其可靠性。

SnowNLP 保留为最后一级降级。新闻领域 Transformer 通过可插拔接口预留，不在没有领域评估数据时设为默认路径。

## 3. 总体实施顺序

### 阶段 A：恢复全链路基线

先解决当前 31 个失败测试的共同上游问题：

- `ArticleKeyword` 对象不能直接写入 SQLAlchemy JSON 字段。
- JSON 自动刷新发生在 BGE 查询前，导致测试传入的编码器没有被调用。
- 内容分析失败进一步引起热点、任务、缓存、聚合和情感连锁失败。

完成标准：内容分析专项测试通过，BGE 编码与缓存测试通过，热点连锁失败显著收敛。

### 阶段 B：核心算法可靠性

依次修复事件聚合、传播路径和生命周期。这三个模块直接决定事件边界、时间阶段和关系图可信度，优先于展示型优化。

### 阶段 C：分析质量

依次修复风险评估、关键词、情感分析、AI 元数据和热度计算。

### 阶段 D：今日 Top10

建立独立热榜种子快照、跨平台排名融合和异步事件补全链路。

### 阶段 E：契约与全量验证

统一验证 API 字段、权限、错误响应、迁移、任务恢复和全量测试。

## 4. 内容分析基线修复

### 4.1 JSON 序列化边界

`extract_article_keywords()` 继续返回领域对象 `ArticleKeyword`，因为算法层使用结构化对象更清晰。服务层在写入 `AnalysisRunArticle.keywords` 前统一调用 `as_dict()` 转为普通字典。

禁止依赖异常分支才做转换。无论是否启用 BGE，JSON 字段始终只接收字典、列表、字符串、数字、布尔值或空值。

### 4.2 BGE 调用顺序

关键词转换完成后再进入嵌入缓存流程，避免 SQLAlchemy 自动刷新提前序列化领域对象。编码失败时回滚嵌入事务，重新加载运行记录并保留已经转换的关键词。

### 4.3 验证

- 无 BGE 时关键词能持久化。
- 传入 FakeEncoder 时只调用一次并缓存全部向量。
- 编码失败时运行仍成功，并带 `BGE_UNAVAILABLE`。
- 内容版本变化、租约过期等原有保护继续生效。

## 5. 事件聚合与聚类

### 5.1 余弦相似度

余弦相似度的贡献改为：

```text
max(0, raw_cosine)
```

正交向量得分为 0，负相关向量同样不贡献相似度，不再使用 `(cosine + 1) / 2` 膨胀负值。

### 5.2 三段决策区间

`create_threshold < attach_threshold` 的语义固定为：

- `score >= attach_threshold`：加入现有簇。
- `score < create_threshold`：与现有簇明显不同，创建新簇。
- 中间区间：标记为 `ambiguous`，进入第二轮评估，不立即强行创建或加入。

没有任何候选簇时，第一篇有效文档可以创建种子簇。

### 5.3 第二轮重分配

第一轮完成后：

1. 先处理 `ambiguous` 文档，使用已经稳定的全部簇中心重新打分。
2. 再复核已分配文档是否存在显著更优簇。
3. 只有新簇得分至少比原簇高 `move_margin`，且不存在实体硬冲突时才移动。
4. 移动后重新计算受影响簇中心，删除空簇并重新编号。

该过程减少单遍时间顺序依赖，同时避免反复抖动。

### 5.4 时间兼容度

时间得分从超过固定天数直接归零，改为半衰期式平滑衰减：

```text
time_score = exp(-ln(2) * gap_days / half_life_days)
```

现有 `maximum_event_gap_days` 在兼容期内作为默认半衰期使用，后续配置改名时保留旧环境变量映射。

### 5.5 事件类别

使用现有 `topic_category` 归一化到六个大类：自然灾害、公共卫生、社会民生、政治政策、经济市场、科技网络。分类置信度不足时使用通用参数，不加载类别特定参数。

第一版类别适配只用于时间半衰期等有明确量纲的参数，不复制六套未经校准的完整权重表。

## 6. 传播路径

### 6.1 功能语义

传播路径只表示：

- 平台原始转发、引用或父级关系；或
- 同一事件内有充分语义、时间和来源证据的推断跟进关系。

关键词共现图属于主题关系分析，可以独立提供，但不能替换后仍沿用“传播路径”名称。

### 6.2 显式证据

支持：

- `parent_id`、`root_id`、`retweeted_status`、`quoted_status` 等结构化字段。
- 微博 `//@账号`、`转发微博`、`来源：账号` 等文本证据。
- 父节点时间不得晚于子节点。

### 6.3 推断证据

推断边至少满足：

1. 父节点时间早于子节点。
2. 语义相似度达到最低门槛。
3. 具有来源提及、共享实体、标题关键词覆盖或跨平台跟进中的至少一项附加证据。
4. 综合置信度达到配置门槛。

Bigram 只能作为弱证据，不能重复计权。无 BGE 时使用 TF-IDF/关键词/实体组合，不能只凭时间接近连接文章。

### 6.4 响应状态

传播接口新增：

```json
{
  "coverage_status": "sufficient | insufficient",
  "graph_mode": "propagation",
  "limitations": [],
  "summary": {},
  "graph": {"nodes": [], "links": []}
}
```

证据不足时节点仍可返回，边可以为空。

### 6.5 清理

删除 `get_propagation_data()` 中立即返回后的不可达旧实现，只保留一个权威构建器。

## 7. 生命周期

### 7.1 契约

继续返回四阶段，并新增：

```json
{
  "lifecycle_stage": "潜伏期",
  "lifecycle_status": "data_insufficient | classified",
  "lifecycle_confidence": 0.0,
  "lifecycle_evidence": {
    "point_count": 3,
    "peak": 5,
    "recent_slope": 0.0,
    "peak_ratio": 1.0
  }
}
```

### 7.2 判定方法

- 少于最低有效时间点时不强行给高置信分类。
- 使用相对峰值、近期斜率、持续增长/下降天数和数据覆盖度。
- 阶段采用带滞回的单调状态机：潜伏期 → 成长期 → 高潮期 → 消退期。
- 已进入消退期后，若事件重新爆发，应记录为新的活跃周期或明确的重新激活，而不是静默倒退。

### 7.3 持久化时机

生命周期在事件数据或热度快照更新时重新计算并保存。事件详情查询只读取结果，不因浏览页面产生数据库写入。

## 8. 可疑信息风险评估

### 8.1 定位

输出继续命名为“可疑信息风险”，不输出“真实/虚假”的权威结论。

### 8.2 文章级交叉验证

单平台信号不再直接使用整个事件的平台集合。对当前文章寻找：

- 相同显式来源文章；
- 相同重复组或内容哈希文章；
- 高语义相似并共享关键实体的跨平台报道。

返回 `corroborating_platform_count` 和证据文章 ID。

### 8.3 标题正文一致性

优先使用可配置语义编码器比较标题与正文摘要。编码器不可用时，使用中文分词、关键词覆盖和短标题友好的 token recall，禁止用短标题字符集合除以标题字符数的单一规则。

### 8.4 连续特征

作者可信度、来源可追溯性、交叉验证、互动异常、煽动性和广告引流均输出 0～1 特征值。总分由配置权重计算，并保存：

- `risk_score`
- `risk_level`
- `rule_version`
- `feature_scores`
- `evidence`
- `limitations`

没有标注数据前，权重明确标注为经验配置，不宣称经过训练。

### 8.5 LLM 灰区复核

LLM 用于规则结果的灰色区间，不只处理已经很高的分数。LLM 返回必须是结构化 JSON，并且不能覆盖显式证据，只能补充判断和理由。

## 9. 关键词

### 9.1 提取与持久化

- 单篇 LLM 提取改为小批量调用，控制输入长度和批次大小。
- 失败文章继续使用 TF-IDF、实体和主题回退。
- 所有写入 JSON 的关键词先转普通字典。

### 9.2 查询词处理

查询词不再因为覆盖率高而直接删除。精确等于搜索查询的词保留，但可降低展示权重，并标记 `source=query`，避免搜索词压制更具体的事件词。

### 9.3 规范化

增加通用规范化接口：全半角、大小写、空格、常见别名、可配置领域词典。领域词典由配置加载，不把台风等单一领域词直接写死在核心算法中。

### 9.4 摘要传参

事件摘要生成只接收事件或明确的关键词列表，不允许把 `Article` 对象传给期望 `Event` 的函数。

## 10. 情感分析

### 10.1 主路径和降级

顺序为：

1. 已有兼容缓存。
2. LLM 结构化分析。
3. 可选新闻领域 Transformer provider。
4. SnowNLP 最后降级。

SnowNLP 结果带 `DOMAIN_MISMATCH_FALLBACK` 警告并限制置信度。

### 10.2 聚合权重

互动传播权重上限从约 2.0 降为 1.5，防止单篇极端热门内容支配事件整体情感。

### 10.3 摘要一致性

情感摘要从同一持久化快照的比例、主导标签和代表性理由确定性生成，不再额外调用一次可能产生矛盾的 LLM。

## 11. 热度

### 11.1 统一快照

搜索事件发布后也创建热度快照，不再只更新 `event.heat_index`。热度组成：

- 独立报道量；
- 平台覆盖；
- 互动强度的稳健归一化；
- 近期增长；
- 时间衰减。

### 11.2 时间衰减

使用半衰期衰减，不再采用文章数永久累加。没有可靠发布时间时降低时间置信度，并在快照中记录限制。

### 11.3 公式版本

每个快照保存原始统计、分项得分、权重、公式版本和计算时间。`spread_heat` 不应因搜索模式而长期为空。

## 12. AI 元数据

### 12.1 生成时机

事件发布或成员发生实质变化时生成。详情查询不调用 LLM。

### 12.2 确定性优先

- `time_code` 优先取可靠的首发时间范围。
- `location`、`key_figures`、`cause` 使用结构化 LLM 补充。
- LLM 时间与真实发布时间冲突时采用真实发布时间，并记录冲突警告。

### 12.3 结构化输出

提示词要求严格 JSON，包含字段值、置信度和证据文章 ID。解析失败时保留已有值，不用空字符串覆盖。

### 12.4 版本信息

Event 增加元数据生成状态、版本、置信度和更新时间字段，迁移同时支持 SQLite 开发库和 MySQL 生产库。

## 13. 今日 Top10 热点

### 13.1 数据来源

第一批可靠来源：

- `weibo_hot`：微博热搜；
- `baidu_hot`：百度热榜；
- `zhihu_hot`：知乎热榜。

小红书、抖音和 B 站热榜在适配器稳定后作为可选来源，不作为第一阶段可用性的硬依赖。

### 13.2 两层数据模型

新增：

1. `daily_hot_run`：一次热榜采集运行，记录日期、来源、状态、错误和完成时间。
2. `daily_hot_item`：规范化热点条目，记录标题、规范化键、融合分数、最终排名、来源排名、首次/最后发现时间、关联事件和补全状态。

热榜种子与正式 Event 分离，避免原始热词未经分析就污染正式事件库。

### 13.3 排名融合

对多个来源使用 Reciprocal Rank Fusion：

```text
score(item) = sum(1 / (k + source_rank))
```

同一来源只贡献一次。`k` 使用配置值，来源权重只有在真实评估后才调整。相似标题通过规范化文本和语义相似度合并。

### 13.4 异步补全

Top10 首先快速保存和返回。后台按融合排名扩展搜索、内容分析、聚合、情感和正式事件发布。每条热点带：

- `enrichment_status`
- `event_id`
- `analysis_task_id`
- `source_ranks`
- `last_updated_at`

单条补全失败不影响其他热点展示。

### 13.5 API

新增：

```text
GET /api/hotspots/today?limit=10
POST /api/hotspots/today/refresh
```

查询接口要求登录；刷新接口要求管理员权限，返回后台任务 ID，并复用近期相同刷新任务。

响应示例：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "date": "2026-07-13",
    "generated_at": "2026-07-13T12:00:00",
    "stale": false,
    "available_sources": ["weibo_hot", "baidu_hot", "zhihu_hot"],
    "failed_sources": [],
    "items": [
      {
        "rank": 1,
        "title": "热点标题",
        "score": 0.048,
        "source_ranks": {"weibo_hot": 1, "baidu_hot": 5},
        "enrichment_status": "completed",
        "event_id": 12
      }
    ]
  }
}
```

现有 `GET /api/hotspots` 继续返回正式热点事件，并默认支持 `limit=10`。前端首页应优先消费 `/api/hotspots/today`，但本轮只保证后端接口。

### 13.6 刷新策略

- 支持管理员手动刷新。
- APScheduler 定时创建幂等热榜任务。
- 多进程部署时通过数据库任务唯一键/租约避免重复运行。
- 缓存过期仍可返回上次快照，但必须标记 `stale=true`。

## 14. API 契约

### 14.1 生命周期增强字段

事件列表和详情允许新增生命周期证据字段，原字段保持不变。

### 14.2 传播图增强字段

增加覆盖状态、限制说明和证据类型。前端可以安全展示空边图或证据不足状态。

### 14.3 错误响应

继续使用：

```json
{"code": 400, "message": "具体错误", "data": {}}
```

外部平台失败信息必须脱敏，不回显 API Key、Authorization 或完整第三方响应体。

## 15. 数据迁移

迁移遵循：

- 模型定义、MySQL SQL 迁移和 Python 迁移器同步更新。
- SQLite 测试通过 `db.create_all()` 验证新表和字段。
- 迁移脚本可重复执行，已有字段或表不会导致失败。
- 不修改或删除现有用户数据。

## 16. 测试策略

### 16.1 每个问题的红—绿循环

每项修复必须：

1. 写出能够稳定复现问题的失败测试。
2. 确认失败原因与目标问题一致。
3. 编写最小实现。
4. 运行该测试直至通过。
5. 运行同模块全部测试。
6. 运行直接下游模块测试。

### 16.2 算法案例

至少覆盖：

- 自然灾害、经济市场、公共卫生、社会民生等不同事件类型。
- 正交向量、负相关向量和缺失向量。
- 聚类顺序变化后结果稳定。
- 无关文章不产生传播边。
- 明确转发关系优先于推断关系。
- 生命周期低量、持续增长、高位稳定、持续下降和数据不足。
- 单平台但有跨平台相同报道、单平台且无佐证、官媒参与和匿名来源。
- LLM、BGE、第三方平台不可用时的降级。

### 16.3 API 和任务

- 今日 Top10 的排序、去重、来源失败、陈旧缓存和权限。
- 热榜补全任务的幂等、单项失败隔离和租约恢复。
- 事件详情查询不触发 LLM 或数据库写入。
- 响应枚举与前端消费字段保持兼容。

### 16.4 最终验证


## 17. 可观测性与成熟度要求

- 每次算法运行记录版本、配置哈希和警告。

- 后台任务具有幂等键、租约、心跳和失败结果。
- 对无法证明的结果明确返回限制说明。
- 设计文档、接口文档和最新测试结果在实现结束后同步更新。

## 18. 明确不做的事项

- 不修改前端视觉样式或页面布局。
- 不把关键词共现图包装成传播图。
- 不在没有标注数据时宣称风险准确率或事实核查能力。
- 不强制引入 BERTopic、HDBSCAN、Celery、Redis 或新数据库。
- 不把外部第三方热榜聚合服务作为唯一生产数据源。

## 19. 参考依据

- Reimers & Gurevych, Sentence-BERT：语义嵌入可用于高效相似度检索和聚类。
- Grootendorst, BERTopic：文档嵌入聚类与 c-TF-IDF 的主题表示方案。
- Gomez-Rodriguez、Leskovec、Krause, NetInf：传播网络推断依赖大量真实级联和时间序列，本项目当前数据不足以直接采用。
- Cormack、Clarke、Buettcher, Reciprocal Rank Fusion：用于融合多个热榜排名。
- `imsyy/DailyHotApi`：作为多来源热榜接口形态参考，不作为项目唯一数据源。
