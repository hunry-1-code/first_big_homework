# 最终真实验证与算法自评设计

## 1. 目标

在不泄露 `backend/.env`、不污染开发数据库、不意外启动后台任务的前提下，完成今日 Top10 和 LLM 的低频真实验证；随后基于代码、测试、真实调用结果和现有审计材料，对后端全链路算法的科学性、合理性与完整性进行诚实、可追溯的自评。

本阶段不调整前端视觉设计，也不以“测试通过”替代真实准确率、召回率或人工标注评估。

## 2. 当前问题与根因

直接使用默认 `create_app()` 进行一次性 Top10 验证时，应用继承了生产默认配置 `TASK_RECOVER_ON_STARTUP=true`。应用工厂因此启动任务恢复调度器，并可能恢复或创建后台任务。短命令退出时，调度器和任务线程仍在工作，产生 APScheduler 解释器关闭异常、SQLite 锁竞争和不必要的 BGE 模型加载。

Top10 采集本身已经成功，三项直接热榜来源均可用。问题位于“一次性验证进程错误启用了长期后台服务”，而不是热榜融合算法或爬虫失败。

## 3. 方案选择

### 3.1 采用方案

新增专用真实验证工具。工具继承正式配置中的爬虫和 LLM 参数，但覆盖以下运行属性：

- 使用临时 SQLite 数据库；
- `TASK_RECOVER_ON_STARTUP=false`；
- `TASKS_RUN_SYNC=true`；
- `BGE_ENABLED=false`；
- 使用最小请求数量；
- 只输出固定结构的脱敏摘要；
- 无论成功或失败都清理临时数据库。

该方案验证真实适配器、服务和外部接口，同时不改变正式服务器的调度行为。

### 3.2 不采用的方案

不为一次性验证而关闭生产默认调度器。正常后端进程需要任务恢复、租约心跳和定时热榜刷新。

不直接复用跟踪在 Git 中的开发 SQLite 文件。真实验证不应留下 DailyHotRun、Task、Article 或事件补全数据，也不应与正在运行的开发服务争用数据库。

不在验证工具中打印配置对象、环境变量、请求头、原始异常响应或第三方完整 payload。

## 4. Top10 真实验证

验证工具使用三个直接热榜来源：

- `weibo_hot`
- `baidu_hot`
- `zhihu_hot`

每个平台请求 `limit=1`，融合结果限制为 1，RRF 使用正式默认值 `k=60`。验证内容包括：

1. 三个爬虫适配器能否注册；
2. 外部请求是否成功；
3. RawDocument 是否满足字段契约；
4. `collect_daily_hot()` 是否成功持久化独立 Run/Item；
5. 序列化结果是否包含来源诊断、RRF 条目和可空 `event_id`；
6. 错误是否被分类且保持脱敏；
7. 进程是否可以正常退出，不启动恢复调度器或补全任务。

结果写入 JSON 文件，只保存时间、状态、来源、数量、错误分类和契约布尔值。

## 5. LLM 真实验证

LLM 只进行一次最小、确定性的结构化调用，使用应用现有 OpenAI-compatible 客户端和配置，不读取或显示密钥。

验证内容包括：

1. 客户端能够根据正式配置建立请求；
2. 模型返回非空内容；
3. 返回模型名称可记录；
4. 响应可以解析为预期的最小 JSON；
5. 输出文件和控制台不包含 API Key、Authorization、Bearer 或 Cookie；
6. 失败时分类为认证、配额、网络、远端结构变化或模型输出格式问题。

LLM 真实调用只证明“配置和接口当前可用”，不证明模型在舆情任务上的准确率。

## 6. 敏感信息安全

验证产物和待提交文档必须扫描：

- 已知密钥值；
- `Authorization`；
- `Bearer`；
- `Cookie`；
- `api_key`、`access_token`、`secret` 等字段名；
- 完整第三方异常响应。

扫描密钥值时只在内存中从应用配置取得，不打印值本身。检查输出只报告命中数量和文件位置；若发现命中，先清理产物，再允许提交。

真实 `.env` 永远不加入 Git，不修改其内容。

## 7. 算法自评范围

最终文档覆盖以下模块：

1. 多平台采集与热榜适配；
2. 正文提取、清洗、字段标准化、质量权重和去重；
3. jieba 分词、TF-IDF、关键词、实体和 BGE 表示；
4. LDA 主题发现与主题命名；
5. 事件聚合、二次分配、合并候选和相似事件；
6. 今日 Top10 标题归一化与 RRF 融合；
7. LLM/SnowNLP 情感分析与事件聚合；
8. 热度计算与时间衰减；
9. 生命周期阶段判断；
10. 可疑信息风险评估；
11. 传播路径证据图；
12. 事件 AI 元数据、问答和报告；
13. 任务租约、缓存、持久化与 API 契约等工程保障。

## 8. 评价方法

每个模块按三项主维度评分，每项 1～5 分：

- **科学性**：理论依据是否成立，变量和假设是否可解释，是否存在明显概念替代；
- **合理性**：是否适合当前中文网络舆情场景，参数、降级和成本是否符合业务；
- **完整性**：输入输出、异常处理、版本、证据、持久化、测试和接口是否形成闭环。

同时标注证据等级：

- **E0**：仅有设计或经验判断；
- **E1**：单元/契约测试；
- **E2**：集成测试和真实接口连通；
- **E3**：小规模人工案例检查；
- **E4**：有代表性的标注数据评估；
- **E5**：线上长期监控或对照实验。

没有 E4/E5 证据时，不使用“准确”“可靠识别”“科学验证完成”等结论。

## 9. 文档结构

算法自评文档包含：

1. 总体结论和成熟度边界；
2. 全模块评分矩阵；
3. 每个模块的实现摘要；
4. 科学性分析；
5. 工程合理性分析；
6. 完整性和测试证据；
7. 已知局限；
8. 短期、中期和研究型改进建议；
9. 不应对外宣称的能力；
10. 建议的标注集、指标和复现实验。

文档会明确指出风险评估不是事实核查、传播图包含推断边、生命周期是规则状态机、LLM 输出具有模型不确定性，以及跨领域参数尚未通过代表性标注数据校准。

## 10. 验收标准

- 隔离验证工具具有自动化测试；
- Top10 专项和全量测试保持零失败；
- 三平台热榜真实结果有明确状态记录；
- 最小 Top10 服务验证正常退出且不污染跟踪数据库；
- 完成一次低频 LLM 调用或给出明确外部失败分类；
- 所有提交产物通过敏感信息扫描；
- 开发 SQLite 恢复到验证前状态；
- 集成报告、后端现状文档和 Top10 计划记录精确结果；
- 算法自评覆盖全部约定模块，并对证据不足保持保守结论。
