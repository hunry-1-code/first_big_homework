## 三、基础回归与 API 契约修复

### 3.1 内容分析关键词 JSON 持久化

原代码把 `ArticleKeyword` 对象直接写入 SQLAlchemy JSON 字段，运行时会出现：

```text
TypeError: Object of type ArticleKeyword is not JSON serializable
```

这不是一个孤立错误，它会使内容分析任务中断，并继续影响爬虫完整流程、热点发现、聚合、情感分析和搜索缓存。今天将关键词对象转换为可序列化结构后再写入 JSON，同时补充回归测试，保证关键词内容和权重可以正常持久化。

### 3.2 BGE 编码分支恢复

此前测试传入 `FakeEncoder` 后实际调用次数为 0，说明内容分析的新逻辑绕开了 embedding 分支。今天恢复了特征提取流程中对编码器的调用，并验证每批文本只进行预期次数的编码。真实《功夫女足》链路最终生成了 20 条 BGE embedding。

### 3.3 生命周期与情感枚举对齐

生命周期统一使用：

- 潜伏期
- 成长期
- 高潮期
- 消退期

后端内部结果、API 返回值和测试预期不再混用“衰退期/消退期”。情感标签和接口映射也进行了统一，避免前端因枚举不一致出现空状态或错误颜色。

### 3.4 聚合默认参数统一

聚合服务、配置默认值和 API 契约中的相似度阈值、时间窗口等参数已对齐。这样可以避免同一个请求从不同入口进入时，得到不同的默认聚类行为。

## 四、事件聚合与相似事件算法

### 4.1 相似度与时间衰减修正

事件相似度继续综合关键词、标题、文本向量、平台和时间信息，但修正了时间衰减方向和语义组合方式。较近的报道不再因为公式方向错误而获得更低的关联度，时间因素也不再掩盖明显的语义差异。

### 4.2 三段式聚类

聚类过程增加了“明确归属—歧义区—二次重分配”三个阶段：

1. 高于高阈值的文章直接进入对应事件簇。
2. 低于低阈值的文章保持独立或不分配。
3. 位于中间歧义区的文章结合簇中心和上下文进行确定性重分配。

这一调整的目标是减少两个相反问题：阈值太低导致不同事件被合并，阈值太高又会把同一个事件切成大量碎片。所有重分配规则保持确定性，便于测试和复现。

## 五、传播路径证据图

传播图过去会把“暴雨预警”和“电影票房增长”这样的无关文章连接起来。今天重新调整了传播边的最低证据门槛，并拆分时间、语义、来源、内容承接等证据评分。

主要变化包括：

- 语义关联不足时不允许仅凭发布时间接近而连边。
- 删除容易与新实现产生分叉的旧传播逻辑。
- API 增加 `coverage_status` 等证据覆盖状态。
- 证据不足时允许返回节点但不返回边。
- 前端明确提示“当前证据不足，不代表已验证传播路径”。

《功夫女足》真实案例中返回 11 个节点、0 条传播边。这个结果看起来不如一张连满箭头的图丰富，但它更符合证据情况：当前采集数据没有平台原生转发链或足够强的内容承接证据，不应为了展示效果虚构传播关系。

## 六、生命周期判断

生命周期算法不再只根据某一个增长率或绝对报道量直接给出阶段，而是综合观察窗口、当前报道量、近期斜率、峰值位置和峰后衰减情况。

本次修复重点解决：

- 极低报道量被误判为高潮期。
- 长期稳定高位数据被误判为成长期。
- 生命周期名称不一致。
- 接口读取事件详情时重新计算并修改持久化结果。

新的结果同时保存置信度、低样本标志和判定证据。事件详情接口只读取已经持久化的生命周期结果，不再因为一次页面访问而改变数据库中的事件状态。

## 七、可疑信息与风险评估

风险模块被明确定位为“可疑性评估”，不是权威事实核查。今天对它进行了三方面调整。

第一，跨平台验证改到文章级执行。不能因为整个事件覆盖多个平台，就认定事件中的每一篇文章都得到了多平台佐证。

第二，标题正文一致性改为对短标题更友好的判断方式，减少短标题因词数少而被机械判定为不一致。

第三，将部分二值规则改为连续特征，降低阈值附近分数突变。LLM 只处理规则处于灰区且确实需要补充判断的样本，不再对所有文章无差别调用。

前端风险雷达也删除了固定的 `45` 占位值，改为使用后端真实字段推导。没有证据时返回有限的零值，不使用热度等无关指标臆造风险维度。

## 八、关键词、摘要、情感与热度

### 8.1 关键词输入修正

事件摘要和关键词提取不再错误复用不完整输入。查询词会被保留用于解释事件来源，但在最终关键词排名中进行适度降权，避免整个词云只是在重复用户输入。

### 8.2 LLM 关键词批处理与回退

关键词提取增加批量 LLM 调用，并在单篇解析失败时进行逐篇回退。这样可以兼顾调用成本和失败隔离，不会因为一篇文章格式异常导致整个批次没有关键词。

### 8.3 情感聚合修正

事件级情感仍以文章级结果为基础，但限制了传播互动量对权重的影响，避免单篇高互动文章完全主导整体结果。重复的 LLM 摘要调用被移除，确定性结果优先复用。

### 8.4 AI 元数据持久化

事件摘要、AI 关键词、模型名、生成方法等信息写入正式事件。详情接口以读取持久化数据为主，避免用户每次打开页面都触发新的模型调用，也避免同一事件在不同页面看到不一致的摘要。

### 8.5 真实热度快照

通过搜索链路发布的正式事件会生成真实热度快照。趋势接口在没有连续快照时会回退到文章发布日期分布，不再把单个热度快照错误展示成只有一天的趋势。

## 九、今日 Top10 热点链路

主页热点不应直接使用普通正式事件列表按数量铺满。今天新增了独立的“今日 Top10”链路，其数据源为：

- 微博热搜 `weibo_hot`
- 百度热榜 `baidu_hot`
- 知乎热榜 `zhihu_hot`

### 9.1 融合方法

不同平台的热度值、点赞数和榜单分数没有统一量纲，不能直接相加。当前实现先对标题进行归一化和跨平台去重，再使用 Reciprocal Rank Fusion（RRF）融合各平台名次。RRF 主要依赖相对排名，对平台间不可比较的原始分数更稳健。

### 9.2 持久化与失败隔离

新增每日热榜批次和条目模型，以及相应数据库迁移。每次刷新记录成功来源、失败来源、生成时间、缓存状态和融合结果。单一平台失败时仍保留其他成功来源，只有全部来源失败时才返回整体失败。

### 9.3 API 与调度

新增：

```text
GET  /api/hotspots/today
POST /api/hotspots/today/refresh
```

普通登录用户可以读取最近的 Top10，管理员可以主动刷新。后台刷新任务带租约，避免多进程或重复调度同时刷新同一天的榜单。默认结果支持 TTL 和 stale 标记，前端能够区分“最新结果”“可用但已陈旧”和“当前没有缓存”。

### 9.4 独立事件补全

Top10 首先快速保存榜单快照，然后为排名靠前的条目异步启动独立的搜索、清洗、分析、聚合和发布任务。各条热点相互隔离，一条失败不会回滚整批热榜。条目在补全后可带正式 `event_id`，供前端跳转事件详情。

## 十五、当前科学性边界

今天的修改显著提高了系统的工程完整性和结果可解释性，但仍不能把它描述为已经通过真实行业标注集验证的生产级舆情算法。

目前需要保留的限制包括：

1. 《功夫女足》是一次完整案例，不是覆盖不同领域的大规模标注集。
2. 聚类、情感、生命周期、风险和 Top10 排序仍需要人工标注数据进行参数校准和指标评估。
3. 风险分只表示可疑程度，不能代替事实核查机构的结论。
4. 传播图缺少平台原生转发、引用和用户关系数据时，只能构建证据图，不能声称还原了真实传播网络。
5. LLM 和第三方爬虫的一次成功调用只能证明当前配置在该时刻可用，不能保证外部服务长期稳定。
6. `/api/qa/ask/stream` 仍不是真正逐 token 的 SSE 流式输出。
7. 报告 PDF 导出尚未完成独立的文件格式和内容验证。

## 十六、结论

今天的工作把项目从“主要流程能够运行”推进到“核心结果可持久化、关键算法有回归测试、接口语义较一致、外部调用可隔离验证、前端能够表达数据限制”的状态。

最重要的变化不是增加了多少页面或图表，而是减少了几类会误导用户的结果：低数据量不再轻易显示为高潮期，无关文章不再为了形成传播图而强行连边，风险雷达不再使用固定占位值，事件详情不再因读取动作重新计算并改变数据，主页热点也不再直接用普通事件列表代替今日热榜。

当前分支仍为 `branchhu`。今天的 42 个功能、测试和原有文档提交，以及本报告提交，均尚未推送或合并到远端。
