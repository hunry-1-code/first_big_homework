# 事件聚合模块设计方案

## 1. 文档目的

本文档给出“网络舆情事件智能分析系统”事件聚合模块的第一版实现方案。设计以项目当前代码为基础，不重新定义已经完成的数据清洗、内容分析和热点发现职责。

本模块需要完成两类相互关联、但读写边界不同的能力：

1. 将描述同一具体事件的多篇报道归入稳定的 `event.id`。
2. 支持正式历史事件搜索，并为一个事件返回相似但不同的历史事件。

需求层面将二者统一归入“舆情事件聚合模块”；代码内部拆分为事件聚合和事件检索两个子模块，避免只读检索意外修改事件成员关系。

---

## 2. 当前项目基础与缺口

### 2.1 已经具备的基础

当前项目已经实现以下上游能力：

- 文章采集、快照和预处理。
- 分层去重和重复组代表文章选择。
- `analysis_run` 数据快照。
- 任务内 Count/TF-IDF 特征矩阵。
- 文章关键词。
- 本地 BGE 文章向量及缓存，BGE 不可用时允许内容分析降级。
- `hotspot_run`、LDA 主题发现、主题分类和主题命名。
- `topic_article_assignment` 运行级主题分配。
- `event_heat_snapshot` 和事件热度计算。

这些能力提供了聚合所需的文本、向量、主题和时间证据，但没有完成正式的文章—事件归属判断。

### 2.2 当前代码中的明确缺口

`backend/app/analysis/hot_detector.py` 中的 `assign_event_cluster()` 仍是兼容占位实现，没有计算相似度，也不会查找或更新事件。

`backend/app/services/hotspot_service.py` 当前只读取已经存在的 `Article.event_id`。如果输入文章没有事件归属，热点主题发现可以成功，但热度阶段返回：

```text
EVENT_AGGREGATION_PENDING
```

因此当前自动流程实际停在：

```text
爬虫扩展采集
→ 预处理
→ 内容分析
→ 热点主题发现
→ 缺少事件聚合
→ 无法稳定计算事件热度
```

### 2.3 对旧需求描述的修正

旧需求书提出使用事件 TF-IDF 中心向量进行跨历史运行比较。内容分析模块设计已经修正这一点：

- TF-IDF 只在同一个 `analysis_run` 内比较。
- 不同运行的词表不同，禁止直接比较其 TF-IDF 向量。
- 跨运行和历史事件匹配使用固定版本 BGE 表示。
- TF-IDF、关键词、实体和时间信息用于任务内聚类、候选筛选和边界判断。

本设计遵守上述约束，不把任务相关 TF-IDF 向量永久解释为全局事件中心。

## 3. 范围与边界

### 3.1 本模块负责

```text
analysis_run / hotspot_run 冻结输入
→ 创建可复现的 aggregation_run
→ 对代表文章执行确定性 Single-Pass 聚类
→ 匹配正式历史事件或创建新事件
→ 保存运行级事件簇和文章判定明细
→ 维护正式文章—事件成员历史
→ 更新正式事件的版本化 BGE 表示
→ 向热点模块返回稳定事件映射
→ 提供历史事件搜索和相似历史事件 Top 5
```

具体包括：

1. `hot`、`search`、`manual` 和 `publish` 模式的聚合作用域。
2. 运行复用、失败重试、快照验证和任务租约检查。
3. LDA/关键词/实体/时间驱动的候选筛选。
4. 任务内 TF-IDF 与跨运行 BGE 的混合相似度判定。
5. 正式 `event.id` 的稳定维护。
6. 文章归属的创建、保持、移动和延迟处理。
7. 运行内聚类后的合并复核。
8. 正式事件合并候选及管理员确认记录。
9. 24 小时共享搜索聚合缓存。
10. 正式历史事件的关键词搜索、语义重排和相似事件推荐。

### 3.2 本模块不负责

- 爬虫适配器和数据源请求。
- 正文提取、清洗、分词、质量评估和去重。
- Count/TF-IDF 矩阵和文章关键词的构建。
- LDA 主题训练、分类和命名。
- 热度公式、热点证据门槛和 Top 20 排名。
- 情感分析、趋势预测、风险评估和传播路径溯源。
- 使用 LLM 对文章进行两两聚合判断。
- 自动破坏性合并两个已经存在的正式历史事件。

### 3.3 模块内部边界

需求层面统一称为“事件聚合模块”，内部拆分为：

```text
事件聚合子模块
→ 写路径：创建事件、归属文章、更新事件表示

事件检索子模块
→ 读路径：历史搜索、相似事件排序

共享相似度算法
→ BGE、关键词、实体、主题和时间证据
```

聚合使用严格阈值判断“是否为同一事件”；检索使用较宽松的相似度判断“是否具有参考关联”。检索结果不得修改任何文章归属。

## 4. 已确认的核心决策

| 决策 | 第一版方案 |
|------|------------|
| 聚合粒度 | 具体事件级，不把宽泛话题直接作为事件 |
| 核心算法 | 分层候选 + Single-Pass + 混合相似度 + 运行内合并复核 |
| 任务内比较 | TF-IDF、关键词、主题、实体和时间 |
| 跨运行比较 | 固定版本 BGE + 关键词、实体和时间规则 |
| LDA 语义 | 只用于候选缩小和主题证据，不作为稳定事件身份 |
| 正式身份 | `event.id` 跨全局运行稳定 |
| 搜索事件 | 运行级隔离事件簇，不自动写入正式 `Event` |
| 搜索缓存 | 全用户共享，完成后 24 小时内直接复用 |
| 过期缓存 | 先返回旧结果，再后台增量刷新 |
| 用户数据 | 搜索聚合结果共享；搜索历史和权限按用户隔离 |
| 手动发布 | 管理员将搜索/手动事件重新提交全局匹配，不直接复制 |
| 正式事件合并 | 第一版只生成候选，管理员确认后执行 |
| 历史事件检索 | 关键词初筛 + BGE 语义重排 |
| 相似事件 | `related_event`，保持不同 `event.id` |
| BGE 故障 | TF-IDF/关键词/实体/时间降级并记录警告 |
| 向量数据库 | 第一版不引入，候选初筛后在 Python 中计算少量向量 |

## 5. 事件语义与典型边界

### 5.1 具体事件与宽泛主题

以下内容应拆成不同具体事件：

```text
重庆某次暴雨救援
广东某次暴雨停课
```

它们可以属于同一个“自然灾害/暴雨”主题，但地点、主体和事实不同，不能共享 `event.id`。

以下内容可以保持同一事件：

```text
某案件首次通报
当事方回应
调查进展
一审判决
后续处罚
```

前提是人物、机构、地点和核心事实能够指向同一案件。

### 5.2 `same_event` 与 `related_event`

```text
same_event
→ 同一具体事件的新增报道或后续进展
→ 允许加入已有 event.id

related_event
→ 主题、原因或发展模式相似，但事实身份不同
→ 只用于历史事件推荐
→ 不修改任何成员关系
```

例如“2026 年重庆暴雨”和“2025 年重庆暴雨”可能是相似历史事件，但若发生时间和具体事实不同，不能聚合为同一个事件。

### 5.3 重复文章与同一事件报道

去重回答“是否为同一份或近似转载内容”；事件聚合回答“是否描述同一现实事件”。

重复文章不参与事件中心计算，避免转载量改变语义中心；代表文章确定事件后，同一重复组的其他文章继承当前事件归属，用于传播展示和热度统计。

## 6. 总体架构与数据流

### 6.1 建议组件

| 组件 | 职责 |
|------|------|
| `event_clusterer.py` | 确定性 Single-Pass、中心更新、运行内簇合并 |
| `event_similarity.py` | 纯相似度计算、权重归一化、硬冲突规则 |
| `event_candidate_retriever.py` | 根据主题、关键词、实体、时间筛选候选 |
| `event_aggregation_service.py` | 创建运行、加载快照、编排算法、持久化和事务 |
| `event_similarity_service.py` | 历史事件搜索和相似事件 Top 5，只读 |
| `event_service.py` | 对外序列化正式事件及检索结果 |
| `jobs.py` | 后台阶段编排，不放聚类算法 |

纯算法放在 `backend/app/analysis`，不得依赖 Flask 或 SQLAlchemy；运行、数据库和权限逻辑放在 `backend/app/services`。

### 6.2 全局热点闭环

```text
爬虫采集
→ 预处理
→ analysis_run 内容分析
→ hotspot_run LDA 主题发现
→ aggregation_run(scope=global)
→ 正式文章—事件映射
→ 热点模块计算事件热度
→ event_heat_snapshot
→ 公共热点榜单
```

热点模块提供主题证据，聚合模块提供稳定事件映射，热点模块再消费映射计算热度。双方不直接控制对方内部逻辑。

### 6.3 搜索闭环

```text
用户搜索
→ 标准化 query_fingerprint
→ 查询 24 小时共享缓存
   ├─ 新鲜：直接返回
   ├─ 过期：返回旧结果并启动后台刷新
   └─ 未命中：采集和分析
→ aggregation_run(scope=search_shared)
→ 保存运行级搜索事件簇
→ 不修改正式 Event 和公共榜单
```

### 6.4 管理员发布

```text
管理员选择搜索/手动事件簇
→ 冻结成员文章和内容版本
→ 创建 scope=global、mode=publish 的新聚合运行
→ 重新匹配正式历史事件
   ├─ 命中：加入已有 Event
   └─ 未命中：创建新 Event
→ 重新计算正式事件汇总
```

发布不是复制搜索排名，也不能直接把临时簇编号写成正式 `event.id`。

## 7. 数据模型

### 7.1 `aggregation_run`

保存一次可复现的事件聚合运行：

| 字段 | 说明 |
|------|------|
| `id` | 聚合运行 ID |
| `analysis_run_id` | 上游内容分析运行 |
| `hotspot_run_id` | 可空，上游热点主题运行 |
| `source_task_id` | 后台任务 |
| `scope` | global/search_shared/manual |
| `mode` | hot/search/manual/publish |
| `attempt` | 失败重试次数 |
| `query_fingerprint` | 搜索模式查询指纹 |
| `dataset_hash` | 冻结数据集哈希 |
| `config_hash` | 聚合配置哈希 |
| `versions` | 算法、BGE 和预处理版本 |
| `statistics` | 输入、簇、动作和耗时统计 |
| `status` | pending/running/success/failed |
| `warnings` | 降级和质量警告 |
| `error_code/error_message` | 最终错误摘要 |
| `cache_expires_at` | 搜索缓存过期时间 |
| 时间字段 | created/started/completed |

运行复用键至少包含：

```text
scope
analysis_run_id
hotspot_run_id
dataset_hash
config_hash
algorithm_version
BGE model/preprocess version
```

成功运行不可覆盖；失败后创建新的 `attempt`。

### 7.2 `aggregation_cluster`

保存本次运行形成的事件簇：

| 字段 | 说明 |
|------|------|
| `aggregation_run_id` | 所属运行 |
| `cluster_index` | 本运行内稳定编号 |
| `resolved_event_id` | global 模式对应正式 Event，搜索模式为空 |
| `title` | 运行级事件标题 |
| `topic_category/topic_name` | 主题证据 |
| `keywords/entities` | 聚合关键词和实体 |
| `first_publish_time` | 最早有效发布时间 |
| `last_activity_time` | 最近活动时间 |
| `representative_article_id` | 代表文章 |
| `member_count/platform_count` | 运行级统计 |
| `confidence` | 簇总体置信度 |

唯一约束：

```text
(aggregation_run_id, cluster_index)
```

### 7.3 `aggregation_assignment`

保存每篇文章的判定依据：

| 字段 | 说明 |
|------|------|
| `aggregation_run_id` | 所属运行 |
| `aggregation_cluster_id` | 所属运行级事件簇 |
| `article_id/content_identity` | 冻结文章身份 |
| `resolved_event_id` | 最终正式事件，可空 |
| `membership_action` | create/attach/unchanged/moved/deferred |
| `candidate_event_id` | 最优候选正式事件 |
| `tfidf_similarity` | 任务内相似度 |
| `bge_similarity` | 跨运行语义相似度 |
| `keyword_similarity` | 关键词相似度 |
| `entity_similarity` | 实体相似度 |
| `time_compatibility` | 时间兼容度 |
| `final_score` | 综合分 |
| `decision_reason` | 结构化判定原因 |
| `is_representative` | 是否参与中心计算 |

唯一约束：

```text
(aggregation_run_id, article_id)
```

### 7.4 `event_article_membership`

保存正式事件成员历史：

| 字段 | 说明 |
|------|------|
| `event_id/article_id` | 正式事件和文章 |
| `source_aggregation_run_id` | 产生该成员关系的运行 |
| `confidence` | 归属置信度 |
| `decision_method` | 判定方法 |
| `is_active` | 是否为当前有效归属 |
| `active_article_id` | 活跃时等于 article_id，失效后置空，用于唯一约束 |
| `valid_from/valid_to` | 有效区间 |

`active_article_id` 建立唯一约束。MySQL 和 SQLite 都允许唯一列存在多个 `NULL`，因此历史失效记录可以并存，而同一篇文章不能同时出现两条活跃成员记录。文章被纠正时，旧记录的 `is_active` 设为 false、`active_article_id` 置空并填写 `valid_to`，不物理删除。

现有 `Article.event_id` 继续保留，作为当前有效归属的快速查询字段；它与成员历史在同一事务中同步更新。

### 7.5 `event_representation`

保存正式事件的版本化跨运行表示：

| 字段 | 说明 |
|------|------|
| `event_id` | 正式事件 |
| `model_name/model_version` | BGE 标识 |
| `preprocess_version` | 向量文本预处理版本 |
| `dimension/vector` | 归一化事件中心向量 |
| `keywords/entities` | 事件关键词和实体摘要 |
| `member_count` | 计算时有效代表成员数 |
| `source_aggregation_run_id` | 最近更新来源 |
| `updated_at` | 更新时间 |

不同 BGE 模型版本不得混用。模型升级后创建或重算新版本表示。

`Event.center_vector` 不再解释为可跨运行比较的 TF-IDF 中心，第一版保留字段以兼容旧结构，但新逻辑读取 `event_representation`。

### 7.6 `event_merge_record`

保存正式事件合并候选和确认结果：

| 字段 | 说明 |
|------|------|
| `source_event_id/target_event_id` | 来源和目标事件 |
| `aggregation_run_id` | 发现候选的运行 |
| `similarity_evidence` | 分项分数和冲突检查 |
| `reason` | 候选原因 |
| `status` | pending/confirmed/rejected |
| `reviewed_by/reviewed_at` | 管理员审核 |

合并后来源事件保留为别名，不物理删除。历史热度快照仍保留原始事件标识。

## 8. 聚合输入与确定性顺序

### 8.1 输入条件

参与聚类中心计算的文章必须满足：

```text
属于当前 analysis_run
clean_status = success
AnalysisRunArticle.is_representative = true
content_identity 与运行快照一致
```

低质量但仍可分析的文章可以参与，具体权重和警告来自上游；完全无有效文本和语义证据的文章进入 `deferred`。

### 8.2 固定处理顺序

Single-Pass 对文章顺序敏感，因此使用：

```text
publish_time 升序
→ 缺失时 first_crawled_at 升序
→ 时间相同时 article.id 升序
```

相同数据集、配置、算法版本和向量版本必须产生相同结果。

---

## 9. 分层候选检索

### 9.1 运行内候选

当前运行形成的事件簇优先使用：

- LDA 主主题和主题关键词。
- 文章关键词重合。
- `DocumentFeatures.topics/mentions`。
- 发布时间区间。
- 运行内 TF-IDF 中心。

LDA 主题编号不是硬门槛。同一事件可能因窗口变化落入不同主题，因此主题只用于排序和缩小候选。

### 9.2 正式历史事件候选

跨运行匹配先按数据库字段缩小范围：

```text
主题分类或关键词相近
＋实体存在交集或不存在明确冲突
＋事件状态可匹配
＋时间处于合理延续范围，或存在重新升温证据
```

然后加载少量 `event_representation` 在 Python 中计算 BGE 余弦相似度。

第一版数据量较小，不引入向量数据库；候选量增长后再考虑向量索引。

## 10. 混合相似度与阈值

### 10.1 分项分数

聚合判定使用：

```text
BGE 相似度
TF-IDF 相似度
关键词相似度
实体相似度
时间兼容度
主题兼容证据
```

初始综合权重：

```text
BGE               0.45
TF-IDF             0.25
实体证据           0.20
时间兼容度         0.10
```

关键词和主题主要参与候选筛选、冲突检查和判定原因；第一版可在配置中将关键词分数并入实体/文本证据，不单独扩大总权重。

某项不可用时不记为零，而是对剩余有效权重重新归一化，并写入警告。

### 10.2 初始判定区间

```text
final_score >= 0.72 且无硬冲突
→ attach

final_score < 0.58
→ create

0.58 <= final_score < 0.72
→ 边界复核
→ 仍不足时保守创建新运行内事件簇
```

旧需求中的单一 `0.5` 阈值不直接作为最终标准。所有阈值进入配置和配置哈希，通过人工标注集继续调整。

### 10.3 硬冲突

以下可靠证据可以否决高语义相似度：

- 不同地点发生的独立事故。
- 同名人物但身份或机构明确不同。
- 相同天气、节日或政策主题下，时间和主体均不同。
- 两篇文章明确描述不同案件、不同学校、不同企业或不同航班等具体对象。
- 时间跨度过大，且没有通报、回应、调查、处罚、判决或再次传播证据。

字段缺失不算冲突，只有明确不一致时才应用硬冲突。

### 10.4 已有归属的移动规则

文章已有正式事件归属时：

```text
仍匹配当前事件
→ unchanged

新候选只略高
→ 保留原事件，记录边界警告

新候选明显更可靠
→ moved
```

移动需要满足：

```text
新事件分数 >= attach_threshold
且新事件分数至少比当前事件高 0.15
且不存在关键实体或时间冲突
```

该分差也进入配置，避免文章在事件间来回跳动。

## 11. 事件中心与运行内合并

### 11.1 运行内中心

事件簇维护：

```text
TF-IDF 中心 = 代表文章 TF-IDF 向量均值
BGE 中心 = 代表文章 BGE 向量均值后 L2 归一化
关键词 = 成员关键词汇总
实体 = 成员实体频次和置信度汇总
时间范围 = 最早发布时间至最后活动时间
```

### 11.2 正式事件表示更新

正式事件中心从当前有效代表成员重新计算，不使用 `count += 1` 式累计。这样任务重试不会重复增加权重，文章移动后旧成员也不会继续影响中心。

### 11.3 运行内合并复核

Single-Pass 结束后，只对本次运行形成的相近簇执行复核：

```text
主题或关键词相近
＋BGE 高相似
＋实体和时间不冲突
→ 合并运行内事件簇
```

第一版不自动合并两个已经存在的正式事件。发现重复时写入 `event_merge_record(status=pending)`。

---

## 12. 正式事件字段更新

创建正式事件时：

- 标题优先使用代表文章标题。
- 主题分类和主题名称读取热点主题证据。
- 最早和最近时间从有效成员计算。
- `independent_report_count` 只统计代表文章。
- `platform_count` 按有效成员平台去重。
- 摘要、情感和生命周期由后续模块负责。

向已有事件加入文章时不应反复覆盖稳定标题。只有原标题明显为空、质量不足或管理员确认时才更新。

## 13. 24 小时共享搜索缓存

### 13.1 缓存指纹

缓存命中要求：

```text
标准化 query_fingerprint 一致
平台集合一致
搜索范围和语言一致
算法及配置版本一致
成功运行完成时间不足 24 小时
```

查询标准化包括 Unicode NFKC、空白、大小写、`#话题#` 包裹符号和平台集合排序。

### 13.2 缓存行为

```text
不足 24 小时
→ cached=true，直接返回

达到或超过 24 小时
→ stale=true，先返回最近成功结果
→ 后台启动增量刷新
```

同一查询已有刷新任务时复用原任务。刷新只处理新增文章、内容变化文章和版本变化后必须重算的结果；未变化文章继续复用预处理、关键词和 BGE 缓存。

刷新失败时旧结果仍可展示，但必须返回：

```text
last_success_at
refresh_failed
refresh_error
```

### 13.3 用户隔离

共享缓存不记录用户私有信息。用户搜索历史单独保存：

```text
user_id
query
query_fingerprint
aggregation_run_id
searched_at
```

用户可以复用相同公共数据结果，但不能查看其他用户的搜索历史。

## 14. 历史事件与相似事件检索

### 14.1 历史事件搜索

正式历史搜索只查询 `Event`，不混入搜索临时事件簇。

流程：

```text
标题、主题名称和关键词初筛
→ 读取兼容版本 event_representation
→ BGE 语义重排
→ 返回正式历史事件
```

现有 `event_service.search_events()` 的 `LIKE` 查询可以保留为基础降级路径。

### 14.2 相似历史事件

给定正式事件：

```text
读取当前 event_representation
→ 排除自身、已合并别名和无效事件
→ 筛选正式历史候选
→ 计算 related_score
→ 返回 Top 5
```

初始检索权重：

```text
BGE 语义相似度     0.65
关键词相似度       0.15
实体相似度         0.15
主题兼容度         0.05
```

时间不作为主要降分项，因为用户需要检索较早的相似事件；时间用于 `before/after` 筛选。

### 14.3 只读保证

相似检索输出 `related_event`，不得：

- 修改 `Article.event_id`。
- 创建或关闭成员关系。
- 合并两个事件。
- 更新事件热度。

第一版实时计算 Top 5，不新增永久相似度表。数据量增长后可增加带算法版本的相似度缓存。

## 16. 幂等、并发与事务

### 16.1 运行幂等

相同复用键已有成功运行时直接返回；已有运行中任务时返回原任务；失败后创建新 `attempt`。

唯一约束至少包括：

```text
(aggregation_run_id, article_id)
(aggregation_run_id, cluster_index)
```

### 16.2 全局写入并发

同一个全局数据窗口只允许一个聚合任务进入正式事件写入阶段。

更新已有事件时：

```text
锁定 Event
→ 重新读取当前成员和事件表示
→ 验证候选判断仍然有效
→ 写成员历史
→ 重算事件表示
→ 同步 Article.event_id
→ 提交事务
```

MySQL 使用行级锁；SQLite 测试环境使用进程内锁覆盖关键写入边界。

### 16.3 快照和租约

算法可以在事务外计算，正式写入前必须再次验证：

- `analysis_run.dataset_hash`。
- 每篇文章的 `content_identity`。
- 任务租约仍有效。
- 聚合配置和 BGE 版本未变化。

验证失败整次回滚，不写部分正式成员关系。

## 17. 错误处理与降级

| 错误或缺失 | 处理 |
|------------|------|
| `BGE_UNAVAILABLE` | TF-IDF、关键词、实体和时间降级，重新归一化权重 |
| TF-IDF 空词表 | BGE、关键词和实体继续 |
| LDA 结果缺失 | 不阻止搜索/手动聚合 |
| 发布时间缺失 | 使用 `first_crawled_at`，降低时间置信度 |
| 无历史候选 | 创建新运行内事件簇 |
| 文本和语义证据均不足 | `deferred`，不强行创建正式事件 |
| `DATASET_CHANGED` | 整次运行失败并回滚 |
| 任务租约失效 | 禁止提交 |
| 正式写入冲突 | 回滚后重新加载，有限重试 |
| 个别文章失败 | 记录文章级警告，其他文章继续 |
| 失败比例超过配置 | 整次运行失败 |

所有降级必须保存到 `aggregation_run.warnings` 和对应 assignment 的判定原因中。

## 18. 测试方案

### 18.1 纯算法测试

1. 相同具体事件的跨平台改写进入同一簇。
2. 同主题、不同地点或主体的事件被拆开。
3. 跨运行 TF-IDF 比较被拒绝。
4. 相同版本 BGE 可以跨运行比较。
5. 不同 BGE 版本不能混用。
6. 缺失分项时权重正确归一化。
7. 实体硬冲突覆盖高 BGE 分数。
8. 固定输入排序后结果可复现。
9. attach/create/边界/deferred 规则正确。
10. 运行内合并不会自动合并正式历史事件。

### 18.2 典型语义案例

至少覆盖：

```text
新闻稿与社交媒体转述
案件首次通报、回应、判决和处罚
相同人物参与的不同事件
同名人物或机构
不同城市发生的相似事故
同一地区不同年份的自然灾害
相似标题但事实主体不同
标题不同但事实相同
事件沉寂后再次传播
官方回应导致再次升温
```

### 18.3 服务和持久化测试

- 成功运行复用和失败 `attempt`。
- 快照变化完整回滚。
- 每篇文章每次运行只有一个 assignment。
- 每篇文章只有一个有效正式归属。
- `Article.event_id` 与成员历史一致。
- create/attach/unchanged/moved/deferred 持久化。
- 重试不重复增加成员数。
- 正式事件表示从有效成员重算。
- 重复文章继承代表文章事件但不进入中心。
- 搜索事件不写正式 `Event`。
- 发布搜索事件创建或关联正式事件。
- 重复发布保持幂等。
- 正式事件疑似重复只生成合并候选。

### 18.4 搜索缓存测试

```text
23:59:59 → 新鲜缓存
24:00:00 → 过期
超过 24 小时 → 返回旧结果并启动刷新
```

验证平台集合、算法版本、并发刷新、刷新失败回退和增量复用。

### 18.5 历史与相似事件测试

- 同一案件后续识别为 `same_event`。
- 不同地区同类事件识别为 `related_event`。
- 相似查询不修改成员关系。
- 结果排除自身、临时事件和已合并别名。
- BGE 不可用时关键词、实体和主题降级。
- Top 5 顺序稳定并返回匹配原因。
- 正式成员变化后使用最新事件表示。

### 18.6 迁移测试

验证新增表、外键、索引和唯一约束；迁移必须可重复执行，兼容 SQLite 测试和 MySQL 8，不删除或重置现有数据。

### 18.7 端到端测试

```text
爬虫
→ 预处理
→ 内容分析
→ 热点主题发现
→ 事件聚合
→ 事件热度快照
→ 公共热点榜单
```

以及：

```text
首次搜索
→ 保存共享搜索事件
→ 24 小时内复用
→ 过期后增量刷新
→ 管理员发布为正式事件
```

### 18.8 人工标注集

沿用：

```text
same_event
different_event_same_topic
unrelated
```

第一版至少 50～100 对，后续扩展到 100～200 对。初始目标：

```text
same_event precision >= 0.90
same_event recall    >= 0.80
macro F1             >= 0.80
明确地点/主体冲突误合并数 = 0
```

样本不足时只报告初始调参结果，不宣称普遍准确率。

## 19. 第一版验收标准

1. 热点运行不再长期停留在 `EVENT_AGGREGATION_PENDING`。
2. 同一具体事件的文章能够跨平台聚合。
3. 同主题下不同具体事件能够拆开。
4. 同一正式事件跨多次全局运行保持稳定 `event.id`。
5. 不直接比较跨运行 TF-IDF。
6. BGE 不可用时存在明确降级。
7. 聚合运行、配置、判定分数和原因可追溯。
8. 搜索结果在完成后 24 小时内全用户共享复用。
9. 过期搜索先返回旧结果，再后台增量刷新。
10. 搜索事件不污染公共事件看板。
11. 管理员可以将搜索事件受控发布到全局。
12. 重试和并发不会产生重复成员或重复事件。
13. 正式事件合并不会静默破坏历史身份和热度快照。
14. 历史搜索支持关键词初筛和 BGE 重排。
15. 事件详情能够返回最多 5 个相似正式历史事件。
16. `related_event` 检索不会修改文章归属。
17. 完整闭环具备自动化测试。
18. 当前内容分析、热点发现及其他既有测试继续通过。

## 20. 实施顺序

建议按以下顺序实现，避免在模型和算法尚未稳定时提前修改任务闭环：

1. 聚合配置、纯相似度算法和 Single-Pass 测试。
2. 数据模型、SQL 迁移和迁移测试。
3. 聚合运行、搜索隔离结果和正式成员持久化。
4. 正式事件表示和跨运行历史匹配。
5. 历史搜索和相似事件接口。
6. 热点主题—聚合—热度的后台闭环。
7. 24 小时共享搜索缓存和增量刷新。
8. 管理员发布及正式事件合并候选。
9. 专项、相关回归和完整后端测试。
