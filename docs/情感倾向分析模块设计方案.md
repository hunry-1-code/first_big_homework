# 情感倾向分析模块设计方案

## 1. 文档目的

本文档定义“网络舆情事件智能分析系统”第一版情感倾向分析模块。模块承接已经完成的预处理、内容分析和事件聚合结果，对事件成员文章进行版本化的逐篇混合情感分析，并生成事件总体、每日趋势和分平台内容立场分布。

本文档以当前代码和最新模块设计为基础。与 `项目需求规格说明书.md`、早期 README 或技术选型文档冲突时，以实际实现和本文为准。

## 2. 范围与边界

### 2.1 本模块负责

```text
事件聚合完成
→ 冻结事件或搜索事件簇成员及文章内容版本
→ 创建 sentiment_run
→ 对去重代表文章逐篇执行 LLM 情感分析
→ LLM 失败时逐篇降级 SnowNLP
→ 重复文章继承代表文章结果
→ 保存版本化文章结果
→ 生成事件总体、每日趋势和分平台情感快照
→ 更新 Article/Event 当前摘要字段
```

具体职责包括：

1. 支持 `hot`、`search`、`manual` 和 `publish` 模式。
2. 使用事件聚合后的稳定成员关系或隔离搜索事件簇作为分析范围。
3. 判断正面、负面和中性三类内容立场。
4. 对灾害、事故类文本执行类别感知判定。
5. 保存极性分数、置信度、判定维度、评价目标、理由、模型和 Prompt 版本。
6. 复用相同文章内容版本和相同算法版本的成功结果。
7. 生成原始数量与加权比例两套事件统计口径。
8. 生成事件总体每日情感趋势。
9. 生成各平台当前情感分布和样本充分性提示。
10. 保证搜索和未发布手动结果不修改正式 `Event`。

### 2.2 本模块不负责

- 评论、回复、弹幕的独立采集与情感分析。
- 情绪细分类，如愤怒、恐惧、悲伤和喜悦。
- 事件热度计算、趋势预测、风险评估和报告生成。
- 使用情感结果修改文章—事件归属。
- 将分平台结果解释为平台全部用户的真实态度。

第一版“分平台情感”表示各平台已采集文章、帖子和视频文本呈现出的内容立场，不表示平台全体用户意见。

## 3. 已确认的核心决策

| 决策 | 第一版方案 |
|------|------------|
| 总体方案 | 版本化的逐篇混合分析 |
| 主分析方法 | DeepSeek/OpenAI 兼容 LLM |
| 降级方法 | SnowNLP |
| 基本标签 | 正面、负面、中性 |
| 判定语义 | 面向事件核心对象或处置结果的内容立场 |
| 灾害事故 | 统一三分类，但使用类别感知规则并记录判定维度 |
| 独立调用对象 | 去重代表文章 |
| 重复文章 | 继承代表文章结果，以较低权重参与汇总 |
| 保存方式 | 运行、文章结果和事件快照全部版本化 |
| 当前字段 | `Article`、`Event` 仅保存最新摘要 |
| 事件口径 | 同时返回原始数量和加权比例 |
| 时间趋势 | 事件总体按日趋势 |
| 平台比较 | 各平台当前分布；第一版不做分平台每日趋势 |
| 搜索边界 | 搜索事件结果隔离，不更新正式 `Event` |

## 4. 有效文章定义

文章进入情感分析范围必须满足：

1. `clean_status = success`。
2. 标题或 `clean_content` 至少一个包含有效文本。
3. `nlp_weight > 0`。
4. 不是解析失败页、登录页、验证码页或其他无效页面。
5. 默认排除已确认的广告和垃圾内容。
6. 文章内容身份与本次运行冻结的 `content_identity` 一致。
7. 正式事件分析时，文章具有当前有效的 `event_article_membership`。
8. 搜索或手动分析时，文章属于对应 `aggregation_cluster`。
9. 热榜种子不作为正式报道进入情感统计。

有效文章分为：

```text
代表文章
→ 独立执行 LLM/SnowNLP 分析

重复文章
→ 不重复调用模型
→ 继承 duplicate_of_id 对应代表文章的结果
→ 保存独立继承记录并参与降权汇总
```

缺少可靠发布时间不影响文章级判断。进入每日趋势时优先使用 `publish_time`；缺失时可以使用 `first_crawled_at` 作为观测时间，但必须记录时间降级，不能显示为真实发布时间。

## 5. 情感语义

### 5.1 基本标签

```text
positive  正面
negative  负面
neutral   中性
```

文章结果使用稳定英文枚举保存，API 可以同时返回中文展示名称。

### 5.2 判定目标

优先判断文章对事件核心对象、责任主体、处置行为或结果的态度：

- 肯定、支持、赞扬、风险解除或积极改善：正面。
- 质疑、批评、谴责、损害扩大或明显担忧：负面。
- 客观通报、无明显评价或正负证据平衡：中性。

不能只按正负词数量判断，也不能因为文章描述灾害、伤亡等事实就机械套用普通新闻语气规则。

### 5.3 灾害和事故类别感知

灾害、事故类文章仍输出统一三分类，同时记录：

```text
dimension = stance | impact | factual
target = 本次判断针对的对象或影响
```

判定顺序：

1. 存在对机构、人物、责任或处置行为的明确评价时，使用 `stance`。
2. 没有明确评价、主要描述伤亡、损失、风险、救援或恢复时，使用 `impact`。
3. 只陈述时间、地点和普通进展，且缺少明显价值倾向时，使用 `factual`。

示例：

```json
{
  "label": "negative",
  "score": -0.82,
  "confidence": 0.91,
  "dimension": "impact",
  "target": "暴雨造成的人员与财产损失",
  "reason": "报道重点描述伤亡、损失和风险扩大"
}
```

## 6. 总体架构与数据流

### 6.1 组件边界

```text
backend/app/analysis/
├── sentiment_config.py       # 参数、版本和配置哈希
├── sentiment_analyzer.py     # 纯分析、LLM 解析和 SnowNLP 降级
└── sentiment_aggregator.py   # 权重、总体、每日和平台汇总

backend/app/services/
└── sentiment_analysis_service.py
    # 运行创建、快照验证、缓存复用、持久化和事务

backend/app/api/
└── sentiment.py              # 运行与结果查询接口
```

纯算法组件不依赖 Flask 或 SQLAlchemy；数据库、权限和任务租约由服务层处理。

### 6.2 全局热点流程

```text
爬虫
→ 预处理
→ 内容分析
→ 热点主题发现
→ 事件聚合
→ 热度快照
→ 情感分析
→ Event 当前情感摘要
→ 趋势预测、风险评估和报告生成
```

情感分析不参与事件聚合和热度计算，避免形成循环依赖。

### 6.3 搜索流程

```text
用户搜索
→ 共享搜索事件聚合
→ 搜索范围 sentiment_run
→ 搜索事件簇情感快照
→ 返回总体、每日和分平台内容立场
```

搜索情感结果可以全用户共享复用，但用户搜索历史和权限继续隔离。搜索运行不修改正式 `Event`。

### 6.4 手动与发布流程

手动运行默认隔离。管理员发布搜索或手动事件后：

- 相同文章内容版本、模型版本和 Prompt 版本的成功文章结果可以复用。
- 正式事件需要重新按当前有效成员生成事件级快照。
- 不能直接复制隔离事件簇的比例作为正式事件当前摘要。

## 7. 版本化数据模型

### 7.1 `sentiment_run`

记录一次可复现情感分析运行：

| 字段 | 说明 |
|------|------|
| `id` | 运行 ID |
| `aggregation_run_id` | 上游事件聚合运行 |
| `source_task_id` | 后台任务，可空 |
| `user_id` | 搜索或手动运行用户，可空 |
| `scope` | global/search_shared/manual |
| `mode` | hot/search/manual/publish |
| `attempt` | 失败重试次数 |
| `dataset_hash` | 冻结成员和内容版本哈希 |
| `config_hash` | 情感配置哈希 |
| `versions` | 算法、LLM、Prompt、SnowNLP 版本 |
| `statistics` | 输入、复用、LLM、降级、继承和失败数量 |
| `status` | pending/running/success/failed |
| `warnings` | 运行级警告 |
| `error_code/error_message` | 错误摘要 |
| 时间字段 | created/started/completed |

成功运行不可覆盖；失败后创建新的 `attempt`。

### 7.2 `article_sentiment_result`

保存每篇有效文章在某次运行中的结果：

| 字段 | 说明 |
|------|------|
| `sentiment_run_id` | 所属运行 |
| `article_id` | 文章 |
| `content_identity` | 冻结内容身份 |
| `aggregation_cluster_id` | 本次运行级事件簇 |
| `event_id` | 正式事件，可空 |
| `label` | positive/negative/neutral |
| `score` | -1.0～1.0 的极性分数 |
| `confidence` | 0.0～1.0 的判断置信度 |
| `dimension` | stance/impact/factual |
| `target` | 判断目标 |
| `reason` | 简短依据 |
| `method` | llm/snownlp/inherited |
| `model_name/model_version` | 实际模型版本 |
| `prompt_version` | Prompt 版本 |
| `raw_response` | 可选的受控结构化原始响应 |
| `inherited_from_result_id` | 重复文章继承来源，可空 |
| `weight_details` | 汇总使用的权重明细 |
| `warnings` | 降级和质量警告 |

唯一约束：

```text
(sentiment_run_id, article_id)
```

跨运行复用键：

```text
article_id
+ content_identity
+ algorithm_version
+ model_name/model_version
+ prompt_version
+ sentiment_preprocess_version
```

### 7.3 `event_sentiment_snapshot`

保存某次运行中的事件或隔离事件簇汇总：

| 字段 | 说明 |
|------|------|
| `sentiment_run_id` | 所属运行 |
| `event_id` | 正式事件，可空 |
| `aggregation_cluster_id` | 运行级事件簇，可空 |
| `calculated_at` | 计算时间 |
| `article_count` | 有效文章数 |
| `representative_count` | 代表文章数 |
| `raw_counts` | 三类原始数量 |
| `weighted_ratios` | 三类加权比例 |
| `dominant_label` | 主要倾向 |
| `average_score` | 加权平均极性分数 |
| `daily_trend` | 事件总体每日趋势 |
| `platform_distribution` | 各平台当前分布 |
| `time_confidence` | 时间数据可信度 |
| `calculation_details` | 权重、缺失项和降级信息 |
| `algorithm_version` | 汇总算法版本 |

正式事件保存 `event_id`，搜索事件保存 `aggregation_cluster_id`。两者必须至少存在一个。

### 7.4 当前摘要字段

现有字段继续保留：

```text

Article.sentiment_score
Article.sentiment_reason
Article.sentiment_method
```

这些字段只用于快速读取，历史和趋势以版本化结果表为准。

## 8. LLM 分析

### 8.1 输入文本

每篇代表文章输入：

```text
事件主题分类和主题名称
+ 事件或事件簇标题
+ 文章标题
+ 正文前 500 个有效字符
```

正文截断在清理空白之后执行。不得发送 `raw_html`、无关 `raw_json` 或其他文章全文。

### 8.2 结构化输出

LLM 必须只返回 JSON：

```json
{
  "label": "positive | negative | neutral",
  "score": -0.75,
  "confidence": 0.88,
  "dimension": "stance | impact | factual",
  "target": "相关机构的应急处置",
  "reason": "文章质疑响应迟缓并强调损失扩大"
}
```

校验规则：

- `label`、`dimension` 必须属于固定枚举。
- `score` 在 -1～1。
- `confidence` 在 0～1。
- 正面分数不得明显小于零，负面分数不得明显大于零。
- 中性分数应处于可配置中性区间。
- `target` 和 `reason` 限制长度，去除多余解释和 Markdown。
- 输出非法时先进行一次确定性清理；仍非法则视为本次 LLM 失败。

### 8.3 调用控制

- 仅对未命中版本缓存的代表文章调用 LLM。
- 默认有限并发，第一版建议最大 2～4 个并发请求。
- 429、5xx、超时和临时网络错误最多重试 3 次。
- 401、403、配置缺失和明确参数错误立即降级，不持续重试。
- 单篇失败只影响该篇，不中止整次运行。
- 运行统计必须区分 LLM 成功、缓存复用、SnowNLP 降级、重复继承和最终失败。

## 9. SnowNLP 降级

当前依赖中尚未包含 SnowNLP，实现时新增受版本约束的依赖。

SnowNLP 返回正向概率 `p` 后转换：

```text
score = 2 × p - 1

p >= 0.60 → positive
p <= 0.40 → negative
其他       → neutral
```

阈值可配置并进入配置哈希。SnowNLP 不具备可靠的事件目标和灾害语义理解能力，因此降级结果必须：

- `method = snownlp`。
- `confidence` 使用极性距离生成，并设置上限，避免伪装成高置信 LLM 结果。
- `dimension` 默认为 `factual`；能够从事件类别确定灾害事故时可标记 `impact`，但不能伪造明确评价目标。
- `warnings` 包含 LLM 失败原因和 `SNOWNLP_FALLBACK`。

SnowNLP 自身不可用或推理失败时，该篇记录为失败；当成功或可降级文章比例低于配置门槛时，整次运行失败且不更新当前事件摘要。

## 10. 重复文章继承

重复文章通过 `duplicate_of_id` 查找代表文章结果：

1. 代表文章必须属于当前冻结范围，或存在兼容版本的可复用成功结果。
2. 继承结果保留相同标签、分数、维度、目标和理由。
3. `method = inherited`。
4. 保存 `inherited_from_result_id`。
5. 重复文章不增加模型调用次数，不参与代表文章置信度统计。
6. 汇总时乘以 `duplicate_weight`，避免大量转载支配分布。

找不到代表文章结果时，重复文章不独立调用 LLM，记录 `DUPLICATE_SENTIMENT_SOURCE_MISSING` 并从本次汇总中跳过。

## 11. 事件级汇总

### 11.1 两套口径

每个事件或事件簇同时保存：

```text
raw_counts
→ 各标签有效文章数量，用于解释

weighted_ratios
→ 各标签加权比例，用于主要图表和总体倾向
```

### 11.2 文章权重

初始权重：

```text
quality_weight = clamp(nlp_weight, 0, 1)
spam_weight    = clamp(article.spam_weight 或 1, 0, 1)

spread_factor =
  heat_contribution 可用时：
    1 + log1p(max(heat_contribution, 0))
        / log1p(事件内最大 heat_contribution)
  否则：1

duplicate_factor =
  代表文章：1
  重复文章：clamp(duplicate_weight 或默认降权值, 0, 1)

article_weight =
  quality_weight × spam_weight × spread_factor × duplicate_factor
```

传播因子范围为 1～2，避免单篇爆款完全支配事件结果。缺失互动量或热度贡献时退化为质量权重，不把未知值当成零。

### 11.3 比例与总体分数

```text
positive_ratio = Σ positive文章权重 / Σ全部文章权重
negative_ratio = Σ negative文章权重 / Σ全部文章权重
neutral_ratio  = Σ neutral文章权重  / Σ全部文章权重

average_score = Σ(score × article_weight) / Σarticle_weight
```

主要倾向使用最大加权比例；最高比例差值过小或有效权重不足时可以标记为中性或 `mixed` 展示状态，但持久化基础标签仍保持三分类比例。

## 12. 每日趋势

事件总体趋势按自然日分组，输出：

```text
date
raw_counts
weighted_ratios
average_score
article_count
representative_count
time_source_counts
```

时间来源优先级：

```text
publish_time
→ 缺失时 first_crawled_at
→ 两者均缺失时不进入每日趋势，但仍进入总体分布
```

趋势接口必须提示使用采集时间降级的文章数量。

## 13. 分平台内容立场

每个平台保存：

```text
platform
article_count
representative_count
raw_counts
weighted_ratios
average_score
effective_weight
sample_insufficient
warnings
```

第一版规则：

- 平台样本少于 3 篇有效文章，或少于 2 篇代表文章时，`sample_insufficient=true`。
- 样本不足仍返回结果，但前端必须显示提示。
- 不对不同平台原始互动量直接比较；文章传播因子只在当前事件范围内进行有界处理。
- 第一版不生成分平台每日趋势。

## 14. 幂等、缓存与并发

### 14.1 运行复用

运行复用键至少包含：

```text
aggregation_run_id
+ dataset_hash
+ config_hash
+ algorithm_version
+ LLM model/version
+ prompt_version
+ SnowNLP version
```

已有成功运行时直接复用；已有运行中任务时返回原运行；失败后创建新 `attempt`。

### 14.2 文章结果复用

文章结果只在内容身份和全部相关版本一致时复用。模型、Prompt、预处理或标签规则变化时重新分析，但保留旧结果。

### 14.3 并发与事务

- 同一 `sentiment_run` 只能有一个执行实例提交结果。
- LLM 调用可以在数据库事务外执行。
- 最终写入前重新验证聚合运行、成员、内容身份和任务租约。
- 快照变化时整次运行失败，不更新 `Article`、`Event` 当前摘要。
- 文章结果和事件快照在同一最终事务中提交。

## 15. 错误处理与降级

| 错误或警告 | 处理 |
|------------|------|
| `NO_VALID_DOCUMENT` | 运行失败 |
| `DATASET_CHANGED` | 整次回滚 |
| `LLM_UNAVAILABLE` | 单篇降级 SnowNLP |
| `LLM_OUTPUT_INVALID` | 清理后仍无效则降级 SnowNLP |
| `SNOWNLP_FALLBACK` | 保存降级结果和原因 |
| `SNOWNLP_UNAVAILABLE` | 单篇失败 |
| `DUPLICATE_SENTIMENT_SOURCE_MISSING` | 跳过该重复文章 |
| `TIME_SOURCE_DEGRADED` | 使用采集时间生成趋势并提示 |
| `PLATFORM_SAMPLE_INSUFFICIENT` | 返回结果但提示样本不足 |
| `STALE_TASK_LEASE` | 禁止提交 |
| 最终成功比例过低 | 整次运行失败，不更新当前摘要 |

LLM 或 SnowNLP 的单篇失败不得清空已有成功事件快照。

## 19. 测试方案

### 19.1 纯算法测试

1. LLM 合法 JSON 正确解析。
2. Markdown 包裹和轻微格式问题可以确定性清理。
3. 非法枚举、越界分数和标签—分数冲突被拒绝。
4. 灾害损失、救援成效、责任质疑和客观通报按类别感知规则处理。
5. SnowNLP 概率正确映射为三分类和极性分数。
6. 缺失传播量时权重正确降级。
7. 重复文章应用重复降权。
8. 原始数量、加权比例和平均分计算正确。
9. 每日趋势使用发布时间优先和采集时间降级。
10. 分平台结果和样本不足标记正确。

### 19.2 服务和持久化测试

1. 只有成功聚合运行可以创建情感运行。
2. 相同数据和版本复用成功运行。
3. 失败运行创建新 `attempt`。
4. 相同文章内容和版本复用历史结果。
5. Prompt 或模型版本变化时重新分析。
6. 代表文章独立分析，重复文章继承。
7. LLM 失败时单篇降级 SnowNLP。
8. 搜索运行不更新正式 `Event`。
9. 全局运行更新 `Article` 和 `Event` 当前摘要。
10. 内容版本变化时整次回滚。
11. 任务租约失效时禁止提交。
12. 运行重试不产生重复文章结果和快照。

### 19.3 API 与权限测试

1. 管理员可以创建和查询情感运行。
2. 普通用户可以读取正式事件汇总。
3. 普通用户可以读取共享搜索事件簇汇总。
4. 私有手动运行不能被其他普通用户读取。
5. 详细文章判定和原始响应只向管理员开放。
6. 原有事件情感接口保持兼容。

### 19.4 端到端测试

```text
爬虫
→ 预处理
→ 内容分析
→ 热点发现
→ 事件聚合
→ 热度快照
→ 情感分析

```


自动化测试不得请求真实 LLM API，使用可注入假客户端和确定性 SnowNLP 替身。

## 20. 第一版验收标准

1. 能对事件内有效代表文章执行逐篇 LLM 情感分析。
2. 输出正面、负面、中性标签及 -1～1 极性分数。
3. 输出置信度、判定维度、目标和简短理由。
4. 灾害和事故类文本执行类别感知判定。
5. LLM 不可用或输出无效时自动降级 SnowNLP。
6. 重复文章继承代表文章结果，不重复调用模型。
7. 运行、文章结果和事件快照均可按内容和算法版本追溯。
8. 事件接口同时返回原始数量和加权比例。
9. 能返回事件总体每日情感趋势。
10. 能返回各平台当前内容立场分布和样本不足提示。
11. 搜索和未发布手动结果不修改正式 `Event`。
12. 相同文章内容和版本可以复用，模型或 Prompt 变化会重新分析。
13. 任务重试和并发不会产生重复结果。
14. 自动化测试覆盖正常、缓存、继承、降级、隔离、版本变化和权限场景。
15. 现有爬虫、预处理、内容分析、热点发现和事件聚合功能保持兼容。
