# 内容分析模块设计方案

## 1. 文档目的

本文档定义网络舆情事件智能分析系统第一版的内容分析模块。模块承接已经实现的数据清洗与预处理结果，完成特征表示和关键词提取，为热点事件发现、事件聚合、历史事件检索和可视化提供可复现的文本特征。

本文档是当前内容分析实现的设计依据。`项目需求规格说明书.md` 中与本文冲突的旧表述，应以后续同步修订后的本文方案为准。

---

## 2. 范围与边界

### 2.1 本模块负责

```text
预处理完成的文章与分词结果
→ 筛选有效代表文章
→ 构建本次 analysis_run 数据集快照
→ 生成词频矩阵和 TF-IDF 稀疏矩阵
→ 提取单篇文章关键词
→ 按下游事件映射汇总事件级关键词
→ 输出可复现的特征元数据和统计信息
```

具体职责包括：

1. 消费 `article.clean_content`、`document_features.tfidf_tokens`、实体、质量和去重结果。
2. 在本次 `analysis_run` 内统一构建词表、IDF 和 TF-IDF 矩阵。
3. 为热点事件发现模块提供词频矩阵，但不在本模块内执行 LDA。
4. 为事件聚合模块提供本次任务内可比较的 TF-IDF 向量。
5. 提取文章级最多 10 个关键词和事件级最多 20 个关键词。
6. 保存分析参数、数据集哈希、版本和降级信息，支持重新计算。

### 2.2 本模块不负责

- 正文提取、清洗、去噪、分词和去重：由 `backend/app/preprocessing` 负责。
- LDA 主题发现、主题数选择和主题命名：属于热点事件发现模块。
- Single-Pass 聚类、事件中心更新和全局事件匹配：属于事件聚合模块。
- 正面、负面、中性判断：属于情感分析模块。
- 热度、生命周期和风险等级计算：属于热点与趋势模块。
- 虚假文本检测、事件溯源和关键传播路径：属于高级功能。

正文提取和自动分词虽然属于题目“内容分析”的最低要求，但工程上已经在预处理模块完成。本模块直接复用其输出，不重复实现。

## 3. 已确认的核心决策

| 决策 | 第一版方案 |
|------|------------|
| 特征表示 | TF-IDF |
| 处理范围 | 每个 `analysis_run` 独立构建词表和 IDF |
| TF-IDF 特征数 | 最多 5000 |
| n-gram | `(1, 2)`，即单词和二词短语 |
| 词频 | `sublinear_tf=True` |
| 归一化 | L2 |
| 最低文档频次 | `min_df=1` |
| 最高文档比例 | 正常语料 `max_df=0.90`，小语料降级为 `1.0` |
| 标题/正文初始权重 | `1.0 : 1.0`，不未经验证地固定为 `2:1` |
| 标题权重调优 | 在标注集上从 `0.5/1.0/1.5/2.0/3.0` 中选择 |
| 单篇关键词 | 最多 10 个，TF-IDF 为主、TextRank 补充 |
| 事件关键词 | 最多 20 个，综合 TF-IDF、文章覆盖和平台覆盖 |
| 完整 TF-IDF 矩阵 | 仅在任务运行期间存在，不写入数据库 |
| 跨任务相似度 | 使用固定模型版本的 BGE 向量，不比较不同任务的 TF-IDF |
| 小语料 | 2～4 篇降级运行；1 篇改用 TextRank/词频/实体；0 篇失败 |

## 4. 上游输入

### 4.1 输入字段

内容分析至少读取：

```text
article.id
article.title
article.clean_content
article.clean_status
article.content_version
article.latest_snapshot_id
article.platform
article.quality_score
article.nlp_weight
article.is_advertisement
article.is_duplicate
article.duplicate_of_id
article.duplicate_group_id
document_features.tfidf_tokens
document_features.mentions
document_features.segment_version
```

### 4.2 有效文章筛选

进入 TF-IDF 的文章必须满足：

1. `clean_status = success`。
2. 存在有效标题或正文，且 `tfidf_tokens` 不为空。
3. 不是解析失败页、登录拦截页或其他无效页面。
4. 默认不使用已判定广告或垃圾页面；需要保留时必须显式配置。
5. 同一重复组只使用预处理模块选出的代表文章。

重复文章不删除，仍用于平台分布、传播量和热度等下游统计；只是不会重复进入 TF-IDF、LDA、关键词和摘要材料集合。

### 4.3 `nlp_weight` 的使用

`nlp_weight` 不直接改变单篇文章向量的方向。单纯将一个 L2 归一化向量乘以标量，不会改变余弦相似度，因此不能达到有效降权目的。

第一版在以下位置使用 `nlp_weight`：

- 汇总事件级关键词时，降低低质量文章的贡献。
- 下游更新事件中心时，作为文章向量的聚合权重。
- 选择代表文章、摘要材料和主题代表文档时参与排序。

低质量但仍可用的文章不会被无条件删除；`nlp_weight = 0` 的失败记录不进入分析。

## 5. `analysis_run` 数据集快照

### 5.1 创建时机

用户搜索模式：

```text
用户输入关键词并必须选择至少一个平台
→ 查询 24 小时缓存
→ 必要时只采集过期或缺失的平台
→ 预处理完成
→ 创建 analysis_run 数据集快照
→ 执行内容分析
```

后台热点模式：

```text
定时读取所有已启用且配置可用的平台
→ 采集热点内容
→ 预处理完成
→ 创建后台 analysis_run 或等价批次快照
→ 执行内容分析
```

### 5.2 快照要求

`analysis_run` 不复制文章正文，但必须固定本次使用的数据版本：

- 保存 `article_id`。
- 优先保存 `article_snapshot_id`；没有快照时保存 `content_version`。
- 保存排序后的平台列表和用户查询条件。
- 保存 `dataset_hash`。
- 保存成功、跳过、失败和重复代表文章数量。

建议的 `dataset_hash` 输入为：

```text
排序后的 (article_id, snapshot_id/content_version)
+ 分词版本
+ 停用词表版本
+ TF-IDF 参数版本
```

同一用户、相同查询条件、相同数据版本和相同算法版本可以复用已有分析结果；平台组合或数据版本变化时创建新的 `analysis_run`。

## 6. 特征矩阵构建

### 6.1 两种矩阵

同一批 `tfidf_tokens` 生成两种内存矩阵：

```text
词频矩阵（Bag-of-Words）
→ 提供给热点事件发现模块的 LDA

TF-IDF 稀疏矩阵
→ 文章关键词
→ 本次任务内文章相似度
→ 本次任务内事件聚合
```

LDA 应输入词频矩阵，而不是 TF-IDF 矩阵。LDA 是基于词项计数的概率生成模型，将 TF-IDF 直接称为“文档-词频矩阵”在技术上不准确。

提供给 LDA 的矩阵是标题与正文合并后的未加权整数词频矩阵；标题/正文权重只用于 TF-IDF 加权词频。热点事件发现模块可以基于后台批次规模单独调整 LDA 的 `max_df`、词表和主题模型参数，不能把内容分析的字段权重误当成 LDA 概率计数。

### 6.2 词表与字段权重

标题和正文使用同一词表。处理步骤为：

```text
在标题和正文的合并 token 上拟合 CountVectorizer 词表
→ 分别转换标题 token 和正文 token
→ 计算加权词频
→ TfidfTransformer 生成 TF-IDF
```

加权词频：

```text
weighted_tf(term)
= title_weight × title_term_count
+ body_weight × body_term_count
```

第一版默认：

```text
title_weight = 1.0
body_weight = 1.0
```

字段加权的思想参考 BM25F 和新闻流聚类研究，但没有通用证据证明标题固定为正文的两倍。后续使用人工标注的同事件/不同事件样本选择标题权重，不从其他数据集直接复制系数。

### 6.3 TF-IDF 初始参数

等价配置：

```python
CountVectorizer(
    max_features=5000,
    ngram_range=(1, 2),
    min_df=1,
    max_df=0.90,
    lowercase=False,
    token_pattern=None,
)

TfidfTransformer(
    norm="l2",
    use_idf=True,
    smooth_idf=True,
    sublinear_tf=True,
)
```

实际实现输入已经是 token 列表，应使用明确的预分词输入适配器，不能让 sklearn 再按默认英文正则切词。

`max_features=5000`、`ngram_range=(1, 2)` 和字段权重均为可配置初始值，不作为不可变常量。

用户查询词、来源话题标签和已识别实体即使因为 `max_df` 过高未进入 TF-IDF 词表，也应作为独立关键词候选保留。`max_df` 用于删除缺少区分度的向量特征，不代表这些词在事件说明中没有展示价值。

### 6.4 二词短语

二词特征用于保留：

```text
官方 通报
网络 暴力
应急 响应
城市 内涝
```

输出关键词时，如果高分二词短语已经完整覆盖低信息量单词，优先保留短语。例如“官方通报”得分较高时，通常不再单独输出“官方”；如果单词在大量文章中具有独立意义，则允许同时保留。

## 7. 小语料与异常语料降级

| 有效代表文章数 | 处理方式 |
|----------------|----------|
| `>= 5` | 正常运行，`max_df=0.90` |
| `2～4` | 运行 TF-IDF，`max_df=1.0`，记录 `SMALL_CORPUS` |
| `1` | 不生成语料级 TF-IDF；使用 TextRank、词频和实体提取关键词 |
| `0` | 任务失败，错误码 `NO_VALID_DOCUMENT` |

其他降级规则：

- 停用词过滤后词表为空：记录 `EMPTY_VOCABULARY`，改用 TextRank 和实体词。
- 单篇文章字段异常：只跳过该文章，其他文章继续。
- 标题为空：仅使用正文。
- 正文为空但属于有效热搜标题或短社交内容：允许仅使用标题，并记录 `TITLE_ONLY_DOCUMENT`。
- jieba fallback 结果可进入分析，但记录对应分词警告和版本。
- 最终结果必须保存成功、跳过、降级和失败数量，不能静默返回空数组。

“5 篇”只是工程上的稳定性提示阈值，不宣称具有严格统计分界。

## 8. 关键词提取

### 8.1 单篇文章关键词

每篇文章最多输出 10 个关键词：

```text
读取该文章的 TF-IDF 非零特征
→ 按权重取得候选词
→ 过滤停用词、纯数字、无意义单字和模板词
→ 规范化明确别名和重复表达
→ 不足 10 个时使用 TextRank 补充
→ 合并相关人物、地点、机构和话题标签
→ 输出结构化结果
```

第一版不设置缺乏标注依据的“TF-IDF 70% + TextRank 30%”固定比例。TextRank 只负责补足有效候选，不参与事件相似度和聚类。

### 8.2 事件级关键词

事件级关键词在下游事件聚合产生文章映射后，由内容分析服务使用本次 `analysis_run` 的 TF-IDF 结果汇总：

```text
事件内重复组代表文章
→ 按 nlp_weight 汇总 TF-IDF 权重
→ 统计出现文章数、文章比例和覆盖平台数
→ 通常要求至少出现在 2 篇文章中
→ 人物、地点、机构和话题标签允许例外保留
→ 输出最多 20 个关键词
```

同一词只在单篇低质量文章中异常升高时，不应进入事件级 Top 20。

### 8.3 规范化与同义表达

第一版执行：

- Unicode NFKC、全角半角、大小写和空白统一。
- 使用已有人工维护别名词典合并明确同义表达。
- 保留人物、地点、机构和话题标签类型。
- 保留规范词和必要的原始变体，便于展示和追溯。

第一版不调用 LLM 自动合并同义词，避免将同姓人物、相似地名或不同事件错误合并。

### 8.4 结构化输出

文章级示例：

```json
{
  "term": "官方通报",
  "score": 0.86,
  "rank": 1,
  "source": "tfidf",
  "type": "phrase"
}
```

事件级示例：

```json
{
  "term": "重庆",
  "score": 0.91,
  "rank": 1,
  "source": "tfidf",
  "type": "location",
  "document_count": 18,
  "document_ratio": 0.45,
  "platform_count": 3
}
```

`score` 在当前文章或当前事件内部归一化到 `0～1`，用于排序、词云字号和展示；不同 `analysis_run` 的归一化分数不能直接横向比较。

## 9. 任务内 TF-IDF 与跨任务 BGE

TF-IDF 的词表和 IDF 依赖本次语料库。不同 `analysis_run` 独立拟合的向量即使维度相同，也不属于同一特征空间，不能直接计算余弦相似度。

因此明确分工：

```text
任务内 TF-IDF
→ 本次关键词
→ 本次文章相似度
→ 本次事件聚合候选

固定版本 BGE
→ 跨 analysis_run 相似事件检索
→ 用户分析结果匹配全局事件
→ TF-IDF 边界候选的语义兜底
```

第一版本地模型为：

```text
BAAI/bge-small-zh-v1.5
```

BGE 向量按以下键缓存：

```text
article_id
+ content_version/article_snapshot_id
+ model_name
+ model_version
+ embedding_preprocess_version
```

BGE 不可用时不阻塞内容分析。任务继续输出 TF-IDF 和关键词，并记录 `BGE_UNAVAILABLE`；跨任务相似检索和边界语义判断进入降级状态。

## 10. 数据保存

### 10.1 `document_features`

永久保存与文章内容本身相关、可跨任务复用的特征：

```text
tokens
tfidf_tokens
sentiment_tokens
topics
mentions
segment_version
updated_at
```

其中 `topics` 表示平台原始话题标签或预处理阶段抽取的显式话题，不表示某次 LDA 运行产生的主题归属。

当前已有 `document_features.tfidf_vector` 字段暂时保留为可空兼容字段，避免破坏现有迁移；第一版内容分析不再向该字段写入任务相关向量。

### 10.2 `analysis_run`

建议保存：

```text
id / user_id / mode / keyword / platforms
query_fingerprint / dataset_hash
article_count / representative_count
tfidf_config / title_weight / body_weight
segment_version / stopword_version / synonym_version
algorithm_version / sklearn_version
status / warnings / error_code
created_at / completed_at
```

### 10.3 `analysis_run_article`

建议保存：

```text
analysis_run_id
article_id
article_snapshot_id 或 content_version
is_representative
nlp_weight
feature_status
keywords_json
warnings
```

文章级关键词属于 `analysis_run`，不能写入永久的 `document_features` 后被下一次任务覆盖。

### 10.4 `article_embedding`

固定版本 BGE 向量建议独立保存，不写入 `document_features.tfidf_vector`：

```text
article_id
article_snapshot_id 或 content_version
model_name / model_version
embedding_preprocess_version
dimension
vector 或向量存储引用
created_at
```

同一文章内容版本和模型版本只计算一次。更换模型时保留版本隔离，不将不同维度或不同模型产生的向量混合比较。

### 10.5 不保存的内容

- 不保存每篇文章的完整 TF-IDF 稀疏向量。
- 不保存本次完整文档-词矩阵。
- 不复制 `raw_content`、`clean_content` 或 `raw_json`。

需要复核时，根据文章快照、数据集哈希和版本化参数重新计算。

## 11. 服务与组件设计

建议在 `backend/app/analysis` 和 `backend/app/services` 中按职责拆分：

```text
backend/app/analysis/
├── feature_matrix.py       # 词表、词频矩阵和 TF-IDF 矩阵
├── keyword_extractor.py    # 文章级和事件级关键词
├── feature_config.py       # 可版本化参数对象
└── result.py               # 结构化结果和警告

backend/app/services/
└── content_analysis_service.py  # analysis_run 编排、读取和持久化
```

组件接口建议：

```python
build_feature_matrices(documents, config) -> FeatureMatrixResult

extract_article_keywords(matrix_result, documents, config)
    -> list[ArticleKeywordResult]

aggregate_event_keywords(matrix_result, event_mapping, documents, config)
    -> list[EventKeywordResult]

run_content_analysis(analysis_run_id) -> ContentAnalysisResult
```

`analysis_service.py` 负责总流程编排，不应把矩阵构建、关键词规则和数据库写入全部堆在一个函数中。

## 13. 幂等、缓存与并发

### 13.1 幂等键

建议内容分析幂等键为：

```text
analysis_run_id
+ dataset_hash
+ tfidf_config_hash
+ algorithm_version
```

同一幂等键已经成功时直接复用结果。

### 13.2 缓存规则

- 用户相同关键词、平台组合和采集条件在 24 小时内优先复用文章数据。
- 文章数据相同但 TF-IDF 或关键词算法版本变化时，复用文章并重新分析。
- 数据集和分析配置完全相同时，可以复用分析结果。
- 失败或取消的任务不能作为成功缓存。

### 13.3 并发

单个 `analysis_run` 的 TF-IDF 应统一拟合，不能由多个工作线程分别构建不同词表后合并。多个互不相关的 `analysis_run` 可以并行，但应限制 CPU 和内存并发数。

## 14. 错误码与降级

| 错误码 | 含义 | 处理 |
|--------|------|------|
| `NO_VALID_DOCUMENT` | 没有可分析文章 | 任务失败 |
| `SMALL_CORPUS` | 有效代表文章少于 5 篇 | 降级运行并提示 |
| `EMPTY_VOCABULARY` | 过滤后词表为空 | TextRank/实体 fallback |
| `TITLE_ONLY_DOCUMENT` | 仅有有效标题 | 允许分析并记录 |
| `TFIDF_BUILD_ERROR` | 矩阵构建失败 | 任务失败，保留日志 |
| `KEYWORD_EXTRACT_ERROR` | 某篇关键词提取失败 | 跳过单篇，其他继续 |
| `BGE_UNAVAILABLE` | BGE 未加载或推理失败 | TF-IDF 主流程继续 |
| `DATASET_CHANGED` | 运行时文章版本与快照不一致 | 终止并重新创建快照 |

任务日志至少记录：

```text
analysis_run_id / task_id / stage
dataset_hash / config_hash / algorithm_version
input_count / representative_count / skipped_count / failed_count
matrix_shape / vocabulary_size / elapsed_ms
warnings / error_code / error_message
```

## 15. 性能与资源

用户搜索第一版默认所有选中平台合计目标 100 条文章。扣除重复和无效内容后，TF-IDF 矩阵规模较小，使用 scipy/sklearn 稀疏矩阵即可，不需要 GPU、Spark、Redis 向量库或分布式计算。

性能原则：

- 不将稀疏矩阵转换为稠密 JSON。
- 不对 100 篇文章执行全量 BGE 两两比较。
- BGE 只计算需要复用或进入边界判断的文章，并缓存结果。
- TextRank 只在单篇关键词不足或无 TF-IDF 时运行。
- 记录矩阵构建和关键词提取耗时，为后续优化提供依据。

## 16. 测试方案

### 16.1 单元测试

1. 预分词输入不会被 sklearn 默认英文 tokenizer 再切分。
2. `ngram_range=(1, 2)` 能生成“官方 通报”等二词特征。
3. 标题和正文权重为 `1:1` 时结果可复现。
4. 修改标题权重会按预期改变加权词频。
5. 停用词、纯数字、模板词和无意义单字被过滤。
6. TextRank 只补充不足的文章级关键词。
7. 二词短语和包含关系去重符合规则。
8. 结构化关键词分数归一化到 `0～1`。
9. `nlp_weight` 影响事件关键词汇总，不错误改变单篇余弦方向。

### 16.2 集成测试

1. 从已完成预处理的文章创建 `analysis_run` 并生成关键词。
2. 同一重复组只有代表文章进入矩阵。
3. 文章正文更新后旧快照仍可复现旧分析。
4. 相同数据集和配置重复执行可命中幂等结果。
5. 算法版本变化时复用文章但重新分析。
6. 2～4 篇文章产生 `SMALL_CORPUS`。
7. 单篇文章使用 TextRank/词频/实体降级。
8. 空词表和 BGE 不可用时主流程按规则降级。

### 16.3 调参标注集

建立至少 50～100 对人工标注样本，后续扩展到 100～200 对，类别包括：

```text
same_event
different_event_same_topic
unrelated
```

重点包含跨平台改写、相似标题不同地点、标题党、同人物不同事件和同主题不同时间。使用该标注集联合选择：

- 标题权重。
- 事件相似度阈值。
- BGE 边界阈值。

标题权重候选：

```text
0.5 / 1.0 / 1.5 / 2.0 / 3.0
```

第一版默认 `1.0`，直到标注结果证明其他值更好。

## 17. 与下游模块的接口约束

### 17.1 热点事件发现

- 后台热点模式可以读取本模块提供的词频矩阵执行 LDA。
- 用户关键词搜索模式不强制执行 LDA。
- LDA 主题数、主题命名和主题分类不属于本模块。

### 17.2 事件聚合

- 只使用本次 `analysis_run` 内的 TF-IDF 向量进行任务内比较。
- 不同 `analysis_run` 的 TF-IDF 向量禁止直接比较。
- BGE 只处理 TF-IDF、实体和时间规则不能明确判断的边界候选。
- Single-Pass 的具体排序、阈值、中心更新和聚类后合并在事件聚合设计中定义。

### 17.3 历史事件检索

- 跨任务和历史事件相似度使用固定版本 BGE 表示。
- 历史查询可以使用关键词倒排进行初筛，再使用 BGE 排序。
- 更换 BGE 模型版本时不得混用旧向量，应按模型版本分区或重算。

## 20. 第一版验收标准

内容分析模块完成时应满足：

1. 能读取已预处理文章并只选择有效重复组代表文章。
2. 能在单个 `analysis_run` 内生成可复现的词频和 TF-IDF 稀疏矩阵。
3. TF-IDF 参数、数据集和算法版本完整记录。
4. 能为每篇文章输出最多 10 个结构化关键词。
5. 能根据事件文章映射输出最多 20 个结构化事件关键词。
6. 0、1、2～4 篇文章和空词表均有明确处理结果。
7. 不把完整 TF-IDF 矩阵或任务相关关键词写入永久文章特征表。
8. 不直接比较不同 `analysis_run` 的 TF-IDF 向量。
9. BGE 不可用时 TF-IDF 和关键词主流程仍可完成。
10. 单元测试和集成测试覆盖正常、降级、幂等和版本变化场景。
