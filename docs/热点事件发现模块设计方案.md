# 热点事件发现模块设计方案

## 1. 文档目的

本文档定义网络舆情事件智能分析系统第一版的热点事件发现模块。模块承接爬虫、数据清洗与内容分析结果，使用 LDA 发现舆情主题，调用可配置的大模型完成主题语义分类与命名，并在事件聚合模块产生稳定事件后计算事件热度、保存热度快照和生成热点榜单。

本文档同时补充热榜种子如何转化为真实事件语料、LDA 主题如何与持久化事件区分、热点结果如何跨多次任务连续更新等问题。`项目需求规格说明书.md` 中与本文冲突的旧表述，应以后续同步修订后的本文方案为准。

当前仓库中的 `backend/app/analysis/topic_classifier.py`、`backend/app/analysis/hot_detector.py` 和 `backend/app/services/event_service.py` 仍是占位实现。本文档描述的是后续实现依据，不代表这些功能已经完成。

---

## 2. 范围与边界

### 2.1 本模块负责

```text
热榜种子和扩展采集结果
→ 创建可复现的热点分析运行
→ 基于 analysis_run 代表文章重建词频矩阵
→ LDA 主题发现和文档主题分配
→ 固定一级分类与动态主题名称
→ 向事件聚合模块输出运行级主题结果
→ 接收稳定的文章—事件映射
→ 计算核心热度、传播热度和最终热度
→ 保存事件热度时序快照
→ 生成符合最低证据门槛的热点 Top 20
```

具体职责包括：

1. 管理热点分析运行、参数、版本、状态和降级信息。
2. 在最近 7 天滑动窗口内对代表文章执行 LDA 主题发现。
3. 自动选择主题数，并对小语料进行明确降级。
4. 使用 DeepSeek `deepseek-chat` 对主题进行一级分类和二级命名；调用失败时使用关键词降级。
5. 为事件聚合模块提供文章主主题、主题概率、主题关键词和运行标识。
6. 基于稳定 `event.id` 计算热度，不把 LDA 的临时主题编号当作全局事件编号。
7. 保存原始统计、分项热度、最终热度、榜单排名和公式版本。
8. 区分 `hot`、`search` 和 `manual` 三种运行模式，防止隔离语料污染公共热点榜单。

### 2.2 本模块不负责

- 热榜接口和关键词搜索适配器：由 `backend/app/crawler` 负责。
- 正文提取、清洗、质量评估、去重和分词：由 `backend/app/preprocessing` 负责。
- Count/TF-IDF 特征构建和文章关键词：由内容分析模块负责。
- Single-Pass 聚类、稳定事件身份、事件合并和文章归属：由独立事件聚合模块负责。
- 情感判断、生命周期预测、风险评估和报告生成：由后续模块负责。
- 将热榜平台的榜单结果当作真实的全网绝对热度。

### 2.3 与事件聚合模块的关系

热点事件发现和事件聚合分开实现，但通过明确接口连接：

```text
热点事件发现：发现主题、分类命名
        ↓
事件聚合：识别具体事件、维护稳定 event.id
        ↓
热点事件发现：按稳定事件计算热度和榜单
```

热点模块不能绕过事件聚合，直接把一个 LDA 主题写成事件。一个主题可能包含多个具体事件，同一事件也可能随着语料窗口变化而落入不同的 LDA 主题。

## 3. 已确认的核心决策

| 决策 | 第一版方案 |
|------|------------|
| 模块边界 | LDA 主题发现与热度排名属于本模块；Single-Pass 事件聚合独立实现 |
| 采集策略 | 热榜发现种子，再用种子执行多平台扩展搜索 |
| 热榜条目语义 | 搜索线索和传播信号，不计为独立报道 |
| 全局热点语料窗口 | 最近 7 天滑动窗口 |
| 机器学习方法 | LDA 无监督主题模型 |
| LDA 输入 | 标题与正文合并后的整数词频矩阵 `count_matrix` |
| 主题数 | 小语料分级处理；正常语料在限定候选范围内自动选择 K |
| 一级分类 | 社会事件、公共卫生、自然灾害、政治事件、经济事件、娱乐事件、其他 |
| 二级名称 | 根据主题关键词和代表文章标题动态生成 |
| LLM | 默认 DeepSeek `deepseek-chat`，通过 OpenAI 兼容配置切换模型 |
| LLM 故障 | 关键词规则降级，任务成功但带警告 |
| 热度结构 | 核心热度与传播热度分层计算 |
| 最终热度 | `0.7 × core_heat + 0.3 × spread_heat` |
| 热点门槛 | 最低证据门槛后按最终热度取 Top 20 |
| 时间规则 | `publish_time` 优先；缺失时使用 `first_crawled_at` 降级并标记 |
| 稳定身份 | `event.id` 跨运行持久；LDA `topic_index` 仅在单次运行内有效 |
| 历史记录 | 每次计算写入不可覆盖的事件热度快照 |

## 4. 关键对象与语义

### 4.1 热榜种子

热榜种子来自微博热搜、知乎热榜、百度热搜、RSS 或管理员提交的候选词。它回答“系统接下来应该搜索什么”，不等于一篇报道，也不直接等于一个事件。

示例：

```text
微博热搜：“重庆多地暴雨”
知乎热榜：“重庆暴雨为何持续”
百度热搜：“重庆暴雨救援”

→ 合并候选种子：“重庆暴雨救援”
```

### 4.2 LDA 主题

LDA 主题属于某次热点分析运行。其 `topic_index`、关键词和文章概率分布会随语料窗口、词表、主题数和随机种子变化，不能跨运行作为稳定业务标识。

### 4.3 舆情事件

事件由事件聚合模块创建并维护，拥有稳定 `event.id`。新文章可以加入已有事件，事件也可以被合并、重新激活或转为沉寂状态。

### 4.4 热点状态

热点不是永久属性，而是事件在某个计算时点的状态。同一事件可能经历：

```text
普通事件 → 新进入热点 → 热度上升 → 排名下降 → 退出热点 → 再次升温
```

因此只在 `event.heat_index` 中覆盖当前值不够，必须保存版本化热度快照。

## 5. 两阶段种子驱动采集

### 5.1 第一阶段：候选热点发现

后台 `hot` 任务低频抓取各平台热榜：

- 微博热搜：现有 `WeiboHotCrawler`。
- 知乎热榜：配置知乎开放平台后启用 `ZhihuHotCrawler`。
- 百度热搜：配置千帆 API 后启用 `QianfanTrendingCrawler`。
- RSS：补充新闻站、政府机构或媒体发布源。
- 管理员候选：用于补充榜单未覆盖但需要追踪的事件。

热榜采集结果需要保留：

```text
platform
source_article_id / seed_key
title
rank
platform_hot_score
snapshot_time
source_url
raw_json
```

榜单快照只追加，不覆盖历史记录。相同词条在不同时点的排名变化可作为传播信号。

### 5.2 候选词标准化与合并

候选整理依次执行：

1. 去除话题符号、营销后缀和平台固定前缀。
2. 统一全角半角、空白和常见异体表达。
3. 使用规范化文本进行精确合并。
4. 使用关键词重合、标题相似度和可选 BGE 相似度合并近义种子。
5. 保存原始种子到合并候选的映射，不能丢失来源榜单证据。

候选优先级综合：

- 榜单排名；
- 同时出现的平台数量；
- 排名上升速度；
- 是否为首次出现；
- 是否匹配已有活跃事件；
- 最近是否已经执行过扩展采集。

### 5.3 第二阶段：多平台扩展采集

系统对高优先级候选调用普通搜索适配器：

- 百度网页或新闻搜索；
- 知乎搜索；
- B站视频搜索；
- 配置 TikHub 后的微博、小红书和抖音搜索；
- 已配置 RSS 源的相关文章。

热榜种子与扩展任务之间必须保存追溯关系：

```text
seed_id
normalized_query
crawl_task_id
platform
article_id
discovered_at
```

热榜条目自身不增加 `independent_report_count`。只有扩展采集后通过清洗和去重的代表文章才算独立报道。

### 5.4 采集预算

第一版使用有限预算，避免对每个榜单词无限扩展：

1. 每个平台先读取榜单 Top 20。
2. 合并近义种子后形成候选池。
3. 按候选优先级依次扩展，服从现有任务最大采集量和单平台上限。
4. 同一候选在缓存有效期内不重复创建相同搜索任务。
5. 活跃事件提高补采频率，沉寂事件逐步降低频率。

### 5.5 当前实现缺口

现有 `hot` 爬虫任务只会选择名称以 `_hot` 结尾的适配器并保存热榜条目，尚未自动执行“种子整理 → 多平台搜索 → 内容分析”的第二阶段。本模块实现前需要由任务编排层补上该接口，但搜索适配器本身仍属于爬虫模块。

## 6. 运行模式

### 6.1 `hot`

- 系统或管理员触发的全局热点模式。
- 默认读取最近 7 天内的全局扩展采集文章。
- 可以创建或更新全局事件和公共热点榜单。
- `user_id` 可以为空或记录触发管理员，但结果归属全局命名空间。

### 6.2 `search`

- 用户主动输入关键词产生的分析。
- 结果按用户和 `analysis_run` 隔离。
- 不更新公共热点榜单，不自动写入全局事件。
- 后续可以在用户自己的搜索结果中展示局部主题和局部热度。

### 6.3 `manual`

- 人工选择文章进行测试、重算或管理分析。
- 默认隔离，不进入全局事件流程。
- 管理员可以显式发布到全局流程。
- 发布操作必须记录管理员、源运行、发布时间和发布原因，且同一源运行只能幂等发布一次。

```text
article_snapshot.crawled_at
article_snapshot.comments_count
article_snapshot.reposts_count
article_snapshot.likes_count
article_snapshot.views_count
```

### 7.2 代表文章

LDA 和事件聚合只使用 `analysis_run_article.is_representative = true` 的文章。重复文章不重复进入主题模型和独立报道数量，但其互动快照可以在传播热度中以 `duplicate_weight` 修正后使用。

### 7.3 时间窗口

`hot` 模式默认选取最近 7 天文章。时间字段按以下顺序确定：

1. 有可靠 `publish_time` 时使用发布时间。
2. 缺少发布时间时使用 `first_crawled_at` 作为 `observed_at`。
3. 降级文章标记 `time_source=observed_at`，不能在页面或报告中显示成真实发布时间。

事件的发生时间和最早报道时间只使用可靠发布时间；时间衰减和滑动窗口允许使用降级观测时间。

### 7.4 时间可信度

按事件成员中可靠发布时间占比记录：

- `high`：不低于 80%。
- `medium`：不低于 50% 且低于 80%。
- `low`：低于 50%。

全部成员都缺少可靠发布时间时，事件仍可参加热点计算，但接口必须返回“按采集时间计算”的提示。

## 8. LDA 主题发现

### 8.1 为什么使用 LDA

题目要求利用机器学习方法进行舆情主题分类。LDA 是无监督概率主题模型，可以从未标注语料中学习主题—词语分布和文章—主题分布，适合当前缺少人工分类语料的项目。

LDA 负责发现主题，不负责识别稳定事件。事件识别仍由后续 Single-Pass 聚类完成。

### 8.2 输入矩阵

LDA 使用内容分析模块生成的未加权整数 `count_matrix`：

```text
标题 tokens + 正文 tfidf_tokens
→ CountVectorizer
→ 文档—词语整数词频矩阵
→ LDA
```

标题/正文 TF-IDF 权重不用于 LDA 概率计数。LDA 可以独立配置最大词表、`min_df` 和 `max_df`，但必须保存配置哈希和有效参数。

### 8.3 小语料分级

| 代表文章数 | 处理方式 | 运行状态 |
|------------|----------|----------|
| 0 | 不运行，返回 `NO_VALID_DOCUMENT` | failed |
| 1～4 | 不运行 LDA，以全语料关键词生成一个降级主题 | success，`warnings` 非空 |
| 5～9 | 固定 `K=2` 运行小语料 LDA | success with `SMALL_CORPUS` |
| 不少于 10 | 在限定候选范围内自动选择 K | success |

1～4 篇降级时仍输出统一结构，`method=keyword_topic_fallback`。事件聚合可以继续处理文章，不因 LDA 降级中断。

### 8.4 候选主题数

第一版候选范围：

| 代表文章数 N | 候选 K |
|---------------|--------|
| 10～29 | 2、3、4 |
| 30～99 | 3、4、5、6、7、8 |
| 不少于 100 | 从 5 到 `min(12, round(sqrt(N)))` |

如果不少于 100 篇但上限计算结果小于 5，则仍使用 `K=5`。最大主题数固定为 12，避免课程项目在有限语料上产生过度细分和过高计算成本。

### 8.5 模型选择

每个候选 K 使用相同词表、固定 `random_state=42` 和批量学习方式训练。选择顺序为：

1. 过滤主题关键词高度重复的模型。主题多样性定义为各主题 Top 20 关键词去重数除以 `K × 20`，第一版最低为 0.70。
2. 过滤过小主题比例超过 20% 的模型。过小主题指主分配文章数低于 `max(2, ceil(N × 0.03))`。
3. 在剩余模型中选择困惑度最低者。
4. 困惑度差异不超过 2% 时选择较小 K，减少无意义拆分。
5. 如果全部候选被过滤，选择困惑度最低的模型并记录 `TOPIC_QUALITY_DEGRADED`。

模型运行至少保存：

```text
candidate_k
selected_k
perplexity
topic_diversity
small_topic_ratio
random_state
feature_count
document_count
```

### 8.6 文档主题分配

每篇文章保存完整主题概率中概率较高的部分，并取最大概率主题作为主主题：

```text
primary_topic_id = argmax(document_topic_distribution)
primary_probability = max(document_topic_distribution)
```

主概率低于可配置阈值时标记 `LOW_TOPIC_CONFIDENCE`，但不直接丢弃文章。事件聚合仍应以文章向量相似度为主要依据，主题只用于缩小候选范围和增强解释。

## 9. 主题分类与命名

### 9.1 两级标签体系

一级分类固定为：

```text
社会事件
公共卫生
自然灾害
政治事件
经济事件
娱乐事件
其他
```

二级主题名称动态生成，例如“重庆暴雨救援”“某品牌食品安全争议”。一级分类用于筛选和统计，二级名称用于看板展示和事件聚合上下文。

### 9.2 LLM 输入

每个主题只调用一次大模型，输入：

- 主题 Top 20 关键词；
- 主概率最高的少量代表文章标题；
- 固定一级分类列表；
- 严格 JSON 输出约束。

正文不全量发送，避免无必要的成本和不可信网页内容对提示词产生影响。

### 9.3 LLM 配置

默认配置沿用项目现有环境变量：

```text
LLM_BASE_URL=https://api.deepseek.com
LLM_MODEL_NAME=deepseek-chat
LLM_API_KEY=<environment secret>
```

业务代码只依赖统一 `LLMClient`，不直接依赖某个厂商版本。

### 9.4 结构化输出

```json
{
  "category": "自然灾害",
  "topic_name": "重庆暴雨救援",
  "confidence": 0.91,
  "method": "lda_llm"
}
```

必须校验一级分类枚举、名称长度、置信度范围和 JSON 结构。

### 9.5 降级

LLM 超时、限流、鉴权失败或输出不合法时：

1. 保存 LDA 原始结果。
2. 一级分类使用“其他”。
3. 二级名称由前三至五个有效关键词组合。
4. 标记 `method=lda_keyword_fallback`。
5. 热点任务成功完成，但记录 `LLM_TOPIC_NAMING_DEGRADED`。

相同主题关键词签名可以复用既有命名，减少重复调用。


### 10.2 事件聚合返回

事件聚合完成后返回：

```text

similarity
aggregation_version
event_first_publish_time
event_last_activity_time

```

### 10.3 稳定身份规则

- `topic_index` 只在一个 `hotspot_run` 内唯一。
- `event.id` 跨运行保持稳定。
- 同一文章在同一聚合版本中只能有一个有效事件归属。
- 事件合并后保留旧事件到新事件的别名或合并记录，历史热度快照不能静默丢失。
- 事件重新升温时优先激活已有事件；是否创建新事件由事件聚合模块根据时间跨度和语义差异决定。

## 11. 热度计算

### 11.1 原始统计

每个事件至少统计：

```text
independent_report_count_7d
independent_report_count_24h
independent_report_count_previous_24h
report_growth_rate
platform_count
latest_activity_time
comments_count
reposts_count
likes_count
views_count
hotlist_platform_count
best_hotlist_rank
```

独立报道数量只统计去重代表文章。互动量允许读取全部有效文章和最新快照，但需要应用 `duplicate_weight` 和 `spam_weight`。

### 11.2 核心热度

核心热度由四项组成：

1. 独立报道数量；
2. 最近 24 小时相对前 24 小时的报道增长率；
3. 独立平台覆盖数；
4. 距离最近一次活动的时间衰减。

前三项在本次事件候选集合中转换为 0～100 的百分位分数；候选事件只有一个时取 50。时间衰减使用 24 小时半衰期：

```text
freshness_score = 100 × exp(-ln(2) × hours_since_last_activity / 24)
```

第一版四项等权：

```text
core_heat = mean(
    report_count_score,
    report_growth_score,
    platform_coverage_score,
    freshness_score
)
```

等权只是透明的第一版基线。原始值和分项分数必须保存，后续可以在标注集上调整内部权重。

### 11.3 传播热度

传播热度处理评论、转发、点赞、浏览和热榜信号：

1. 互动量先执行 `log1p`，降低头部账号极端数值影响。
2. 每种指标在同一平台内单独计算百分位，不能直接比较微博点赞和B站播放量。
3. 单篇文章只平均其实际可用指标，未知值不当作零。
4. 单篇传播贡献乘以时间衰减、`duplicate_weight` 和 `spam_weight`。
5. 事件内累加文章传播贡献，再在候选事件集合中转换为 0～100 百分位。
6. 热榜排名作为独立传播信号参与可用信号平均，但热榜条目不增加独立报道数。

全部传播信号缺失时标记 `spread_available=false`，不把未知传播热度写成零。

### 11.4 最终热度

传播数据可用时：

```text
final_heat = 0.7 × core_heat + 0.3 × spread_heat
```

传播数据完全不可用时：

```text
final_heat = core_heat
```

每次计算保存 `core_weight=0.7`、`spread_weight=0.3` 和 `heat_formula_version=v1`。70/30 是第一版经验权重，不表述为训练所得结论。

### 11.5 热点证据门槛

事件必须先满足：

```text
最近 7 天至少 3 篇去重后的独立报道
并且：
    覆盖至少 2 个平台或独立来源
    或存在正式热榜种子证据
并且最近 24 小时仍有新增报道或互动
```

满足门槛的事件按 `final_heat` 降序排列，公共榜单最多展示 20 个。未满足门槛的事件仍保留为普通事件，不进入热点榜单。

### 11.6 热度语义

系统展示的是“当前已配置数据源和采集频率下的观测热度”，不是全互联网绝对热度。接口和报告应返回覆盖平台、缺失指标和计算时间，避免过度解释排名。

## 12. 跨运行状态与热度快照

### 12.1 为什么不能只覆盖 `event.heat_index`

如果每次任务只更新当前值，将无法回答：

- 事件何时进入热点；
- 排名为何上升或下降；
- 哪个版本公式生成了某次结果；
- 任务重试是否重复计算；
- 事件合并前后的历史如何追溯。

### 12.2 快照原则

- 每个 `hotspot_run` 对同一 `event_id` 最多写一条快照。
- 快照写入后不覆盖，只允许以新的运行追加。
- `event.heat_index` 可以保存最新值，作为看板快速读取字段。
- 历史趋势以热度快照为准，不从当前值反推。
- 事件合并必须保留原事件快照和合并关系。

### 12.3 热点状态变化

快照可以派生：

```text
new_hot       首次满足热点门槛
rising        最终热度或排名明显上升
stable        变化处于稳定区间
cooling       热度下降但仍在榜
left_hot      本次不再满足门槛或跌出 Top 20
reheated      退出后再次进入热点
```

具体生命周期阶段由趋势预测模块负责；本模块只保存足够的时序证据。

## 14. 服务与组件设计

建议保持单一职责：

| 组件 | 职责 |
|------|------|
| `hot_seed_service.py` | 读取、规范化和合并热榜种子；实际爬取仍由 crawler 完成 |
| `topic_model.py` | 训练候选 LDA、计算指标和选择 K |
| `topic_classifier.py` | 文档主题分配、LLM 分类命名和关键词降级 |
| `hotspot_service.py` | 创建运行、验证快照、调用主题发现和保存结果 |
| `heat_calculator.py` | 计算原始统计、分项热度和最终热度 |
| `hotspot_ranking_service.py` | 应用证据门槛、生成 Top 20 和状态变化 |
| `event_aggregation_service.py` | 独立模块，维护稳定事件和文章映射 |
| `jobs.py` | 只负责后台阶段编排和任务进度，不放算法实现 |

现有 `hot_detector.py` 不应同时承担种子采集、LDA、聚类、热度计算、数据库写入和 API 序列化。

## 15. 后台任务流程

### 15.1 全局热点任务

```text
1. 拉取热榜快照
2. 规范化并合并候选种子
3. 为高优先级种子执行多平台扩展搜索
4. 对新文章执行预处理
5. 创建并运行 mode=hot 的 analysis_run
6. 创建 hotspot_run
7. 运行 LDA 和主题分类命名
8. 调用事件聚合模块
9. 计算事件热度快照
10. 更新事件当前热度和公共 Top 20
11. 记录任务结果、警告和各阶段计数
```

第一版可以沿用当前线程任务执行器和任务租约，不引入 Celery。定时频率沿用爬虫方案中的每天 4 次；管理员可以手动触发，但相同配置和时间窗口内需要复用或拒绝重复任务。

### 15.2 搜索分析任务

`search` 模式只运行用户范围内的主题发现和可选局部聚合，不更新全局 `event` 当前热度，也不写公共榜单排名。

### 15.3 管理员发布

`manual` 运行发布到全局时：

1. 验证发布者管理员权限。
2. 固化原运行数据集和内容版本。
3. 创建一次全局发布记录。
4. 交给事件聚合模块处理。
5. 由新的全局热点运行计算热度，不能直接复制隔离运行的排名。

## 16. 幂等、并发与重试

### 16.1 幂等键

热点运行复用至少考虑：

```text
mode/scope
window_start/window_end
analysis_run.dataset_hash
topic_config_hash
heat_config_hash
algorithm_versions
```

同一幂等键已有成功结果时直接复用；已有运行中任务时返回原任务；失败运行可以创建新尝试，但不能覆盖旧记录。

### 16.2 事件热度幂等

- `event_heat_snapshot` 使用 `(hotspot_run_id, event_id)` 唯一约束。
- 任务重试不重复增加报道数量，因为统计从文章—事件映射重新聚合，而不是累加旧值。
- 更新 `event` 当前汇总字段前校验当前快照是否比数据库中的快照更新。

### 16.3 并发

- 同一全局时间窗口只允许一个热点任务进入写事件和榜单阶段。
- 主题训练可以在数据库锁外运行，最终写入前再次验证 `analysis_run` 数据集身份。
- 任务租约失效后禁止提交主题、事件和热度结果。
- 新爬虫数据在运行中到达时不加入当前快照，进入下一次运行。

## 17. 错误处理与降级

| 错误码/警告 | 场景 | 处理 |
|-------------|------|------|
| `NO_VALID_DOCUMENT` | 无代表文章 | 运行失败 |
| `KEYWORD_TOPIC_FALLBACK` | 1～4 篇文章 | 关键词生成单主题，继续 |
| `SMALL_CORPUS` | 5～9 篇文章 | 固定 K=2，继续 |
| `EMPTY_VOCABULARY` | 词频矩阵为空 | 运行失败并保留诊断统计 |
| `TOPIC_QUALITY_DEGRADED` | 候选 K 均未通过质量过滤 | 选择最低困惑度模型，继续 |
| `LOW_TOPIC_CONFIDENCE` | 文章主主题概率低 | 保存标记，交给聚合模块谨慎处理 |
| `LLM_TOPIC_NAMING_DEGRADED` | LLM 不可用或输出无效 | 关键词命名，继续 |
| `EVENT_AGGREGATION_UNAVAILABLE` | 聚合模块失败 | 保存主题结果，不生成新热度快照 |
| `SPREAD_DATA_UNAVAILABLE` | 全部互动和热榜信号缺失 | 最终热度退化为核心热度 |
| `TIME_SOURCE_DEGRADED` | 使用采集时间代替发布时间参与计算 | 继续并降低时间可信度 |
| `DATASET_CHANGED` | 内容版本与快照不一致 | 运行失败，不写部分结果 |
| `STALE_TASK_LEASE` | 后台任务租约过期 | 回滚未提交结果 |

主题发现成功但事件聚合失败时，`hotspot_run` 应区分阶段状态，不能把已经完成的主题结果误报为全部失败，也不能把旧榜单清空。

## 20. 测试方案

### 20.1 单元测试

1. 候选种子规范化和近义合并。
2. 热榜条目不计入独立报道数。
3. 小语料 0、1～4、5～9 和不少于 10 篇的分支。
4. 候选 K 生成、质量过滤和较小 K 平局规则。
5. LDA 固定随机种子后的可复现性。
6. 文档主主题和低置信度标记。
7. LLM JSON 校验、缓存和关键词降级。
8. 发布时间优先与采集时间降级。
9. 同平台互动量归一化和缺失值处理。
10. 核心热度、传播热度和 70/30 最终热度。
11. 传播数据完全缺失时退化为核心热度。
12. 最低证据门槛和 Top 20 截断。
13. 热度状态变化推导。

### 20.2 集成测试

1. 热榜种子触发多平台扩展搜索并保存追溯映射。
2. 扩展文章经预处理和内容分析后创建 `hotspot_run`。
3. 主题结果正确关联 `analysis_run` 和文章内容版本。
4. 事件聚合返回稳定 `event.id` 后生成热度快照。
5. 同一任务重试不重复写主题分配和热度快照。
6. 新数据在下一次运行加入旧事件并更新热度。
7. LDA 主题编号变化时稳定事件身份不变。
8. 事件退出热点后旧快照仍可查询。
9. `search` 和未发布 `manual` 运行不污染公共榜单。
10. 管理员发布 `manual` 运行具有完整审计记录。
11. LLM、部分热榜源或互动数据不可用时主流程可降级完成。
12. 事件聚合失败时保留旧榜单且不产生不完整的新榜单。

### 20.3 人工评估集

第一版至少整理一批可人工判断的事件语料，标注：

- 文章主题是否合理；
- 两篇文章是否属于同一事件；
- 主题名称是否可读；
- Top 20 排名是否符合人工直觉；
- 热榜种子与扩展文章是否相关；
- 时间缺失和跨平台互动异常样本。

该评估集用于调整 LDA 主题数范围、主主题置信度、事件聚合阈值和热度内部权重。没有标注结果前，70/30 只作为第一版经验配置。

## 21. 性能与资源

- LDA 只处理最近 7 天代表文章，不处理重复文章和无效文本。
- 词表规模沿用内容分析约束，并允许热点模式独立设置更严格上限。
- 最大候选主题数为 12，候选模型按顺序训练，避免同时占用过多内存。
- LLM 按主题调用，不按文章调用；一次运行通常只调用 K 次。
- 主题关键词签名可缓存命名结果。
- 热度统计优先使用数据库聚合和最新快照，避免逐条远程请求平台接口。
- 公共事件列表读取 `event` 当前汇总；趋势页再读取 `event_heat_snapshot`。

## 22. 对现有需求和代码骨架的修正

1. LDA 输入应为整数词频矩阵，不表述为 TF-IDF 概率计数。
2. LDA 主题发现与 Single-Pass 事件聚合分开，不继续把二者全部塞入 `hot_detector.py`。
3. 热榜词条只作为搜索种子和传播信号，不直接创建正式事件或增加独立报道数。
4. `topic_index` 是运行级临时编号，不能写入 `article.event_id` 充当稳定事件。
5. `event_service.py` 的固定样例需要在事件聚合和热点模块完成后替换为数据库查询。
6. `event.heat_index` 只保存当前快照摘要，历史以 `event_heat_snapshot` 为准。
7. 当前爬虫 `hot` 模式缺少热榜种子到普通搜索适配器的自动扩展闭环。
8. 第一版线程任务系统继续使用，不按旧爬虫草案引入 Celery。
9. 原始互动量不能跨平台直接相加，必须在平台内归一化。
10. 系统输出应称为“已配置数据源范围内的观测热度”。

## 23. 第一版验收标准

完成实现后应满足：

1. `hot` 模式可以从至少一个真实热榜源获取种子，并触发至少两个普通内容源的扩展采集。
2. 热榜条目、扩展查询和最终文章之间可以追溯。
3. 最近 7 天代表文章能够创建可复现的热点运行。
4. 1～4、5～9 和正常语料均按本文规则处理。
5. 正常语料能够训练多个候选 LDA 并保存选 K 指标。
6. 每篇代表文章具有主主题和概率，主题具有固定一级分类和动态名称。
7. DeepSeek 不可用时主题名称可以降级生成，任务不因命名失败整体中止。
8. 主题结果可以通过明确接口交给独立事件聚合模块。
9. 同一事件跨多次热点运行保持稳定 `event.id`。
10. 每次成功热点运行为事件写入不可覆盖的热度快照。
11. 最终热度按 70% 核心热度和 30% 传播热度计算；传播数据完全缺失时有明确降级。
12. 热点榜单先应用最低证据门槛，再返回最多 20 个事件。
13. `search` 和未发布 `manual` 运行不会修改公共热点榜单。
14. 任务重试不重复累计报道量，不产生重复快照。
15. 关键算法、服务和端到端流程具备自动化测试，且现有测试继续通过。

### 24.3 方法说明

- LDA 用于无监督主题发现和文章主题分配。
- TF-IDF/BGE 与 Single-Pass 用于后续事件相似度和聚合，不以 LDA 主题编号替代事件身份。
- DeepSeek 只承担低频语义分类和命名，失败时允许确定性降级。
- 热度公式属于统计与规则模型，不宣称为机器学习训练结果。
