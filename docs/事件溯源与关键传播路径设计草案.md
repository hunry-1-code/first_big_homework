# 事件溯源与关键传播路径设计草案

## 1. 文档定位

本文档记录“事件溯源与关键传播路径”的后续设计方向，供高级功能阶段继续讨论和补充。

该功能不属于当前第一版内容分析模块。当前内容分析只负责 TF-IDF、关键词、文章特征和事件聚合所需的基础相似度数据；事件溯源需要更完整的平台关系数据、时间序列数据和传播证据，后续单独实现。

模块边界如下：

```text
数据清洗与预处理
→ 正文提取、去噪、分词

内容分析
→ TF-IDF、关键词、文本特征

事件聚合
→ 将相关文章归入同一事件

事件溯源与关键传播路径（高级功能）
→ 识别首发候选、关键传播节点和跨平台传播关系
```

## 2. 功能目标

针对已经完成聚合的事件，尽可能回答以下问题：

1. 当前可观测数据中，最早的高相关内容是什么。
2. 事件最早在哪个平台出现。
3. 事件何时从单个平台扩散到其他平台。
4. 哪些账号、媒体或官方机构构成关键传播节点。
5. 哪些内容对应爆发、放大、回应、反转、衰退和再次传播阶段。
6. 哪些传播关系由平台数据明确提供，哪些关系只是算法推测。

系统输出的是“基于可获得公开数据的传播路径”，不能宣称还原了互联网中的全部真实传播过程。

## 3. 前置数据

现有爬虫和预处理模块需要继续保留以下字段，为后续溯源提供基础：

```text
article.id
article.platform
article.source_article_id
article.url
article.title
article.raw_content / raw_json
article.clean_content
article.author / author_id
article.publish_time
article.comments_count
article.reposts_count
article.likes_count
article.views_count
article.content_hash / simhash
document_features.tokens / tfidf_tokens
```

平台接口如果能够提供以下关系，应原样保存在 `raw_json`，不要在基础清洗阶段丢弃：

```text
转发来源
引用来源
回复对象
父评论或根评论
原视频或原帖 ID
发布账号认证信息
平台提供的传播链字段
```

## 4. 总体流程

```text
选择已聚合事件
→ 获取事件关联文章
→ 相关性复核
→ 识别活跃时间段
→ 必要时定向补充采集
→ 提取关键节点候选
→ 构建明确传播关系
→ 推断缺失的传播关系
→ 生成带置信度的传播路径图
```

事件溯源不能只依赖关键词。内容相关性应综合使用：

```text
人物、地点、机构等实体
+ TF-IDF 字面相似度
+ BGE 语义相似度
+ 发布时间接近程度
+ 平台提供的引用或转发证据
```

## 5. 时间范围识别

### 5.1 不使用固定最近七天

用户通常不知道事件发生时间，例如直接搜索一个历史事件名称。溯源任务不能要求用户提前提供准确起止日期，也不能固定查询最近七天。

系统应先利用初始搜索结果生成时间分布：

```text
发布时间
→ 按天或按周分桶
→ 统计相关内容数量、互动量和平台数量
→ 找出内容密集区间和热度突变区间
```

### 5.2 自适应区间搜索

普通二分查找不适合事件传播，因为事件热度不是单调变化的。事件可能经历沉寂、爆发、衰退和再次传播，中间某个时间点没有内容，不代表整个区间都没有价值。

推荐采用“指数扩展 + 高信息区间递归细分”：

1. 从初始高相关内容的密集时间段开始。
2. 向前和向后按 1 天、3 天、7 天、14 天、30 天逐步扩大范围。
3. 连续多个区间没有高相关内容时，停止该方向的细粒度扩展。
4. 对内容数量、互动增长、跨平台数量或新观点变化较大的区间继续拆分。
5. 对长期低信息区间停止细分，减少无效请求。
6. 使用较粗时间粒度扫描远期内容，避免遗漏事件再次传播。

该方法借鉴二分思想，但判断依据是区间的信息量，而不是只检查时间中点。

## 6. 关键节点识别

关键节点候选包括：

1. 当前数据中发布时间最早且相关性较高的内容。
2. 每个平台最早出现的高相关内容。
3. 第一次获得明显互动增长的内容。
4. 第一次被高影响力账号或媒体转发、引用的内容。
5. 第一次进入官方媒体或官方机构账号的内容。
6. 首次出现新事实、新证据、新观点或事件反转的内容。
7. 跨平台传播时，各平台的首个代表内容。
8. 热度峰值和再次传播阶段的代表内容。

关键节点不能只按点赞量排序。高互动量文章可能只是传播结果，而不是传播起点。

建议的综合评分方向为：

```text
关键节点分数 =
  时间先发程度
+ 内容新颖程度
+ 互动增长程度
+ 作者或媒体影响力
+ 跨平台扩散贡献
+ 明确引用关系权重
```

具体权重应在获得真实传播样本后标定。

## 7. 传播关系和可信度

传播边分为两类。

### 7.1 明确关系

由平台接口或页面直接提供：

```text
转发
引用
回复
转载来源
父子评论
原帖或原视频关联
```

明确关系标记为：

```text
evidence_type = explicit
confidence = 1.0 或接近 1.0
```

### 7.2 推测关系

平台没有提供明确关系时，可以根据以下证据推测：

```text
发布时间先后
标题和正文相似度
BGE 语义相似度
相同人物、地点和机构
是否引用相同图片、链接或表述
是否出现“据某媒体”“网传”“转自”等文本证据
```

推测关系必须保存置信度和依据，前端不能把它展示为确定事实：

```text
evidence_type = inferred
confidence = 0.0～1.0
evidence = [时间证据, 文本证据, 实体证据]
```



`node_type` 可包括：

```text

```

### 8.3 event_trace_edge


```text

platform_count
heat_peak

```

## 9. 图结构

传播路径使用有向图表示：

```text
节点：文章、帖子、视频、媒体报道或官方回应
边：转发、引用、回复、转载、跨平台跟进或推测传播
```

示意：

```text
微博首发候选
  ├─ 大V转发
  │    └─ 新闻媒体报道
  │          └─ 官方回应
  └─ B站解读视频
       └─ 微博二次传播
```

图中必须使用不同样式区分明确关系和推测关系，并允许用户查看边的证据和置信度。

## 10. 与平台选择的关系

传播路径可以基于两种数据范围生成：

```text
用户 analysis_run：只分析用户选择的平台
global_event：使用后台热点模式收集到的全部启用平台
```

只选择部分平台时，系统只能展示这些平台范围内的传播路径，必须提示“当前路径不包含未选平台数据”。

全局事件的传播路径覆盖范围更大，但仍受平台接口、采集频率和历史数据完整度限制。

## 11. 性能方向

普通用户搜索仍以全部选中平台合计 100 条作为基础数据上限。

传播路径属于独立高级任务。如果后续允许定向补充采集，应设置单独的数量上限、时间范围和 API 成本控制，不能无界扩展，也不能影响普通搜索任务。

时间区间任务可使用优先队列管理：

```text
优先分析热度突变区间
→ 再分析跨平台扩散区间
→ 最后分析长期低活跃区间
```

## 12. 失败和降级

1. 缺少发布时间：保留节点，但不能作为可靠先后关系依据。
2. 缺少转发来源：只生成推测关系，不伪造明确边。
3. 平台接口受限：记录覆盖范围和缺失平台。
4. BGE 不可用：退化为实体、TF-IDF 和时间证据。
5. 数据量不足：只输出时间线和关键内容，不强行生成传播图。
6. 出现多个可能源头：全部标记为 `origin_candidate`，不强行选定唯一首发。

## 13. 测试与评估

后续实现前需要准备人工核验案例，至少覆盖：

```text
具有明确转发链的单平台事件
跨平台传播事件
官方回应导致二次爆发的事件
长期沉寂后再次传播的事件
存在同名人物或同名事件的干扰案例
无法确定首发来源的案例
```

重点指标包括：

```text
明确关系解析准确率
关键节点召回率
错误传播边比例
首发候选覆盖率
跨平台关键节点覆盖率
推测关系置信度校准情况
```

## 14. 后续实施前需要重新确认的范围

开始实现该高级功能前，需要单独确认：

1. 哪些平台接口能够提供明确转发、引用和回复关系。
2. 传播链任务是否允许突破普通搜索 100 条上限。
3. 定向补采的最大数量、最长时间跨度和 API 成本预算。
4. 第一版只做单平台明确关系，还是同时做跨平台推测关系。
5. 传播路径是长期保存，还是按需生成并设置缓存期限。
6. 前端采用时间线、关系图，还是两种视图同时提供。

以上内容均属于后续高级功能决策，不影响当前内容分析、事件聚合和用户搜索模块的第一版实现。
