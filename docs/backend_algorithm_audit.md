# 后端算法审计与改进建议

> 基于「台风巴威」全链路真实爬取数据（59 篇文章，百度/B站/微博/知乎 4 平台）的系统性算法评估。

## 数据基线

| 指标 | 数值 | 说明 |
|------|:---:|------|
| 爬取文章 | 59 篇 | 百度17 + B站17 + 微博15 + 知乎10 |
| 清洗成功 | 49 篇 (83%) | 13 篇因 HTML 格式/编码问题失败 |
| BGE 向量 | 48 个 (512维) | bge-small-zh-v1.5, CPU ~3.7s/篇 |
| 聚合事件 | 3 个 | 34篇 + 13篇 + 1篇 |
| LLM 情感 | 48 篇成功 | DeepSeek, 正面4/中立22/负面8（剩余为无数据） |
| LLM 关键词 | 28/48 篇成功 | 其余 20 篇回退 TF-IDF |

## 一、事件聚合与聚类

### 1.1 当前实现

`event_clusterer.py` `cluster_documents()` — 单遍增量聚类，四维相似度：
- BGE 余弦 (0.45) + TF-IDF 余弦 (0.25) + 实体 Jaccard (0.20) + 时间兼容性 (0.10)
- 附着阈值 0.55，创建阈值 0.40

### 1.2 实际表现

| 指标 | 阈值调优前 | 阈值调优后 |
|------|:---:|:---:|
| 簇数 | 40 | 3 |
| 孤篇簇 | 35/40 (87%) | 1/3 |
| attached_article | 8 | 45 |
| BGE 相似度 (同主题文章) | 0.67-0.88 | — |

### 1.3 问题

1. **余弦负值 clamp 膨胀**：`event_similarity.py:26` 把 `cos=-1.0` 映射为 0.0，`cos=0.0` 映射为 0.5。两个完全正交的向量被认定为半相似。正交通常意味无关，应映射为 0.0。

2. **单遍顺序依赖**：按时间排序增量聚类，先到达的文章可能被分到次优簇，而后出现的更匹配簇尚未创建。需要加一轮后优化重分配。

3. **`create_threshold` 未使用**：`aggregation_config.py` 定义了 `create_threshold=0.40`，但 `cluster_documents()` 从未读取，低于此阈值的文档仍会创建新簇。

4. **时间兼容性硬上限 30 天**：`max_event_gap_days=30`，超过 30 天的事件 time_score=0，对慢发酵事件（如环境治理、政策讨论）不适用。

### 1.4 改进建议

```
优先级 高：
  - 修复余弦 clamp：cos < 0 时 contrib = 0（已修正为 (raw+1)/2 → 改为 max(0, raw)）
  - 加第二轮全局重分配：当前单遍聚类后遍历所有 assignment，对孤篇尝试重匹配到已有簇

优先级 中：
  - 时间窗口改为可配置的相对值（如 max_gap = 事件已有时间跨度的 2 倍）
  - 启用 create_threshold 过滤低置信度新簇
```

## 二、传播路径图

### 2.1 当前实现

`propagation/builder.py` `build_propagation_graph()` — 按时间排序的文章 → 标题+正文 bigram 重叠 + 作者引用匹配 → 推理边。

### 2.2 实际表现

| 指标 | 数值 |
|------|:---:|
| 节点 | 34 |
| 边 | 2（阈值 0.15 后） |
| 推理边占比 | 100% |

### 2.3 问题

1. **`shared=sim` 重复变量**：`scorer.py:7-8` 中 `sim=similarity(...)` 后 `shared=sim`，两行定义的是同一个值。`inferred_score` 公式中 `0.35*sim + 0.25*shared = 0.60*sim`。60% 权重来自同一份 bigram 重叠。

2. **Bigram 在中文文本上区分度极低**：两个完全无关的关于"四川地震"和"四川暴雨"的文章，仅靠"四川"的 bigram 就能拿到 >0.15 的分数。作者名匹配（`source_score` 0.15 权重）用 `any(x in author for x in sources)`，正则提取的子串几乎不会精确匹配作者名。

3. **`get_propagation_data()` 有 140 行死代码**：`event_service.py:416-432` 调用 `build_propagation_graph()` 后立即 `return`，后面的 434-577 行（基于 `_title_bigrams` 的旧实现，`_SIMILARITY_MIN=0.05`）永远不可达。

4. **34 节点仅 2 条边**：对本事件无显式转发关系（百度新闻、B站视频、知乎问答之间无转发链），推理边依赖纯文本相似，阈值微调效果有限。这个图本质上不适合渲染此类数据。

### 2.4 改进建议

```
优先级 高：
  - 删除 scorer.py 中 shared=sim 的重复定义
  - 删除 event_service.py 434-577 行死代码
  - 改用关键词共现网络代替文章级传播图（关键词节点 + 共现边 → 更有分析价值）

优先级 中：
  - 作者名匹配改为子串包含匹配（而非精确相等）
  - 增加显式引用检测（如微博"//@某某"转发格式）
```

## 三、风险评估

### 3.1 当前实现

`fake_detector.py` `assess_suspicious_risk()` — 规则打分，基线 25 分，6 项加减因子，clamp 到 [0, 100]，≥40 分为可疑。

### 3.2 实际表现

| 指标 | 数值 |
|------|:---:|
| 评估文章 | 34 篇 |
| 平均风险分 | 9.1 |
| LLM 辅助评估 | 1 篇（≥40 分） |
| 风险因子命中 | 主要来自"单平台惩罚" |

### 3.3 问题

1. **所有权重和基线无实证校准**：基线 25、+15（匿名作者）、+15（单平台）、+10（煽动词）、-20（官媒）——全部是整数，没有任何标注数据或校准实验。启动基线从 0 和从 25 得出的是完全不同的分类结果。

2. **单平台惩罚作用于事件级而非文章级**：`ctx["platforms"]` 是整个事件的文章平台集合。如果事件只被微博报道，所有文章都被扣 15 分。这是双计数——同一个信号既影响了事件级风险又影响了文章级风险。

3. **标题正文一致性检查过于简陋**：`_title_content_consistency_low()` 仅计算中文字符集合重叠率 < 0.15。短标题（"台风巴威登陆" 5 个字）与长正文（500 字）的重叠率天然极低，几乎所有短标题文章都会被误判。

4. **LLM 仅对 ≥40 分调用**：本事件 34 篇文章中仅 1 篇触发。风险信号最弱的文章（25-39 分）完全跳过 LLM 验证，而这些恰恰是最需要辅助判断的灰色地带。

### 3.4 改进建议

```
优先级 高：
  - 将单平台惩罚从事件级改为文章级（检查文章自身是否被多平台转载）
  - 标题-正文一致性改用语义相似度（BGE cosine）而非字符集重叠

优先级 中：
  - 基于标注数据重新校准权重（至少做一次内部评估）
  - 将 LLM 调用扩展至 ≥25 分的所有文章（灰色地带最需要 LLM 判断）
  - 每个因子的置信度范围用连续值替代二元判定（如"作者可信度"用粉丝数/认证状态分档）
```

## 四、生命周期预测

### 4.1 当前实现

`trend_predictor.py` `predict_lifecycle_stage()` — window=3 滑动平均平滑 → 阈值判断四阶段。

### 4.2 实际表现

| 事件 | 数据点 | 峰值 | 总量 | 预测结果 |
|------|:---:|:---:|:---:|------|
| #1 (34篇) | 8 | 15 | 33 | 潜伏期（修复前）→ 成长期（修复后） |
| #2 (13篇) | 0 | 0 | 0 | 潜伏期 |
| #3 (1篇) | 0 | 0 | 0 | 潜伏期 |

### 4.3 问题

1. **小数据集不适用**：window=3 滑动平均对 3 个数据点仅产生 1 个平滑值。`_has_sustained_decline` 需要 4+ 个点否则永远 False。算法假设数据密度 ≥ 每日 1 点 × 7 天，实际数据可能只有 3-8 个快照。

2. **每次页面浏览重新计算并持久化**：`get_event_detail()` 每次调用 `predict_lifecycle_stage()`，结果不同就覆写到 DB。生命周期阶段可能随数据增加来回跳跃（"成长期"→ 因为新数据拉低峰值 → 变回"潜伏期"）。

3. **全硬编码阈值无置信区间**：`LATENT_MAX_DAILY=3`、`PEAK_TO_GROWTH_RATIO=3.0`、`DECLINE_FROM_PEAK=0.50`。对于日均 500 篇的大事件和日均 5 篇的小事件使用同一套绝对阈值。

### 4.4 改进建议

```
优先级 高：
  - 对小数据集（< 7 点）返回"数据积累中"，不强行分类
  - 生命周期阶段改为单调推进（潜伏→成长→高潮→消退，不可逆）

优先级 中：
  - 阈值根据数据规模自适应（基于 max(counts) 而非绝对值）
  - 添加置信度输出："成长期 (置信度: 0.65)"
```

## 五、关键词提取与词云

### 5.1 当前实现

`llm_keywords.py` LLM 多维度提取 → `_event_keywords()` 事件级聚合（查询词过滤 + 排名衰减 + 情感多数投票）。

### 5.2 实际表现

| 指标 | 修复前 (TF-IDF) | 修复后 (LLM) |
|------|------|------|
| 关键词质量 | "一风""全力打""树木被" | "超强台风""防汛救灾""应急救援" |
| 权重梯度 | 2-4 级 | 30 级 |
| 情感标注 | 无 | positive/negative/neutral |
| 实体类型 | 无 | location/organization/person/event/concept |
| LLM 成功率 | — | 28/48 (58%) |

### 5.3 问题

1. **LLM 覆盖率仅 58%**：20 篇文章回退 TF-IDF，产生 sentiment=neutral 的空洞。回退词的 entity_type 固定为 "concept"。

2. **查询词过滤阈值 50% 可能过滤掉真实主题**：如果事件标题恰好是核心关键词且被 LLM 标注为 importance=1.0 的词，会被误杀。

3. **无同义词合并**："台风""热带风暴""气旋"是三个不同的关键词。

4. **`_ai_generate_summary()` 传参 bug**：`event_aggregation_service.py:453` 传 `articles[0]`（Article 对象）给 `_event_keywords()`（期望 Event 对象）。

### 5.4 改进建议

```
优先级 高：
  - 修复 _ai_generate_summary 传参 bug
  - LLM 失败时对回退词也调用一次批量 LLM（而非逐篇），提高覆盖率至 >90%

优先级 中：
  - 查询词过滤改为语义过滤（检查该词是否出现在所有事件中而非所有文章中）
  - 添加同义词映射表（至少覆盖常见气象/灾害术语）
```

## 六、情感分析

### 6.1 当前实现

LLM 主路径（`sentiment_analyzer.py`）→ SnowNLP fallback。stance/impact/factual 三维分类。

### 6.2 实际表现

| 指标 | 数值 |
|------|:---:|
| LLM 成功率 | 100% (48/48) |
| SnowNLP fallback 触发 | 0 次 |
| 情感分布 (#1) | 正面4 (12%), 中立22 (65%), 负面8 (24%) |

### 6.3 问题

1. **SnowNLP 领域不匹配**：SnowNLP 基于电商评论训练，"这个商品不错"和"这次救援不错"的情感表达完全不同。当 LLM 不可用时，新闻文本的情感分析质量会断崖式下降。

2. **aggregator 的 spread_factor 可能过度加权**：`1 + log1p(article_heat) / log1p(event_max_heat)` 给最高热度文章 2x 权重。如果最高热度文章恰好是情感极端值，会倾斜整个事件的情感分布。

3. **`_generate_sentiment_summary()` 是独立的 LLM 调用**：用一个 LLM 生成情感标签，再用另一个 LLM 总结标签——两次调用可能产生不一致。

### 6.4 改进建议

```
优先级 中：
  - SnowNLP 替换为基于新闻语料微调的模型（如 BERT 中文情感分类）
  - spread_factor 上限从 2.0 降至 1.5
  - 情感总结集成到同一 LLM 调用中（让 extraction 直接输出一句总结）
```

## 七、热度计算

### 7.1 当前实现

双层：正式公式 (`heat_calculator.py`) vs. 基础 fallback (`event_aggregation_service.py:91-93`)。

正式公式：百分位排名 × 指数时间衰减 × 增长速率 × 平台覆盖。  
Fallback：`article_count × 2.0 × (1 + platform_count × 0.3)`。

### 7.2 实际表现

| 事件 | 文章数 | 平台数 | heat (fallback) |
|------|:---:|:---:|:---:|
| #1 | 34 | 4 | 100.0 |
| #2 | 13 | 1 | 33.8 |
| #3 | 1 | 1 | 2.6 |

### 7.3 问题

1. **正式公式从未被调用**：搜索模式不走 `HotspotRun` → 不触发 `calculate_event_heats()` → 只走 fallback。正式公式的百分位排名 + 指数衰减设计实际上是死代码。

2. **Fallback 公式无时间衰减**：34 篇文章不管是一天爬的还是一个月爬的，heat 都是 100。事件应该是渐冷的。

3. **`spread_heat` 永远为 None**：因为 fallback 不计算互动得分。

### 7.4 改进建议

```
优先级 高：
  - 搜索模式 publish 后也调用热度计算（至少跑一次基础版，含时间衰减）
  - fallback 公式加入简单时间衰减：heat *= max(0.2, 1 - hours_since_first / (24*7))

优先级 中：
  - 正式公式的 spread_raw 维度不一致问题（序数排名数据 vs 连续互动数据混合）
```

## 八、AI 元数据提取

### 8.1 当前实现

`_extract_event_metadata()` — 一次 LLM 调用提取 time_code/location/key_figures/cause。

### 8.2 实际表现

| 字段 | Event #1 结果 |
|------|------|
| time_code | "2023年（具体日期未统一）" |
| location | "浙江、山东、上海、深圳、福建等地" |
| key_figures | "气象部门、媒体记者、应急人员、普通市民" |
| cause | "台风巴威外围影响及登陆带来的强风暴雨..." |

### 8.3 问题

1. **time_code 不准**：LLM 无法从文章中推断精确时间时，返回模糊描述"2023年"。

2. **无缓存/持久化机制**：每次 `get_event_detail()` 都重新调用 LLM。页面刷新 10 次 = 10 次 LLM 调用。

3. **`EVENT_SUMMARY_PROMPT` 过于简陋**：提示词仅一句话"提取时间、地点、起因、人物和概述"，无输出格式约束，无 few-shot。

### 8.4 改进建议

```
优先级 高：
  - 提取结果持久化到 Event 模型（time_code/location/key_figures/cause 字段已存在）
  - 只首次计算，后续直接从 DB 读

优先级 中：
  - time_code 回退到 event.first_publish_time（确定性数据源）
  - 丰富 EVENT_SUMMARY_PROMPT，加 few-shot 示例
```

## 十、总结

11 个模块中：
- **3 个稳定**（平台分布 5/5，情感 4/5，热度 4/5）
- **5 个需修缮**（趋势/关键词/AI标题摘要/聚类/聚合 3/5）
- **3 个需重建**（传播图 1/5，风险评估 2/5，生命周期 2/5）

核心模式：**算法设计思路正确，但具体参数全部缺乏实证校准**——所有权重、阈值、基线都是整数拍脑袋，没有一个来自标注数据或 A/B 实验。在数据量小（<100 篇/事件）时，统计方法（滑动平均、TF-IDF、bigram 重叠）比大模型方法更容易暴露脆弱性。

## 十一、不同舆情事件类型的适用性分析

当前审计基于「台风巴威」（自然灾害类），以下分析各模块在不同事件类型下的表现差异。

### 11.1 事件类型矩阵

| 事件类型 | 典型特征 | 数据特点 | 最脆弱的模块 |
|------|------|------|------|
| 🌀 自然灾害 | 突发、地点明确、官媒介入快 | 报道集中、多平台转载、互动量两极分化 | 生命周期（爆发急）、传播图（转发链短） |
| 🏛️ 政治事件 | 措辞谨慎、信源集中、长周期 | 报道分散但持续、情感中性居多 | 情感分析（中性比例极高）、风险评估（官媒-20 误杀） |
| 💰 经济事件 | 术语专业、数据密集、涉及多方 | 专业词汇多、正负面取决于立场 | 关键词提取（专业术语分词难）、情感（立场依赖上下文） |
| 🏥 公共卫生 | 科学性要求高、谣言多、持续数月 | 每日更新、专业机构发声、辟谣频繁 | 生命周期（持续期远超 30 天）、风险评估（科学谣言检测难） |
| 👥 社会事件 | 情感极端、引爆点随机、传播快 | 碎片化信息、用户生成内容为主 | 传播图（信息源分散无转发链）、关键词（口语化/网络用语） |
| 🎬 娱乐事件 | 话题周期短、粉丝驱动、平台集中 | 集中在微博/小红书、互动量极端 | 聚类（同质化内容多）、热度（互动量虚高） |

### 11.2 各模块的领域敏感性

| 模块 | 领域敏感度 | 说明 |
|------|:---:|------|
| **热度计算** | 🟢 低 | 百分位排名体系本身领域无关，但互动量权重对不同事件类型应有不同（娱乐事件互动量虚高） |
| **情感分析** | 🟡 中 | LLM 主路径通用；SnowNLP fallback 在政治/经济领域更差（电商→政治的距离 > 电商→灾难的距离） |
| **关键词** | 🟡 中 | LLM 提示词领域无关 ✅，但专业术语（经济/医疗）在分词阶段就碎了，LLM 也救不回来 |
| **生命周期** | 🔴 高 | 自然灾害 3 天高峰，公共卫生 3 个月平台期。同套绝对阈值全灭 |
| **风险评估** | 🔴 高 | 煽动词列表（震惊/炸裂/突发）只覆盖自媒体风格，不适用于专业报道中的隐性误导。官媒-20 对政治事件等于免疫 |
| **传播图** | 🔴 高 | 自然灾害有清晰源头（气象局），社会事件可能无明确起源。bigram 在不同文体（新闻 vs 微博 vs 知乎）上的区分度天差地别 |
| **事件聚合** | 🟢 低 | BGE 语义相似度领域无关；实体冲突检测（location/case/organization）对所有事件类型有效 |

### 11.3 当前测试数据局限

本次仅用自然灾害类事件（台风巴威）验证，以下场景未经测试：

## 十二、事件分类 + 领域自适应算法：一个系统性的解决方向

### 12.1 核心思路

第十一章暴露了一个根本矛盾：**所有事件类型共用一套算法参数，但不同领域对"准确"的定义完全不同。**

公共卫生事件持续 3 个月是常态，自然灾害 3 天就过峰——用同一个生命周期窗口，总有一个是错的。政治事件中"官方媒体"是核心信源，社会事件中"官方媒体"可能只是背景音——用同一个风险减免规则，总有一个是不合理的。

于是自然引出一个设计方向：**先分类，再适配。** 不是为每种事件类型重写算法，而是让同一套算法骨架在不同类别下加载不同的参数配置。

```
输入事件 → LLM 分类（自然灾害/公共卫生/社会民生/政治政策/经济市场/科技网络）
                ↓
         加载对应类的参数表
                ↓
    ┌───────────┼───────────┐
    ↓           ↓           ↓
 生命周期    风险评估    热度计算   ...
（自适应窗口）（差异权重）（差异衰减）
```

### 12.2 需要适配的参数维度

不是所有参数都需要差异化。按领域敏感性分层：

**高敏感——必须区分类别：**

- **生命周期时间窗口**：公共卫生类的事件周期天然比自然灾害长一个数量级。如果窗口固定，要么自然灾害永远不进入消退期，要么公共卫生刚起步就被判为消退。
- **风险评估的官媒权重**：政治事件中官方媒体是核心参与者而非外部验证者，"官媒介入 -20 分"的规则在此类事件中不成立。社会事件中官媒可能延迟介入，经济事件中官方公告本身就是事件的一部分。各类别中"官方来源"的语义不同，减免幅度应不同。
- **风险评估的煽动信号**：自然灾害中的"紧急通知"是正常预警，社会事件中的"震惊""炸裂"是自媒体夸张。同一些词在不同类别中的虚假信号权重应该不同。
- **传播图的边推理阈值**：微博为主的社交事件标题短、互动多、转发链清晰；知乎/百度为主的政策讨论标题长、转发链缺失。不同平台的文体差异导致 bigram 重叠的基线不同，阈值应随类别调整。

**中敏感——区分类别有益但不阻塞：**

- **热度计算的互动权重**：娱乐事件的转评赞数量级碾压自然灾害，如果同等计入热度，台风永远比不过明星绯闻。互动量贡献需要按类别压低或抬高。
- **情感分析的中性基线**：政治事件情感分布天然偏中性（通报文体），社会事件情感天然偏极化。当某类事件的情感分布显著偏离基线时，这个偏差本身就是分析信号，不应被归一化掩盖。
- **关键词分词词典**：经济术语（"量化宽松""去杠杆"）和医疗术语（"核酸检测""无症状感染者"）在通用分词器下会被切碎。分类后可以加载领域词典，提升关键词质量。

**低敏感——通用即可：**

- **BGE 语义向量**：语义空间是跨领域的，不需要区分类别。
- **平台分布统计**：纯统计，无算法参数。
- **事件聚合的实体冲突检测**：location/case/organization 冲突在所有类别中都有效。

### 12.3 需要连带考虑的问题

这个方向存在几个不能回避的连锁问题：

**分类错误会级联放大**。如果分类器把经济事件错判为政治事件，后续加载的官媒权重、生命周期窗口、煽动词表全部错位。分类准确率需要达到什么水平才能让自适应参数的收益大于分类错误的代价？这需要设计一个"分类置信度阈值"：当 LLM 对分类不确定（如 confidence < 0.7）时，回退到通用默认参数。

**边界事件如何处理**。"疫情导致供应链断裂引发企业倒闭"——是公共卫生、经济事件还是社会事件？单一分类可能丢失重要维度。需要支持主导类别 + 辅助标签，还是接受单标签的精度损失？

**人工覆盖入口**。自动分类永远有 5-10% 的错误率。系统需要提供一个入口让运营人员修正事件分类——以及修正后自动触发相关模块的重新计算（如改了分类 → 重新用新参数跑风险评估和生命周期）。

**参数从哪里来**。分类体系的价值取决于参数是否比当前"一刀切"更优。如果新参数也是拍脑袋的 6 组整数，只是把一份未校准变成了六份未校准。理想的校准来源是：（a）对每类事件采集 2-3 个真实案例的完整数据，观察各模块在不同参数下的表现；（b）邀请 1-2 个领域专家做定性判断作为参照。

**对前端的影响**。分类信息对用户有直接价值——看板可以按类别筛选、事件卡片可以显示类别标签、AI 研判摘要可以包含类别上下文。前端不需要大改，但需要预留展示位置。

### 12.4 与现有系统的关系

这个方案是纯增量改动，不需要重写任何模块。现有的 11 个算法函数保持结构不变，只是在函数入口处根据事件类别加载不同的参数表。今天完全没有分类的情况下系统也在运行——分类只是让它在每个领域里跑得更准。

`topic_classifier.py` 已有 LDA + LLM 主题分类的完整链路，目前输出的是细粒度的主题名（如"公共卫生""自然灾害"），恰好可以映射到 6 个大类。这个现有能力是方案可行的关键前提。```
