# 数据清洗与预处理模块设计方案

## 1. 模块定位

数据清洗与预处理模块位于爬虫/样例导入之后、TF-IDF、主题建模、事件聚合和情感分析之前，负责将不同平台的原始数据转换成可追溯、可比较、可重复处理的规范数据。

```text
RawDocument
→ 原始数据入库
→ preprocessing pipeline
→ CleanDocument + DocumentFeatures
→ TF-IDF / LDA / 事件聚合 / 情感分析 / 报告生成
```

第一版采用本地规则和本地模型完成高频预处理，不使用 LLM 执行全量正文提取、全量分词或全量去重。

### 1.1 已确认原则

1. `raw_content` 和 `raw_json` 保留，不被清洗模块覆盖。
2. 低质量、广告和重复文章原则上只标记，不直接删除。
3. 文本质量、NLP 权重、传播热度、来源可信度分别计算。
4. 所有算法模块只接收参数并返回结果，数据库写入由 `pipeline.py` 统一完成。
5. 耗时处理由 APScheduler + MySQL `task` 表在后台执行，HTTP API 只创建任务并返回 `task_id`。
6. 阈值写入配置并记录版本，爬虫产生真实数据后通过测试集调参。

---

## 2. 模块划分

```text
preprocessing/
├── normalizer.py       # 平台字段、URL、时间、编码和空值标准化
├── extractor.py        # 正文提取和四级降级
├── cleaner.py          # 正文去噪和平台标记处理
├── quality.py          # 文本可分析性评分和广告标记
├── deduplicator.py     # URL、哈希、SimHash、BGE 分层去重
├── segmenter.py        # jieba 分词和多用途 token 输出
└── pipeline.py         # 顺序编排、状态、事务、日志和后台任务衔接
```

模块间不互相写数据库，也不直接调用下一个模块。统一由 `pipeline.py` 按顺序调度。

---

## 3. 完整处理流程

```text
article.raw_content / raw_json
→ normalizer：元数据标准化
→ deduplicator：平台内容 ID / URL 初步去重
→ extractor：正文提取
→ cleaner：正文去噪
→ quality：质量评分和广告标记
→ deduplicator：正文精确哈希
→ segmenter：jieba 分词
→ deduplicator：Jaccard / SimHash / 可选 BGE 边界判断
→ pipeline：保存结果和处理日志
→ 批次完成后触发 TF-IDF / LDA / 事件聚合
```

同一篇文章的基础预处理按顺序执行；BGE 等成本较高的步骤可以异步补充，不阻塞基础清洗成功。

## 4. normalizer.py 字段标准化

`normalizer.py` 只统一数据表示，不删除正文语义。

| 字段 | 统一规则 |
|------|----------|
| `platform` | 平台别名映射为固定代码，如 `weibo`、`bilibili`、`baidu_news`、`rss`、`sample` |
| `source_type` | 统一为 `news/social/hotlist/rss/sample` |
| `source_article_id` | 平台原始内容 ID 转为字符串，保留前导零 |
| `url` | 去空白、fragment 和已知跟踪参数，统一协议和域名大小写 |
| `url_hash` | 对规范化 URL 计算 SHA-256，用于唯一索引和 URL 去重 |
| `title` | HTML 实体解码、Unicode NFKC、去零宽字符、合并空白 |
| `raw_content` | 确保为 UTF-8 字符串，不删除 HTML 和原始语义 |
| `content_type` | 统一为 `html/text/json` |
| `publish_time` | 解析平台格式，按来源时区解释后统一以 UTC 约定存储 |
| `author/author_id` | 清理控制字符；平台 ID 统一为字符串 |
| `author_type` | `individual/media/government/organization/unknown` |
| `author_verified` | `true/false/NULL`，未知不能当作 false |
| 互动指标 | 支持 `1.2万`、`2.5k`、千分位等格式，统一为非负整数 |
| `language` | `zh-CN/zh-TW/en/unknown` |
| `crawled_at` | 使用服务器时间记录实际采集时刻 |
| `raw_json` | 转为可写入 MySQL JSON 的结构，保留平台原始字段 |

### 4.1 空值和时间规则

- 未知互动量存 `NULL`，不使用 `0` 冒充“确定没有互动”。
- 未知作者存 `NULL`，不写入“未知”占位字符串。
- 时间解析失败时保留原始值并记录 `NORMALIZE_TIME_PARSE_FAILED`，不得使用 `crawled_at` 冒充 `publish_time`。
- 国内平台缺少时区时默认按 `Asia/Shanghai` 解释，并记录 `timezone_assumed`。
- 只有 `platform`、URL/平台内容 ID、原始内容等关键字段完全无效时才阻止后续处理。

### 4.2 输出结构

```json
{
  "normalized_data": {},
  "warnings": [],
  "errors": [],
  "normalize_version": "v1"
}
```

## 5. extractor.py 正文提取

结构化平台已经返回纯文本时直接使用，不进入 HTML 提取链。HTML 内容按以下四级顺序执行：

| 优先级 | 方法 | 适用场景 |
|--------|------|----------|
| 1 | trafilatura | 新闻网页，自动识别正文和部分元数据 |
| 2 | readability-lxml | 论坛、博客和非标准页面 |
| 3 | BeautifulSoup + 平台 CSS 选择器 | 前两种失败时使用平台规则提取 |
| 4 | 网页全部可见文字 fallback | 尽量保留数据，标记降级提取 |

每一级返回候选正文后都必须通过基础有效性检查，不能以“库没有报错”作为成功标准。

### 5.1 基础有效性检查

- 返回值非空且没有解析异常。
- 新闻正文通常不少于 100 个有效字符，社交短文本通常不少于 20 个有效字符；最终阈值按真实数据调整。
- 结果不能只包含标题、导航、版权、登录提示或按钮文字。
- 不能是验证码、404、访问异常、安全验证等拦截页面。
- 有效中文、英文和数字比例不能过低。
- 菜单、链接和模板文字不能占内容主体。

第四级成功时：

```text
extraction_method = fallback
extraction_degraded = true
status = degraded
```

四级全部无效时：

```text
clean_status = failed
clean_error = EXTRACT_ALL_METHODS_FAILED
```

原始字段继续保留，便于规则升级后重新处理。

## 6. cleaner.py 六步清洗

`cleaner.py` 删除网页噪声，但尽量不破坏舆情语义。

1. 清除 HTML/XML 标签残留、脚本、样式、注释和 HTML 实体。
2. 统一正文中的全角/半角、Unicode、换行格式和不可见控制字符。
3. 规范空白与段落，保留正常段落结构，不把全文压成一行。
4. 删除明确的网页模板噪声，如“打开 APP 阅读全文”“上一篇/下一篇”“相关阅读”等。
5. 提取并保留 `@提及`、`#话题#`、平台表情、emoji 和 URL 的语义信息。
6. 输出清洗正文、结构化特征、删除记录、统计信息和警告。

繁体转简体只用于分析 token，不覆盖展示和溯源使用的 `clean_content`。不能直接删除所有标点、表情、话题或重复字符，因为它们可能表达情绪。

### 6.1 通用规则与平台规则

```text
通用规则：HTML 残留、控制字符、空白、通用网页模板
平台规则：平台专用 CSS、固定尾注、话题和表情格式
```

平台专用规则独立配置，避免一个大正则误删所有平台的有效正文。

### 6.2 输出结构

```json
{
  "clean_content": "清洗后的正文",
  "mentions": [],
  "topics": [],
  "urls": [],
  "emotions": [],
  "removed_lines": [],
  "warnings": [],
  "statistics": {
    "original_length": 0,
    "clean_length": 0,
    "removed_length": 0
  }
}
```

## 7. quality.py 文本质量评估

第一版不使用困惑度模型，不调用 LLM，不因质量分低而直接删除文章。

```text
quality_score =
  0.20 × 有效长度
+ 0.20 × 有效字符比例
+ 0.20 × 低噪声程度
+ 0.15 × 低重复程度
+ 0.15 × 字段完整性
+ 0.10 × 正文提取可信度
```

不同数据类型使用不同长度标准：

| 类型 | 建议最低有效长度 | 理想长度 |
|------|------------------|----------|
| 新闻正文 | 80 字 | 200 字以上 |
| 社交帖子 | 10 字 | 30 字以上 |
| 热搜/热榜标题 | 5 字 | 标题完整即可 |
| 评论 | 5 字 | 15 字以上 |

### 7.1 质量等级与 NLP 权重

| quality_score | 等级 | nlp_weight |
|---------------|------|------------|
| `>= 0.75` | good | 1.0 |
| `>= 0.50` | usable | 0.8 |
| `>= 0.30` | low | 0.5 |
| `< 0.30` | very_low | 0.2 |
| 处理失败 | failed | 0 |

`quality_score/nlp_weight` 只影响 TF-IDF、LDA、聚类、关键词和摘要材料选择，不直接乘入传播热度。

传播热度单独计算：

```text
heat_contribution =
  engagement_score
  × time_decay
  × duplicate_weight
  × spam_weight
```

来源可信程度另存 `source_reliability_score`，用于可疑信息风险评估，不混入文本质量。

### 7.2 广告和质量标签

质量标签包括：

```text
too_short / high_noise / high_repetition / missing_publish_time
missing_author / extraction_degraded / possible_garbled_text / blocked_page
```

广告识别独立输出：

```text
is_advertisement
advertisement_score
advertisement_reasons
```

广告、软文和垃圾文本默认保留，但降低 `spam_weight`。

## 8. deduplicator.py 分层去重

去重与事件聚合严格分开：

- 去重判断同一内容是否被重复采集、复制或转载。
- 事件聚合判断不同报道是否在讨论同一事件。

### 8.1 四层漏斗

| 层次 | 方法 | 初始规则 | 处理 |
|------|------|----------|------|
| 1 | `platform + source_article_id` / `url_hash` | 完全匹配 | 更新已有文章和互动快照 |
| 2 | `content_hash`（SHA-256） | 完全匹配 | 标记精确内容重复 |
| 3 | 标题 Jaccard + 正文 SimHash | Jaccard `>= 0.85` 且海明距离 `<= 3` | 自动判定近似重复 |
| 4 | BGE 余弦相似度 | 仅处理第三层边界候选 | 异步判断，不阻塞基础流程 |

第三层只满足一个条件时进入 BGE；标题少于 10 个有效字符时不能仅靠 Jaccard 自动去重。

### 8.2 BGE 初始阈值

```text
有 Jaccard/SimHash 证据且 BGE >= 0.97 → 自动重复
仅有 BGE 证据且 BGE >= 0.985       → 自动重复
BGE 0.90～0.97                     → 同事件候选，不作为重复删除
BGE < 0.90                         → 不作为重复依据
```

以上为保守初始值。爬虫稳定后建立 `duplicate/same_event/unrelated` 标注集，以重复判断精确率至少 98% 为目标重新标定。

### 8.3 重复数据保留和代表文章

重复记录不删除，增加：

```text
is_duplicate / duplicate_of_id / duplicate_group_id
duplicate_method / duplicate_score / content_hash / simhash / dedup_version
```

同一重复组的代表文章按以下顺序选择：

1. `quality_score` 更高。
2. 正文提取不是 fallback。
3. 标题、时间、作者等字段更完整。
4. 发布时间更早。

TF-IDF、LDA 和摘要只使用代表文章；跨平台分布和传播分析保留全部记录；热度中跨平台转载降低权重但不完全记零。

BGE 不可用时继续使用前三层，并标记 `dedup_pending`。

## 9. segmenter.py 分词

使用 jieba 精确模式，配合基础词典、平台词典、批次动态词典和停用词表。

```text
clean_content
→ 分析专用字符标准化
→ jieba.cut()
→ 合并特殊特征
→ 按用途过滤停用词
→ 输出多套 token
```

输出：

```json
{
  "tokens": [],
  "tfidf_tokens": [],
  "sentiment_tokens": [],
  "topics": [],
  "mentions": [],
  "segment_version": "v1"
}
```

- `tokens` 保存完整可追溯分词结果。
- `tfidf_tokens` 删除普通停用词，用于 TF-IDF、LDA、事件聚合和关键词。
- `sentiment_tokens` 保留“不、没、非常、过于”等否定词和程度词。
- 英文转小写；年份、金额、地名编号等有区分作用的数字保留。
- 繁体转简体只作用于分析 token。
- 同一分析批次使用固定词典版本，不能处理中途动态改变。
- jieba 不可用时使用简单分词 fallback 并记录 `SEGMENT_JIEBA_ERROR`。

分词结果建议存入 `document_features`，不继续向 `article` 堆叠大量分析字段。

## 10. pipeline.py 编排和任务

爬虫或样例导入完成后：

```text
写入 article 和首个 article_snapshot
→ clean_status = pending
→ 创建 preprocess task
→ API 返回 task_id
→ APScheduler 后台领取任务
```

`article.clean_status` 保持：

```text
pending / success / failed
```

具体执行阶段和进度放在 `task` 与 `processing_log` 中。建议批次大小为 50 篇；单篇失败不影响同批其他文章；批次基础预处理完成后统一触发 TF-IDF。

### 10.1 幂等和并发

- 幂等键为 `article_id + preprocess_version`。
- 相同版本已经成功时直接跳过。
- 规则升级时增加 `preprocess_version`，基于原始数据重新处理。
- MySQL 8 使用 `SELECT ... FOR UPDATE SKIP LOCKED` 领取任务，避免多个执行器处理同一文章。
- 每篇文章完成基础处理后提交事务，BGE 等高成本步骤单独异步更新。

## 11. 文章快照

`article` 表保存文章当前规范化状态；`article_snapshot` 保存每次采集观察到的内容和互动变化。

### 11.1 保存规则

```text
第一次采集
→ 保存完整 raw_content/raw_json 和互动量

正文 content_hash 变化
→ 保存完整新正文并触发重新预处理

正文未变化，只有互动量变化
→ raw_content 可为 NULL，只保存互动指标和 crawled_at
```

普通文章每小时最多保存一次互动快照；正文变化和采集状态变化不受该限制。

快照与重复关系不同：

```text
article_snapshot → 同一文章在不同时间的观测
duplicate_of_id  → 不同 URL 之间的复制或转载关系
```

## 12. 失败分类和处理日志

统一处理阶段：

```text
crawl / normalize / extract / clean / quality
deduplicate / segment / persist
```

每个模块返回：

```text
success  → 正常完成
degraded → 使用 fallback 或跳过非必要能力，结果仍可使用
failed   → 无法产生下游所需结果
```

### 12.1 典型错误码

| 阶段 | 错误码示例 |
|------|------------|
| crawl | `CRAWL_TIMEOUT`、`CRAWL_HTTP_403`、`CRAWL_CAPTCHA`、`CRAWL_EMPTY_RESPONSE` |
| normalize | `NORMALIZE_MISSING_REQUIRED_FIELD`、`NORMALIZE_INVALID_URL`、`NORMALIZE_TIME_PARSE_FAILED` |
| extract | `EXTRACT_ALL_METHODS_FAILED`、`EXTRACT_PARSER_ERROR`、`EXTRACT_FALLBACK_USED` |
| clean | `CLEAN_RULE_ERROR`、`CLEAN_EMPTY_AFTER_CLEANING` |
| quality | `QUALITY_EVALUATION_ERROR`；一般质量问题进入 `quality_flags` |
| deduplicate | `DEDUP_INDEX_ERROR`、`DEDUP_BGE_UNAVAILABLE` |
| segment | `SEGMENT_JIEBA_ERROR`、`SEGMENT_DICTIONARY_ERROR` |
| persist | `PERSIST_DEADLOCK`、`PERSIST_UNIQUE_CONFLICT`、`PERSIST_TRANSACTION_FAILED` |

网络超时和数据库死锁最多重试 3 次；验证码和必填字段缺失不立即重试；解析器失败进入下一层 extractor；BGE/jieba 等可降级能力不阻塞主流程。

总任务状态放在 `task`，每篇文章、每个阶段的结果放在 `processing_log`。技术异常堆栈仅写后台日志，不返回普通前端用户。

## 14. 测试集和阈值标定

爬虫产出真实数据后建立两类固定测试集。

### 14.1 清洗回归集

- 新闻 HTML、社交帖子、热搜标题和异常页面共 30～50 条。
- 使用 `must_include/must_exclude` 验证正文，不要求清洗结果逐字相等。
- 我方 LLM 可以完成抽样、初标、测试文件和指标计算；人工只复核边界样本和随机抽查 10%～20%。

### 14.2 去重标注集

```text
duplicate  50 对
same_event 50 对
unrelated  50 对
```

80% 用于调参，20% 作为最终测试集。最终测试集在阈值确定前不参与调整。报告中注明“LLM 辅助标注，人工复核边界样本”。

建议验收指标：

```text
正文提取成功率 >= 95%
关键正文误删率 <= 2%
重复判断精确率 >= 98%
时间解析成功率 >= 95%
相同输入和版本重复执行结果一致
```
