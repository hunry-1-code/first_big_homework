# 数据清洗与预处理模块设计方案

## 1. 模块定位

在爬虫模块输出 `RawDocument` 之后、NLP 分析模块之前，完成数据清洗、标准化、去重和质量控制，输出 `CleanDocument` 供下游使用。

```
RawDocument → 清洗管线 → CleanDocument → 分词/向量化/事件聚合/情感分析...
```

---

## 2. 清洗管线总览

```
RawDocument
  ↓
2.1 HTML正文提取（trafilatura → readability → BS4 三级兜底）
  ↓
2.2 文本去噪（全角/半角统一 → 繁体转简体 → 正则去URL/表情/特殊字符 → 长度过滤）
  ↓
2.3 平台特有标记处理（@提及、#话题#、表情 → 提取为特征）
  ↓
2.4 字段标准化（时间标准化 → URL规范化 → 空值填充 → 互动数字段默认值）
  ↓
2.5 三层去重（URL MD5 → SimHash → bge向量余弦相似度）
  ↓
2.6 质量评估（困惑度 → 信息密度 → 结构完整性 → 广告/软文降权）
  ↓
CleanDocument → 入库
```

---

## 3. 各步骤详细设计

### 3.1 HTML 正文提取

#### 三级兜底策略

| 优先级 | 工具 | 适用场景 |
|--------|------|---------|
| 1 | trafilatura | 新闻网页，自动识别正文+元数据 |
| 2 | readability-lxml | 论坛、博客等非标准网页，容错性更好 |
| 3 | BeautifulSoup 手动选择器 | 前两个都失败时，各平台预配 CSS 选择器兜底 |
| 4 | title + 元描述 + 前500字 | 全部失败，标记 `content_extraction_failed` 不删记录 |

#### 来源

- 技术选型文档推荐 trafilatura / readability-lxml / BeautifulSoup
- 思通舆情（stonedt-yuqing）采用"文本密度算法"自动识别正文区域
- 后端方案文档异常降级表："正文抽取失败：使用标题、摘要和原始片段兜底"

---

### 3.2 文本去噪

| 序号 | 操作 | 方法 | 来源 |
|------|------|------|------|
| 1 | 全角/半角统一 | `str.maketrans()` 映射表，全角字母数字标点→半角 | 中文 NLP 通用实践 |
| 2 | 繁体转简体 | `opencc-python`（Open Chinese Convert） | HaNLP 内置（思通舆情） |
| 3 | 去除 HTML 标签残留 | `BeautifulSoup(text, 'html.parser').get_text()` | 标准做法 |
| 4 | 去除 URL 链接 | `re.sub(r'https?://\S+', '', text)` | BettaFish 清洗管线 |
| 5 | 去除表情符号（微博） | `re.sub(r'\[[一-龥]+\]', '', text)` 匹配 `[笑哭]` | BettaFish |
| 6 | emoji 转文字 | emoji 库 `emoji.demojize()` → `:smile:` → 保留为情感特征 | — |
| 7 | 去除空白/换行 | `text.strip()` + `re.sub(r'\s+', ' ', text)` | 通用做法 |
| 8 | 长度过滤 | 清洗后正文 < 20 字→丢弃 | 思通舆情"对软文、广告文分类并屏蔽" |

#### 旧项目已有实现

`networkPublicOpinionAnalysisSystem/myuntils.py` 第 6-13 行 `handleAndCut()` 函数已实现正则清除标点符号和特殊字符，可作为基础复用。

### 3.3 平台特有标记处理

| 标记类型 | 示例 | 处理策略 |
|---------|------|---------|
| @提及 | `@人民日报` | 提取为人名/机构实体，原文保留 |
| #话题# | `#预制菜进校园#` | 提取为关键词标签，原文保留 |
| 微博表情 | `[笑哭]` `[怒]` | 转为文字标签，可作为情感特征 |
| emoji | 😂🤔😡 | 转为文字 `face_with_tears_of_joy`，作为情感特征 |
| 短链接 | `https://t.cn/xxx` | 先展开再决定保留还是删除 |

#### 来源

- BettaFish 去除了 emoji 和 URL
- MediaCrawler 微博模块保留了 @ 和 # 供后续分析
- 本方案采取"保留原文 + 提取特征"的双轨策略

### 3.4 字段标准化

#### 3.4.1 时间格式标准化

各平台原始时间格式 → 统一 ISO 8601：

| 平台 | 原始格式 | 转换方法 |
|------|---------|---------|
| 微博热搜 | 无 publish_time | 填入抓取时间 |
| 微博搜索 | "X分钟前"、"今天 10:30" | 自写 `parse_weibo_relative_time()` |
| B站 | Unix 时间戳 `1719964800` | `datetime.fromtimestamp()` |
| 知乎 | `edit_time` 字段 | `dateutil.parser.parse()` |
| 小红书 | 需详情接口 | `dateutil.parser.parse()` |
| 百度 | 标准时间字符串 | `dateutil.parser.parse()` |

#### 3.4.2 URL 规范化

```python
from urllib.parse import urlparse, urlunparse
# 去 fragment、去尾部斜杠、统一小写域名
```

#### 3.4.3 空值处理

按爬虫方案第 3 节字段映射表，各平台缺失字段填默认值：

| 字段 | 默认值 | 说明 |
|------|--------|------|
| content | `""` | 热搜无正文，等搜索补全 |
| like_count | `0` | — |
| comment_count | `0` | — |
| repost_count | `0` | — |
| author | `"未知"` | — |

#### 来源

- 时间标准化：后端方案 6.3 节
- URL 规范化：后端方案 6.3 节
- 字段映射 + 空值：爬虫方案第 3 节字段映射表

### 3.5 去重策略（三层）

| 层次 | 方法 | 阈值 | 时机 | 来源 |
|------|------|------|------|------|
| 第一层 | URL MD5 精确去重 | 完全匹配 | 入库前 | 爬虫方案第 4 节 |
| 第二层 | SimHash 标题指纹 + 汉明距离 | ≤ 3 | 入库后 | 爬虫方案第 4 节 + 思通舆情（Clickhouse+海明距离） |
| 第三层 | bge-small-zh 向量 + pgvector 余弦相似度 | > 0.9 | 异步批量（每小时） | 爬虫方案第 4 节 + docx 参考文档 |

#### 漏斗式去重（论文级优化）

参考 ACL 2022 论文 _Deduplicating Training Data Makes Language Models Better_ 的多阶段策略：

```
Step1: 粗筛（hash / n-gram 快速排除不相关对）
Step2: bi-encoder 相似度（bge-small-zh）
Step3: cross-encoder 精排（可选，精度最高但慢）
```

第一版用三层去重即可满足需求，论文级漏斗作为答辩时"可扩展方向"提及。

#### 相关学术参考

- _Deduplicating Training Data Makes Language Models Better_（ACL 2022）
- _Noise-Robust De-Duplication at Scale_（2022）

### 3.6 文本质量评估

#### 3.6.1 困惑度过滤（Perplexity）

用语言模型（如 GPT-2 中文版或开源小模型）对文本打分，"不自然"的文本（乱码、拼接内容、机器生成垃圾）困惑度异常高，直接过滤。

- 阈值：PPL > 1000 → 丢弃
- 来源：docx 参考文档"困惑度过滤"

#### 3.6.2 质量评分（不止于通过/丢弃）

```text
quality_score =
  0.30 × (1 - PPL/1000)      # 语言模型困惑度
+ 0.25 × 信息密度             # 有效汉字 / 总字符数
+ 0.20 × 结构完整性           # 是否有标题+正文
+ 0.15 × (1 - 重复字符比)     # 防御"啊啊啊啊啊"
+ 0.10 × 来源可信度           # 官方媒体 vs 匿名来源
```

低分文档不丢弃，但在事件聚合和热度计算中降低权重。

#### 3.6.3 广告/软文识别

- 关键词匹配（"点击购买"、"领取优惠"、"限时折扣"等）
- URL 密度过高（> 2 个链接 / 100 字）
- 标题与正文语义相似度过低（标题党特征）
- 命中 → 标记 `is_advertisement = true`，降低聚合权重

#### 来源

- 困惑度：docx 参考文档
- 质量评分：docx 参考文档（冗余度、信息密度）
- 广告识别：思通舆情数据处理文档（"自动分类技术对软文、广告文、敏感文章分类"）

### 3.7 清洗日志与可追溯性

在 `clean_documents` 表中增加 `cleaning_metadata` JSONB 字段，记录每一步的改动：

```json
{
  "html_stripped": true,
  "content_extraction_method": "trafilatura",
  "url_normalized": "https://example.com/post/1",
  "time_parsed": "2026-07-09T09:00:00 (from unix timestamp 1719964800)",
  "chars_removed": 234,
  "dedup_result": "new_document",
  "quality_score": 0.87,
  "warnings": ["content shorter than 100 chars after cleaning"]
}
```

#### 来源

- 思通舆情有完整日志收集体系（kafka → Elasticsearch 日志分析）
- 本方案采用轻量 JSONB 字段方案，不引入额外中间件

## 4. 分词与预处理衔接

清洗完成后，进入 NLP 预处理阶段（属于 NLP 模块，此处仅列接口）：

| 步骤 | 方法 | 来源 |
|------|------|------|
| 中文分词 | jieba + 自定义词典 + 停用词表 + 近义词表 | stay-leave 三重词典策略 |
| 停用词表 | 哈工大/百度/川大停用词库组合去重 | 旧项目 `stopword.txt` 已存在 |
| 自定义词典 | 网络新词 + 热搜关键词自动提取 + 人工维护 | stay-leave 项目 |
| 关键词提取 | TF-IDF + TextRank 双轨 | 后端方案 6.4 节 |
| 文本向量化 | bge-small-zh，title+content 前512字，输出512维向量 | 后端方案 6.4 节 |

## 5. 数据库表设计

### 5.1 clean_documents

| 字段 | 类型 | 说明 |
|------|------|------|
| id | SERIAL PK | — |
| raw_document_id | FK → raw_documents.id | 关联原始数据 |
| platform | VARCHAR(32) | 平台名称 |
| source_url | TEXT | 规范化后的 URL |
| url_hash | VARCHAR(64) | URL MD5（精确去重索引） |
| title | TEXT | 清洗后标题 |
| content | TEXT | 正文（纯文本，无HTML） |
| author | VARCHAR(255) | 作者 |
| publish_time | TIMESTAMP | ISO 8601 标准化时间 |
| like_count | INTEGER | 点赞数 |
| comment_count | INTEGER | 评论数 |
| repost_count | INTEGER | 转发数 |
| quality_score | FLOAT | 质量评分（0~1） |
| is_advertisement | BOOLEAN | 是否疑似广告/软文 |
| extraction_method | VARCHAR(32) | 正文提取方法（trafilatura/readability/bs4/fallback） |
| cleaning_metadata | JSONB | 清洗日志 |
| created_at | TIMESTAMP | — |

### 5.2 与下游表的衔接

- `clean_documents.id` → `document_features.document_id`（分词/关键词/向量）
- `clean_documents.id` → `event_documents.document_id`（事件关联）
- `clean_documents.id` → `sentiment_results.document_id`（情感分析）

### 5.3 其他表

沿用爬虫方案的：
- `raw_documents`（爬虫输出）
- `crawl_logs`（爬虫日志）
- `system_config`（系统配置）

## 6. 错误处理与降级

| 异常场景 | 处理方案 | 来源 |
|---------|---------|------|
| 正文抽取全部失败 | 用 title + 元描述兜底，标记失败但保留记录 | 后端方案异常降级表 |
| 时间解析失败 | 降级为抓取时间 `crawled_at` | — |
| bge 模型不可用 | 只做前两层去重，语义去重暂时跳过 | — |
| 某平台字段大面积缺失 | 记录告警日志，不阻断其他平台的清洗 | 后端方案 6.3 节 |
