### 1.2 各项目技术栈详细拆解

#### （1）BettaFish（微舆）—— 最全面的架构参考

| 维度 | 具体选型 |
|------|----------|
| **后端框架** | Flask（主应用）+ FastAPI + Streamlit（单Agent应用）|

| **数据库** | PostgreSQL / MySQL，ORM用 SQLAlchemy 2.0，缓存用 Redis |

| **LLM接入** | OpenAI 兼容格式，同时接入 Kimi / Gemini / DeepSeek / Qwen 多个模型 |
| **爬虫** | Playwright（浏览器自动化），覆盖微博/小红书/抖音/快手/B站/贴吧/知乎等 |
| **分词** | jieba |
| **情感分析** | 五套方案并存：BERT微调 / GPT-2微调 / Qwen微调 / 传统ML / LLM API直接判断 |
| **主题分类** | BERT微调 + LLM API |
| **可视化** | Plotly + Matplotlib + WordCloud |
| **报告生成** | 多轮LLM生成 → IR中间表示 → HTML渲染 → WeasyPrint导出PDF |
| **架构模式** | 多Agent协作：QueryEngine + MediaEngine + InsightEngine + ReportEngine + ForumEngine |
| **部署** | Docker + docker-compose |

**对我们最有价值的参考点**：
- LLM接入的抽象方式（统一 OpenAI 格式，可切换不同厂商）
- `config.py` 的 Pydantic Settings 管理模式
- 情感分析的多模型策略
- 数据库表结构设计（`mindspider_tables.sql`）

**对我们不太适用的点**：
- 多Agent + 论坛协作机制过于复杂，7天不可能复现
- 依赖项太多（50+），调试成本高

## 二、技术选型推荐

基于以上8个项目的实际架构和我们的题目要求（7天开发周期，5人团队），给出以下推荐方案：

```

│                    前端 (Frontend)                │

│   （或简化方案：Flask模板 + Bootstrap + ECharts）  │
└─────────────────────┬───────────────────────────┘
                      │ RESTful API (Axios ↔ Flask)
┌─────────────────────┴───────────────────────────┐
│                  后端 (Backend)                   │

│  │ 用户认证 │ 爬虫调度  │ LLM问答  │ 报告生成   │  │
│  └─────────┴──────────┴──────────┴────────────┘  │
│  ┌─────────┬──────────┬──────────┬────────────┐  │
```

### 2.2 具体技术选型及参考来源

| 层级 | 推荐方案 | 参考项目 | 理由 |
|------|---------|---------|------|
| **后端框架** | **Flask** | BettaFish, Astianjy | 简单轻量，5人团队容易上手；BettaFish用Flask支撑了41.6k star的项目 |
| **备选后端** | FastAPI | BettaFish（部分服务）| 自动生成API文档（Swagger），异步性能好 |
| **ORM** | **SQLAlchemy** | BettaFish | 最成熟的Python ORM，支持MySQL和PostgreSQL |
| **数据库** | **MySQL** | MindSpider, Astianjy, BettaFish | 五个参考项目全部用MySQL，团队熟悉度最高 |
| **前端框架** | **Vue 3 + Element Plus** | — | 题目要求较多交互（看板/详情/问答/个人中心），SPA体验好 |
| **备选前端** | Flask模板 + Bootstrap | BettaFish, Astianjy | 如果团队前端基础弱，可简化，BettaFish就只用了一个index.html |
| **可视化** | **ECharts** | — | 题目明确需要趋势图/饼图/词云，ECharts覆盖全部需求 |
| **爬虫框架** | **Playwright**（主）+ requests（辅） | MediaCrawler, MindSpider, BettaFish | 三大项目都用Playwright，能处理JS渲染页面，支持多平台 |
| **分词** | **jieba** | stay-leave, BettaFish, yutao-arch, Astianjy | 全部中文NLP项目都用jieba，事实标准 |
| **情感分析** | **SnowNLP**（基线）+ **LLM API**（主力） | stay-leave（SnowNLP），BettaFish（LLM） | SnowNLP开箱即用做快速验证；LLM API准确率高，且项目中智能问答已经要接LLM |
| **话题分类/热点发现** | **LLM API** + **LDA**（备选） | MindSpider（LLM），stay-leave（LDA） | LLM做分类最灵活；LDA做无监督聚类不需要标注数据 |
| **趋势预测** | **规则引擎**（基于报道量变化率）→ 可选升级为 **Prophet/ARIMA** | stay-leave 热度计算 | 先简单后复杂，7天内规则引擎最可控 |
| **LLM接入** | **DeepSeek API**（OpenAI格式） | BettaFish, MindSpider | 性价比高，中文能力强，API兼容OpenAI格式便于切换 |
| **LLM抽象层** | 统一 OpenAI 格式调用 | **BettaFish** | 换模型只需改URL和Key，不换代码 |
| **部署** | 本地运行即可 | — | 答辩时一台电脑演示 |

### 2.5 关键技术决策建议

| 决策点 | 建议 | 依据 |
|--------|------|------|
| 前端方案 | **优先建议用 Flask 模板 + ECharts CDN** | 简化前后端联调，减少一个技术栈的学习成本。BettaFish 就只用了一个 `index.html` + `app.py` 支撑全部功能。如果团队有人熟悉 Vue，再升级 |
| 数据库 | **MySQL + SQLAlchemy** | 全部参考项目都用MySQL；SQLAlchemy自动防SQL注入 |
| 爬虫 | **先 requests + BS4**（简单站），需要时再加 Playwright | weiboSpider 和 stay-leave 证明了 requests 对微博够用；Playwright 作为 JS 渲染页面的备选 |
| 情感分析 | **SnowNLP 做第一版 + LLM API 做最终版** | SnowNLP 零成本快速出结果；LLM API 准确率高，且已经在智能问答里调用了 |
| LLM | **DeepSeek API** | 中文能力强，价格便宜，OpenAI 格式兼容，将来切模型零成本 |
| 热点公式 | 参考 stay-leave 的三版热度公式思路 | 先用简单加权公式，后续调优 |
| 报告生成 | **LLM API 生成 Markdown → 前端渲染** | 参考 BettaFish 的 ReportEngine 简化版 |

## 三、从参考项目中可直接复用的实践

### 3.1 BettaFish 的 LLM 调用抽象（最值得学习）

```python
# 参考 BettaFish InsightEngine/llms/base.py
# 统一使用 OpenAI 兼容格式，换模型只改 base_url 和 api_key
from openai import OpenAI

client = OpenAI(
    api_key="your-api-key",
    base_url="https://api.deepseek.com"  # 或其他兼容厂商
)

response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "user", "content": "分析这条微博的情感倾向..."}]
)
```

### 3.2 stay-leave 的热度计算公式思路

该项目提供了三个版本的热度计算：
- `热度_1.py`：基础版
- `热度_2.py`：改进版
- `热度_3.py`：最终版

核心思路：**报道数 × 权重 + 评论/转发/点赞数 × 权重 + 时间衰减因子**

### 3.3 MindSpider 的话题提取流程

```
多个平台热榜 → 采集标题/摘要 → LLM提取关键词 → 关键词作为爬虫搜索词 → 深度爬取
```

### 3.4 stay-leave 的 LDA 主题建模流程

```
原始数据 → jieba分词（加载自建词表+停用词表+近义词表）→ 向量化 → LDA建模 → 主题输出
```

## 四、不太推荐的做法

| 不推荐 | 原因 |
|--------|------|
| 从零复现 BettaFish 的多Agent架构 | 7天不可能完成，Agent协作机制极其复杂 |
| 用 PyQt5 做桌面应用（yutao-arch方案）| 题目明确要求Web系统（登录/看板/详情页等网页交互）|
| 用 TensorFlow/PyTorch 训练专属模型 | 训练周期长、需要GPU，7天内用 LLM API 更务实 |
| 部署全套 Docker + docker-compose | 答辩时本地演示即可，不需要Docker化 |
| 做多平台爬虫全覆盖 | 先做好1-2个平台（微博+知乎），不要贪多 |
