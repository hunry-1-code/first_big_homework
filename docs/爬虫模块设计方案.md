# 爬虫模块设计方案

## 1. 数据源总览

| 平台 | 接口来源 | 成本 | 核心用途 |
|------|---------|------|---------|
| 微博热搜 | 公开接口 `weibo.com/ajax/side/hotSearch` | 免费 | 实时热点发现 |
| 微博搜索 | TikHub API | 付费（$0.001/次起） | 关键词搜索 + 评论 |
| 知乎 | 官方开放平台 `developer.zhihu.com` | 免费（5000次/天） | 深度长文分析 + 智能问答 |
| B站 | bilibili-api Python库 | 免费 | 视频搜索 + 弹幕情绪 |

## 2. 爬取模式：热搜定时 + 搜索按需

### 热搜榜单 → 定时自动爬（轻量，一直在跑）

- 频率：每天 3-4 次（如 9:00、12:00、18:00、21:00）
- 数据量极小，每次几十条词条
- 全部免费接口，零成本
- 目的：让系统首页始终展示当前热点事件

### 关键词搜索 → 用户触发时才爬（按需，爬完再存）

- 用户在搜索框输入关键词 → 系统实时调六个平台接口
- 搜索结果统一格式存入数据库
- 两小时内同关键词再次搜索直接返回缓存
- 目的：每一条数据都是用户真正关心的，不浪费配额

### 单事件采集规模

[已定] 第一版每个事件或关键词任务的初始目标为 **100 条原始报道/帖子**，指所有选定平台合计 100 条，不是每个平台各 100 条。

建议初始分配：

```text
新闻网页：30～40 条
社交平台：40～50 条
热搜、RSS 或其他来源：10～20 条
```

平台数量和接口可用性不同时允许动态调整，但应避免单个平台长期占据绝大多数样本。默认单平台优先不超过 50 条，已配置平台较少时可以放宽。

第一轮采集后根据清洗、相关性过滤和去重结果判断是否追加：

```text
有效独立内容 >= 60 条
→ 数据量满足第一版完整分析要求

有效独立内容 40～59 条
→ 可以分析，但标记 sample_insufficient，允许追加 50 条

有效独立内容 < 40 条
→ 追加采集 50 条
```

后续按 `100 → 150 → 200` 递增，第一版单事件原始采集上限为 200 条。达到上限后仍不足时，不继续无限采集，记录平台覆盖、重复率和有效率供人工判断。

评论、弹幕不计入上述 100 条报道/帖子。后续实现评论采集时单独设置数量限制。

提前停止条件：

- 连续两页新增有效内容比例低于 10%。
- 无关内容比例超过 50%。
- 已达到事件或平台数量上限。
- 平台返回验证码、登录验证或安全限制。
- 已覆盖任务设定的时间范围。

正常情况下，100 条原始内容的采集预计为 5～25 分钟；动态页面或平台限制较多时可能更长。答辩演示仍优先使用提前采集的数据。

### 时序流程

```
系统启动 → 拉一次各平台热搜 → 首页有数据
每天 4 个时间点 → 自动拉热搜榜单 → 热度排行持续更新
用户搜索"预制菜" → 库里没有/已过期 → 实时搜六平台 → 展示 + 入库
另一用户搜"预制菜" → 库里有且未过期 → 直接返回，不动外部接口
```

## 3. 统一数据格式

### 设计原则

- 所有平台数据入库前统一转为 `RawDocument` 格式
- 公共字段供下游模块使用，`raw_data` 保留原始 JSON 用于排查
- 下游（清洗、去重、情感分析、事件聚合）只认一套字段，不关心数据来源

### RawDocument 结构

```
platform       — 平台名称（weibo/zhihu/bilibili/xiaohongshu/baidu）
source_url     — 来源链接
title          — 标题
content        — 正文
author         — 作者
publish_time   — 发布时间
like_count     — 点赞数
comment_count  — 评论数
repost_count   — 转发数
raw_data       — 平台原始 JSON（JSONB 存库，排查问题时用）
```

### 各平台字段映射

| 公共字段 | 微博热搜 | 知乎 | B站 | 小红书 |
|---------|---------|------|-----|--------|
| title | word | title | title（去HTML标签） | display_title |
| content | 空→TikHub补 | summary | description | desc |
| author | 空 | author_name | author | user.nickname |
| publish_time | 无 | edit_time | pubdate（时间戳转换） | 需详情接口 |
| like_count | 无 | vote_up_count | video_review | liked_count |
| comment_count | 无 | comment_count | comment | 需评论接口 |
| source_url | 搜索链接拼接 | url | BV号拼接URL | 分享链接拼接 |

## 4. 去重策略（三层）

### 第一层：URL 精确去重（入库前）

- 对 `source_url` 算 MD5，存入 `url_hash` 字段
- 新数据入库前查 `url_hash` 是否存在，存在则跳过（只更新互动数据）
- 主要拦截热搜接口重复拉取同一词条

### 第二层：标题相似度去重（入库后）

- 用 SimHash 算法对标题计算指纹
- 查汉明距离 ≤ 3 的已有记录，匹配到则追加"也出现在某平台"标记
- 解决同一新闻被不同媒体转载、URL 不同但标题相近的问题

### 第三层：语义向量去重（清洗阶段，异步）

- 用 bge-small-zh 模型对标题+正文前200字生成向量
- 存入 pgvector，查余弦相似度 > 0.9 的已有记录
- 解决不同平台用不同表述讨论同一事件的问题

## 5. 调度策略

### Celery Beat 定时任务

- 热搜榜单：每天 4 次（9:00、12:00、18:00、21:00）
- 过期数据清理：每天凌晨 1 次（删除 7 天前的 raw_documents 和 hot_snapshots）

### Celery Worker 异步任务

- 关键词搜索：用户触发，实时入队执行
- 语义去重：每小时对新入库数据批量处理

---

## 6. 开关控制

### 实现方式

- `system_config` 表中存入 `crawl_enabled` 键值对（true/false）
- 每个定时任务执行前检查该开关，关闭则直接 return
- 手动搜索不受开关影响（用户主动操作）

### 前端按钮

- 首页"采集状态"旁放置开关
- 调用 `PUT /api/config/crawl_enabled` 切换
- 演示时关闭开关，系统纯走本地已有数据，零外部依赖

---

## 7. 增量爬取

- 搜索接口按时间倒序排列，第一条在库中已存在则停止翻页
- 支持时间筛选的接口（知乎、百度）直接限定时间范围拉取新内容

---

## 8. 错误处理与重试

- 429（频率超限）：等待后重试
- 5xx（服务器错误）：最多重试 3 次，指数退避（1s→2s→4s）
- 401/403（认证失败）：立即停止并记录告警日志
- 所有失败写入 `crawl_logs` 表，前端可查看

---

## 9. 频率控制

| 平台 | 限额 | 实际日调用 | 是否安全 |
|------|------|-----------|---------|
| 知乎 | 5000次/天 | ~20次（4次热榜+少量搜索） | 安全 |
| 百度搜索 | 1500次/月 | <100次/月 | 安全 |
| 微博热搜 | 无硬性限制 | 4次/天 | 安全 |
| B站 | 无硬性限制 | 少量按需搜索 | 安全 |
| TikHub | 按次扣费 | 按需搜索，有缓存控制 | 可控 |

## 10. 数据库表结构

| 表名 | 用途 | 操作方式 |
|------|------|---------|
| raw_documents | 统一格式原始数据 | INSERT + SELECT |
| hot_snapshots | 热搜快照（只追加不覆盖） | INSERT only |
| search_cache | 搜索结果缓存 | UPSERT |
| crawl_logs | 爬虫运行日志 | INSERT |
| system_config | 开关等配置 | UPDATE |

### hot_snapshots 表设计

```
id | keyword | rank | hot_score | platform | snapshot_time
───┼─────────┼──────┼───────────┼──────────┼────────────────
 1 | 预制菜   |  3   |   950000  |  weibo   | 2025-07-09 09:00
 2 | 预制菜   |  1   |  1800000  |  weibo   | 2025-07-09 12:00
 3 | 预制菜   |  8   |   420000  |  weibo   | 2025-07-09 18:00
```

- 每次爬取追加新记录，不更新不删除
- 首页读最新快照：`WHERE snapshot_time = (SELECT MAX(snapshot_time))`
- 趋势图：`WHERE keyword = 'xxx' ORDER BY snapshot_time`

### search_cache 表设计

| 字段 | 说明 |
|------|------|
| keyword | 搜索词 |
| platform | 来源平台 |
| results | 搜索结果（JSONB） |
| created_at | 写入时间 |
| ttl_minutes | 有效期（默认 120 分钟） |

---

## 11. 关键词管理

- 用户在个人中心配置关注关键词 → 存入 `user_preferences` 表
- 调度任务启动时读取所有用户关键词，去重后作为搜索列表
- 不同用户看到的事件与自己关注的领域相关

---

## 12. 演示策略

- **演示前**：提前几天打开爬虫，攒够真实的历史数据
- **演示当天**：关闭爬虫开关，所有操作纯走本地数据库
- 前端正常展示热点榜单、事件详情、趋势曲线、跨平台聚合
- 无需 mock 降级，数据真实，系统稳定，流程完整
