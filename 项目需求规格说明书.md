#### 2.2.1 后台任务层补充说明

```
[已定] 第一版采用轻量后台任务方案：APScheduler + MySQL 任务表。

耗时流程不在普通 HTTP 请求中同步执行，包括：
  - 爬虫采集
  - 样例数据导入后的清洗与分析
  - 正文提取、去重、分词、TF-IDF
  - 事件聚合、热度计算、LLM 情感分析
  - 报告生成

Flask API 只负责创建任务、返回 task_id、查询任务状态和读取已生成结果。
智能问答属于用户主动交互，可同步调用 LLM 或通过 SSE 流式返回。

第一版暂不引入 Celery / Redis / RabbitMQ，后续如需分布式任务再扩展。
```

## 四、后端功能详细规格

### 4.1 网络爬虫

> [来自PDF] 实现对主流新闻网站、社交平台中的新闻事件及舆论采集

#### 4.1.1 爬取目标

[已定] 第一版先实现可扩展采集框架，不强绑定具体平台。具体平台根据接口可用性、开发时间和演示数据准备情况确定。

| 数据源类型 | 代表平台示例 | 采集内容 | 第一版状态 |
|------|------|----------|------|
| 热搜/热榜类 | 微博热搜、知乎热榜、百度热搜 | 话题标题、排名、热度值 | 预留适配器 |
| 新闻搜索/网页类 | 百度新闻、人民网、澎湃新闻 | 标题、正文、来源、发布时间、链接 | 优先支持 |
| 社交平台内容类 | 微博、B站、贴吧、小红书 | 帖子标题/正文、作者、评论/转发/点赞等互动数据 | 预留适配器 |
| 样例数据导入 | JSON（必做）、CSV（可选） | 预采集报道数据和平台原始字段 | 必须支持 |

> **设计原则**：第一版重点不是覆盖所有平台，而是完成统一采集框架。不同数据源通过平台适配器输出统一 RawDocument 格式，下游清洗、分析和展示模块不依赖具体平台。
>
> **演示策略**：答辩演示以预采集/样例数据为主，在线采集为辅。演示前准备 3-5 个完整舆情事件样例数据；现场优先读取已入库数据和已生成分析结果，必要时展示“重新分析样例数据”证明系统链路真实可运行。

#### 4.1.2 爬取策略

```
[已定] 爬取流程（参考 MindSpider 两步走）：
  步骤1：从热搜/热榜类数据源获取热点话题标题
  步骤2：以热点话题或用户关键词为输入，在新闻/社交平台适配器中采集相关报道
  步骤3：统一转换为 RawDocument 格式，写入 article.raw_content / article.raw_json

[已定] 触发方式：
  - 定时采集：APScheduler 触发热搜/热榜采集任务
  - 用户关键词采集：普通登录用户可对新关键词发起低频采集任务
  - 管理员手动采集：管理员可触发平台采集、样例数据重新分析等系统级任务
  - 样例数据导入：管理员上传 JSON 样例数据，进入同一清洗/分析流程

[已定] 技术方案：
  - 优先使用 requests + BeautifulSoup4（简单站点）
  - 新闻正文抽取优先使用 BeautifulSoup / readability / newspaper3k 等本地解析工具
  - 遇到 JS 动态渲染页面 → Playwright 作为扩展方案

[已定] 合规与反爬边界：
  - 不绕过验证码
  - 不破解接口签名
  - 不使用代理池冲击平台
  - 不批量注册账号
  - 只采集公开数据或用户授权后的低频公开样本
  - 外部平台不可用时，使用预采集样例数据保证主流程可演示

[已定] 请求控制：
  - 所有外部请求设置超时时间、响应体大小限制、重试次数和请求间隔
  - 同一关键词短时间内不重复采集
  - 用户关键词采集受频率限制，管理员任务记录在 task 表中
```

#### 4.1.3 爬虫输出

每条报道入库时包含：

| 字段 | 说明 | 示例 |
|------|------|------|
| `platform` | 来源平台 | "baidu_news" |

### 4.2 数据清洗及预处理

> [来自PDF] 去重、去噪、格式标准化处理

#### 4.2.1 去重

```
[已定] 三层去重策略：

第一层（URL 精确去重）：
  规则：同一 URL 绝对不重复爬取，数据库 url 字段设 UNIQUE KEY
  来源：MediaCrawler store/ 层的做法

第二层（标题相似度去重）：
  规则：新报道标题与已有报道标题计算 Jaccard 相似度 > 0.85 → 判定为重复
  方法：jieba 分词后取词集合，J(A,B) = |A∩B| / |A∪B|
  阈值：默认 0.85，后续根据样例数据调参

第三层（内容 SimHash 去重，可选）：
  规则：标题不同但正文几乎一样（不同平台转载）
  方法：计算 SimHash 指纹，海明距离 < 3 → 判定为重复
```

#### 4.2.2 去噪

```
[已定] 噪声 = 以下全部：
  1. HTML/XML 标签        → BeautifulSoup .get_text() 去除
  2. 特殊字符 + emoji     → 正则保留中英文、数字、常用标点
  3. 纯数字/纯符号短文本  → 默认长度 < 20 个中文字符，丢弃
  4. 停用词               → 参考 stay-leave 的停用词表
  5. 广告/推广内容        → 关键词规则 + 链接密度判断，标记后降权
```

#### 4.2.3 格式标准化

```
[已定] 统一规则：
  编码：全部转为 UTF-8
```

```
[已定] 热度指数公式（参考 stay-leave 三版热度公式）：

[已定] 热点阈值：
```

### 4.5 舆情事件聚合

> [来自PDF] 对同一事件的多条报道聚合为单一事件，并支持对历史事件和相似事件的检索

#### 4.5.1 事件聚合算法

```
[已定] 算法：Single-Pass 增量聚类 + TF-IDF 余弦相似度（参考 stay-leave）

  对于每一条新报道：
    1. 计算其 TF-IDF 向量
    2. 与所有已有事件的"事件中心向量"计算余弦相似度

    4. 如果最高相似度 >= 0.5：

    5. 否则：
       创建新事件，该报道成为第一个成员

  - TF-IDF + 余弦相似度是纯数学运算，毫秒级、零成本
```

## 六、高级功能

### 6.1 虚假/可疑信息风险评估

> [来自PDF] 对采集到的信息进行真实性判断，给出置信度

#### 6.1.1 检测方法

```
[已定] 第一版实现轻量级“可疑信息风险评估”，不直接宣称真假判定。
系统输出“可疑风险分数”和“风险原因”，不输出绝对真伪结论。

Step 1 — 基础规则评分：
  - 来源可信度较低
  - 缺少多平台交叉验证
  - 标题存在夸张/煽动性表达
  - 负面情绪比例较高
  - 短时间传播速度异常
  - 尚无官方媒体或权威来源回应

Step 2 — 多源交叉验证：
  - 同一事件是否有多个独立信息源报道
  - 多个平台内容是否基本一致
  - 是否存在官方媒体或权威机构回应

Step 3 — LLM 辅助分析：
  - 仅对高风险事件或代表性文章调用 LLM
  - LLM 输出风险原因说明，作为辅助参考
  - 最终展示为“疑似风险/可疑信息评估”，不作为事实裁定
```

#### 6.1.2 风险分数计算

```
[已定] 综合评分：
  初始 50 分
  + 多源交叉验证不足：+15
  + 来源可信度较低：+15
  + 标题/正文存在夸张煽动表达：+10
  + 负面情绪比例较高：+10
  + 传播速度异常：+10
  - 官方媒体或权威来源已回应：-20
  最终分数 clamp 到 0-100

展示方式（前端）：
  0-39   → 低风险
  40-69  → 中风险
  70-100 → 高风险
```

### 9.2 LLM 调用场景汇总

| 场景 | 调用位置 | Prompt 来源 | 频率 | 第一版定位 |
|------|----------|------------|------|------------|
| 情感分析 | sentiment_analyzer.py | prompts.py | 对待分析文章逐篇调用，带缓存和限流 | 必做，SnowNLP 降级 |
| 主题语义命名 | topic_classifier.py | prompts.py | 每次主题聚类后（低频）| 必做 |
| 事件摘要生成 | report/event 服务 | prompts.py | 每个事件生成/更新报告时（低频）| 必做 |
| 智能问答 | qa.py (api) | prompts.py | 用户每次提问（按需）| 必做 |
| 报告自然语言润色 | report 服务 | prompts.py | 每个报告一次（低频）| 必做 |
| 问答引用依据生成 | qa.py (api) | prompts.py | 与问答一起 | 可选增强 |
| 虚假/可疑信息辅助判断 | fake_detector.py | prompts.py | 仅高风险事件或代表文章 | 高级功能 |
| 事件聚合边界样本校正 | hot_detector.py | prompts.py | 仅边界样本 | 扩展预留 |
| 事件级关键词润色 | report 服务 | prompts.py | 每个事件一次 | 可选增强 |

[已定] LLM 不承担全量正文提取、全量关键词提取、全量事件聚合等高频批处理任务。
这些任务由 BeautifulSoup/readability/newspaper3k、jieba、TF-IDF、TextRank、余弦相似度等本地算法完成。

### 9.3 错误处理

```
[已定] LLM 调用失败处理：
  1. 重试 3 次（间隔 2s/4s/8s）
  2. 仍失败 → 根据场景降级：
     - 情感分析 → 降级到 SnowNLP，并记录 sentiment_method = "snownlp"
     - 智能问答 → 返回“AI 暂时无法回答，请查看事件报告”
     - 摘要生成/报告润色 → 使用抽取式摘要（标题 + 代表文章前200字）
     - 可疑信息检测 → 使用规则评分，标记 suspicious_method = "rule"
  3. 记录错误日志，不阻塞事件看板和详情页展示
```

## 十、数据处理全流程

### 10.1 完整数据流

```
[数据采集/导入]
  在线采集 / JSON 样例数据导入 ──→ article.raw_content / article.raw_json
                                 └─→ clean_status = pending

[后台任务调度]
  APScheduler 定时任务 / 用户关键词采集 / 管理员手动任务
    └─→ task 表记录 pending/running/success/failed/progress/message

[数据预处理]
  article.raw_content ──→ extractor.py（正文提取）──→ 候选正文
                      ──→ cleaner.py（去重/去噪/标准化）──→ article.clean_content
                      └─→ clean_status = success/failed，失败写入 clean_error

[内容分析]
  article.clean_content ──→ segmenter.py（jieba 分词 + 停用词过滤）
                         ──→ TF-IDF 批处理统一建词表和向量
                         ──→ article.tfidf_vector
                         ──→ LDA 主题建模 + LLM 主题语义命名

[热点发现/事件聚合]
  新 article 向量 ──→ Single-Pass 增量聚类（默认阈值 0.5）
                   ──→ 归入已有 event 或创建新 event
                   ──→ 热度公式计算（默认半衰期 24 小时）
                   ──→ 生命周期判定

[情感分析]
  article.clean_content ──→ LLM 情感分析（标题 + 正文前500字）
                         └─→ 失败降级 SnowNLP
  聚合到事件级 ──→ event.sentiment_positive/negative/neutral

[可疑信息风险评估]
  来源可信度 / 多源一致性 / 传播速度 / 负面情绪 / LLM 辅助
    └─→ suspicious_score / suspicious_reason

[报告生成]
  事件数据 + 趋势/情感/平台/关键词/风险数据 ──→ report 表
  LLM 生成摘要和自然语言分析，图表数据保持结构化 JSON
  前端请求 API ──→ ECharts + 文本报告展示

[智能问答]
  用户提问 ──→ 构建上下文（事件概述 + 数据 + 报道摘要 + 引用来源）
           ──→ LLM API ──→ 同步或 SSE 流式返回
```

### 10.2 关键数据流触发时机

```
后台任务（APScheduler / 用户关键词采集 / 管理员手动触发）：
  ┌─ 1. 爬虫拉取最新数据或导入样例 JSON
  ├─ 2. 数据清洗 + 正文提取
  ├─ 3. jieba 分词 + TF-IDF 批处理
  ├─ 4. 事件聚合（新报道分配）
  ├─ 5. 热度重算 + 生命周期更新
  ├─ 6. LLM 情感分析（失败降级 SnowNLP）
  ├─ 7. 可疑信息风险评估
  └─ 8. 报告生成/更新

用户访问时（按需）：
  ├─ 事件看板 → 读 event 表（已有数据）
  ├─ 事件详情 → 读 report 表（已有数据）
  ├─ 任务状态 → 读 task 表
  └─ 智能问答 → 实时调 LLM API 或 SSE 流式返回
```

## 十一、系统设计难点与亮点

> [来自PDF] 总结报告需要写"系统设计难点与亮点"。提前规划。

### 11.1 候选亮点

| 亮点 | 说明 |
|------|------|
| 1. TF-IDF 一条线串三个需求 | 同一特征表示同时服务于 LDA 主题建模（热点发现）、余弦相似度（事件聚合）、最近邻搜索（相似检索）——技术选型的巧妙之处 |
| 2. LLM 情感分析 + SnowNLP 降级 | LLM 提升复杂语境判断质量，SnowNLP 作为失败降级，兼顾效果与稳定性 |
| 3. 热度公式的三版迭代 | 从纯报道量 → 加权传播数据 → 指数时间衰减，体现算法演进过程 |
| 4. LDA + LLM 混合主题分类 | LDA 满足"机器学习方法"要求，LLM 补足语义理解和可解释性 |
| 5. 智能问答流式输出 | SSE 实现 ChatGPT 式的逐字显示，用户体验好 |
| 6. 可疑信息风险评估 | 多源交叉验证 + 规则评分 + LLM 辅助说明，只输出风险评估而不做绝对真假裁定 |
| 7. LLM 接入的可替换设计 | OpenAI 兼容格式 + .env 切换，换模型只改一行配置 |
| 8. ECharts 交互式可视化 | 趋势图标注关键节点、力导向图展示传播路径，比静态图片高级 |

### 11.2 候选难点

| 难点 | 说明 |
|------|------|
| 1. Single-Pass 增量聚类的准确性 | 阈值调优、噪声文章的处理、"事件漂移"问题（事件随时间演化后中心向量偏了怎么办）|
| 2. 生命周期判定阈值的确定 | 不同事件类型的报道量级差异很大（小地方事件几百条 vs 全国性事件几万条），固定阈值可能失效 |
| 3. 爬虫稳定性 | 反爬策略、cookie失效、网页结构变化 |
| 4. LLM 输出格式的稳定性 | Prompt 让 LLM 输出 JSON 但偶尔会多输出解释文字，需要 JSON 解析 + 容错处理 |
